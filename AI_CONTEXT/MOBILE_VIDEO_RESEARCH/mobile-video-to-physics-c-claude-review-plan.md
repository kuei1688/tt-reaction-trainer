# 手機版影片→物理切鏡方向 C：最小視覺實驗計畫

> 狀態：**已執行（2026-07-12）；結果見 `prototypes/video-physics-timeline/direction-c/RESULT.md`（C3 勝出）**
> 日期：2026-07-11  
> 依據：`AI_CONTEXT/MOBILE_VIDEO_RESEARCH/mobile-video-to-physics-c-direction.md`、`AI_CONTEXT/ARCHIVE/checkpoints/CHECKPOINT_REAL_VIDEO_HANDOFF_EXPERIMENT.md`、`prototypes/video-physics-timeline/SPEC.md`

## 1. 現況與問題定義

目前唯一的真人影片探索樣本為 `real_backspin_001`。它的觸球時間為 4.300 秒，而觸球後原影片只剩約 0.283 秒；沿用合成 `prototype_short` 玩法 profile 時，影片 anchor 與第一個物理投影的差距約為 349–382 px（960×540）。因此，不能誠實地把它呈現成「同一顆球在同一座標系中自然接續」。

`timeline-config.json` 內的 `generation_status: "ready"` 只是探索實驗的操作性設定，並不表示 handoff 已校準、真人瀏覽器畫面已依 SPEC §8 驗收，或素材可進 Trainer。這一資料例外在本計畫中不修改，也不得被當成放行條件。

手機訓練的主要問題不是重建真人影片的完整軌跡，而是：玩家能否先從人物特寫讀到拍型、觸球與固定旋轉線索，再在清楚的切鏡後，理解一顆「與該線索相容的訓練來球」從接球桌面的固定入口進入可玩區。

## 2. 本輪要驗證的產品語意

以最小 prototype 觀察下列產品語意是否可理解：

> 在直式手機畫面中，接球者桌面從影片開始就持續可見。觸球瞬間，人物特寫影片層明確退場；同一張桌面成為主畫面，唯一一顆可玩物理球從固定「球路入口」出現。

本實驗判斷的是**使用者是否理解並接受這個切鏡語意**，不是判斷物理模型是否真實，也不是替 `real_backspin_001` 做交接校準。它不需要 taskpack、審理表或 checkpoint，前提是所有改動留在 `prototypes/`。

## 3. 方向 C 的不可變規則

1. 觀察階段中，接球桌面始終可辨識：至少能看見球網、遠端桌緣與接球端的空間方向。
2. 觀察階段中，桌面是空的待命場景；不得出現第二顆物理發球，也不得暗示影片球已落入桌面。
3. 觸球後，影片內的球隨影片層退場；畫面上只保留一顆物理球。
4. 物理球的固定球路入口是訓練規則／介面語意，不是影片球位置、真人軌跡量測或 handoff 校準結果。
5. 切換必須清楚表達「從觀察線索轉入訓練來球」，不得用長 crossfade、透明度拖延或位置對齊假裝兩個視覺空間共用同一座標系。
6. 影片固定繼承旋轉類型、旋轉方向與可見動作線索；長短、落點、速度與難度只能從該素材被內容／教練允許的變體中選取。
7. 觸拍／擊球聲不得被當作判斷或掩飾切鏡不連續的線索：真正要訓練的技能是讀取對手視覺動作，快速技術（如殺球）等聲音出現時已來不及反應，且許多技巧本身無明顯聲音。落桌聲例外——它是球本身可代表節奏的真實線索，若加入必須在 C1／C2 之間完全一致，不得成為第二個比較變因。

## 4. 最小實驗範圍

### 固定樣本與資料邊界

- 素材：僅 `real_backspin_001`；不新增或替換影片。
- 旋轉線索：僅該素材既有的下旋內容語意；不從像素計算旋轉或速度。
- 物理：沿用既有、已選定的玩法近似 profile；不改動 position、velocity、spin、重力、shared core 或任何正式 preset。
- 變體：最多使用內容／教練明確允許的少量既有變體；若尚無這份允許清單，實驗停在「單一固定測試來球」，不自行猜測長短／落點範圍。
- 平台：只定義直式手機視覺與互動假設；不接入正式 Trainer、`game4.html`、`match-trainer.html` 或題庫。

### 唯一比較變因：切鏡節奏

| 候選 | 觸球後的畫面行為 | 不變部分 |
|---|---|---|
| C1：立即讓位 | 影片層在觸球點立即退場；原本可見的桌面立刻成為主畫面。 | 同影片、同桌面、同球路入口、同一顆物理球、同一測試變體。 |
| C2：極短收束 | 影片層在觸球點後以極短且明確的回拉／收束退場；桌面不中斷地成為主畫面。 | 同 C1；不得延長到足以掩蓋來源不連續或額外偷走反應時間。 |

本輪可自行選定足以比較 C1／C2 的毫秒數、動畫曲線與 DOM／Canvas 實作方式，並在結果記錄中列出實際採用值；它們是實驗條件，不是正式 UI 規格。

### 其他實作參數（非比較變因，需記錄實際採用值）

- **直式版面配置與球路入口位置**：人物特寫與接球者桌面的相對佈局（例如上下分屏、或人物特寫為前景疊加、桌面於下方常駐可見）、以及物理球固定入口的進場方向與螢幕位置，實作時自行選定一組具體方案；這是讓「桌面持續可見」與「固定入口」這兩項產品語意可被實作與觀察的必要細節，不是新的比較變因，也不是正式 UI 規格。
- **觸球觸發機制**：沿用 `SPEC.md` 既有的 media adapter 判斷式（`currentTime >= physics_trigger_time_sec`）與既定 ±0.05 秒容許誤差，不另訂新的觸發機制或精度要求。
- **落桌聲（可選）**：若製作成本允許，可加入與物理彈跳事件同步的落桌聲作為節奏輔助觀察；若加入，須在 C1／C2 間完全一致，且非本輪必要條件。

## 5. 預期的使用者可見流程

1. 玩家進入短回合，首先看到人物特寫；下方或背景已能辨認接球端桌面。
2. 玩家觀看拍型、觸球與旋轉線索；此時桌面上沒有可玩球，沒有反應倒數，也不要求輸入。
3. 觸球點到達，影片層以 C1 或 C2 的明確方式退場；玩家仍看見同一張桌面。
4. 一顆物理球從固定球路入口開始可玩飛行；此刻才開始定義玩家的有效反應時間。
5. 玩家依未來另定的 Trainer 輸入方式作答／回擊；本實驗不設計、驗證或選擇該輸入方式。
6. 實驗觀察結束於第一顆球進入可玩區及其可理解性；不延伸至對手回球、rally、計分、動畫編排或課程流程。

## 6. 需要先由人類／內容審查確認的前提

以下前提若沒有被明確確認，不能自行用技術或物理推測補上：

| 前提 | 為何必要 | 缺少時的處理 |
|---|---|---|
| 該影片確實適合用來教下旋線索 | 決定影片是否可作為訓練內容，而非決定物理公式。 | 停止；不以檔名或 profile 推論內容價值。 |
| 該素材允許的長短、左右與速度變體 | 防止「固定旋轉」被錯誤放大為「任何球路都合理」。 | 僅使用一個固定測試來球。 |
| 玩家看完觸球才進入反應窗口是否符合訓練目標 | 影響切鏡是否搶走反應時間與回合語意。 | 只做無計時的理解性觀看，不宣稱反應訓練有效。 |
| 桌面從一開始可見是否足以建立接球端空間感 | 這是方向 C 的核心產品假設。 | 只能判為未驗證，不得升級為實作依據。 |

## 7. 評估問題與證據界線

### 可由人工視覺實驗回答

- 玩家是否知道影片在教他讀旋轉／拍型，而不是要求他追蹤同一顆球的完整飛行？
- 玩家是否能辨認桌面一直是同一個接球空間？
- 觸球後是否清楚只剩一顆可玩球？
- 固定球路入口是否被理解為訓練規則，而不是「影片球瞬移」？
- C1 或 C2 是否有一個較少造成困惑、且不明顯壓縮可用反應時間？
- 在內容已允許的變體內，玩家是否覺得生成來球與影片旋轉線索相容？

### 不可由本實驗宣稱的事情

- 真人影片與物理球在幾何、速度、旋轉或落點上連續一致。
- 影片已提供完整物理初始條件，或可證明生成球是實測軌跡。
- 玩家實際反應能力、訓練成效或旋轉辨識率已提升。
- `real_backspin_001` 已校準、已發布、可進正式 Trainer，或其 `ready` 值代表完成驗收。
- C1／C2 的其中之一已是正式 UI 決策。

## 8. 實驗記錄格式（概念層，不建立 schema）

每一次人工觀看至少記下：

- 觀看的候選節奏（C1 或 C2）與素材 ID。
- 固定測試來球或已核准變體的識別方式。
- 玩家是否理解「影片讀線索 → 切換 → 一顆訓練球」的流程。
- 是否誤認影片球與物理球為必須幾何連續的同一顆球。
- 是否誤以為桌面上同時有兩顆球，或在影片階段就可回擊。
- 切鏡是否使反應開始點不清楚，或主觀感受上吃掉太多時間。
- 對「線索與生成球相容」的主觀意見，以及任何不相容的具體描述。

這些紀錄是產品可讀性證據，不是物理量測、統計顯著性結論或發布資格。

## 9. 實驗範圍、完成條件與停止線

### 實驗範圍

在 `prototypes/video-physics-timeline/` 內以單一 `real_backspin_001` 樣本製作 C1 與 C2，並使用同一個固定測試來球；若內容／教練已明確列出變體限制，才可增加少量允許變體。

### 完成條件

人工能觀看相同條件下的 C1 與 C2，確認桌面不中斷、影片期間沒有第二顆可玩球、切換後只有一顆物理球，並記錄第 8 節的理解性觀察。

### 停止線

若需要長轉場掩飾不連續、素材無法限制允許變體，或實驗必須改動核心／正式頁／題庫／preset 才能成立，停止目前實驗並先討論紅線變更。沒有必要因為要改善視覺實驗而自動把它接入 Trainer、全題庫、完整回合或校準工作。

## 10. 檔案與主線保護承諾

本實驗可修改 `prototypes/` 內的程式、設定、複製素材與測試資料。下列紅線檔案或資料不在本實驗範圍；若要修改，必須先討論：

- `shared-physics-core.js`
- `game4.html`
- `match-trainer.html`
- `videos.json`
- `physics-presets.json`

也不得把既有 `real_backspin_001` 的 `generation_status` 當作已發布或已驗收的證據。若未來要改動正式題庫資料或發布狀態，先討論其資料影響與驗證。

## 11. 起草自檢

- 已將影片線索與生成訓練球分成兩個視覺／語意空間。
- 已將「固定球路入口」表述為介面規則，不是 anchor、真人位置或物理量測結果。
- 已把唯一比較變因限定為 C1／C2 切鏡節奏；其餘條件固定。
- 未主張切鏡可修正真人—物理的大 handoff 落差。
- 未新增 profile、物理參數、影片、題庫、發布狀態或正式頁整合。
- 未把觸拍／擊球聲當作判斷或掩飾切鏡不連續的手段；落桌聲僅作為可選、與比較變因無關的節奏輔助。
## 12. 實驗後的決定

完成人工觀看後，依觀察選擇保留 C1、保留 C2、調整並重測，或停止方向 C。只有想把結果接入正式 Trainer、正式題庫或物理核心時，才提出具體紅線變更、風險與驗證供討論。
