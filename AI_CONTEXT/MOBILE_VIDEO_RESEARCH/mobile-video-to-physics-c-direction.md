# DRAFT：手機版影片→物理切鏡方向 C

> 狀態：**Direction C 已驗證（C3 為首選），研究階段完成；隔離 prototype，未接入正式 Trainer**
> 日期：2026-07-11  
> 來源：使用者／Codex 討論

## 1. 要解決的問題

未來訓練主要在手機上進行。真人發球影片為了保留人物、球拍與觸球細節，可能採人物特寫；其鏡頭比例與接球方桌面視角不相容。因此，不應再假定影片中的球位置可與遊戲桌面世界座標連續重合。

既有單支真人影片探索也顯示，觸球 trigger 的影像 anchor 與沿用的物理 profile 投影有 large delta。這是目前 prototype 的已知觀察，不應以長 crossfade 或透明度變化偽裝成自然連接。

## 2. 已知前提與尚未驗證處

### 已知前提

- 每支已入選影片的旋轉屬性是固定內容；使用者說明這些影片已由桌球教練篩選，且既有小遊戲已證明其旋轉可被辨識。
- 影片的核心訓練價值是讓玩家讀取拍型、觸球與旋轉線索；它不是完整球路的量測資料。
- 現有 prototype 的 `real_backspin_001` 探索記錄仍屬未校準、未發布的產品資料；`generation_status: "ready"` 不能被解讀為自然 handoff 或正式 Trainer 驗收已完成。

### 尚未驗證

- 玩家在觸球瞬間切換鏡頭後，是否仍能把剛讀到的旋轉線索正確帶入接球判斷。
- 多長的切鏡／回拉節奏不會偷走可用反應時間。
- 同一影片下可接受的長短、落點與速度變體範圍，需由內容／教練審查定義，不能只由旋轉名稱推導。

## 3. 優先候選：方向 C（桌面持續可見、影片層明確切換）

方向 C 將影片與可玩物理明確分為兩個視覺空間，而非嘗試做同一鏡頭的幾何連續；但**接球者桌面必須從一開始就可見**。玩家應先建立「我站在這張桌子的接球端」的空間直覺，切換的只是發球者特寫影片層，而不是整個遊戲場景。

```text
人物特寫影片為主 + 接球者桌面已在下方／背景待命
  └─ 玩家讀取拍型、觸球、固定旋轉；桌面沒有可玩球
                 ↓ 觸球瞬間，影片層短促收束／退場
同一張接球者桌面成為主畫面
  └─ 唯一的物理球從固定「球路入口」開始可玩飛行
```

影片播放期間，桌面只能作為空的待命場景，不得演出第二段物理發球或第二顆球。觸球後，影片中的球隨影片層退場；畫面只保留一顆進入桌面的物理球。

「球路入口」是手機版介面與玩法的固定語意，不是影片球的位置，也不是對真人軌跡的量測主張。切換必須足夠明確，讓玩家理解這是從動作線索轉入訓練來球；不得以長 crossfade 假裝真人球和物理球處於同一座標系。

## 4. 生成球的產品語意

- **固定自影片繼承**：旋轉類型、旋轉方向，以及該素材已辨識的動作線索。
- **由訓練情境選擇**：短／長、左右落點、速度、彈跳與難度。
- **受到素材限制**：每支影片必須有內容允許的來球變體範圍；例如可列出允許的長短與落點，但不預設所有旋轉都可生成任何球路。
- **可重播**：未來若有隨機抽樣，情境須保留 seed 或 variant ID；本草案不定義 schema 或資料儲存位置。

這是 gameplay approximation：物理球必須與影片旋轉線索相容並提供可訓練的變體，但不宣稱影片已量測出該球完整的真實飛行軌跡。

## 5. 與雙舞台方案的差異

雙舞台不是「同時演兩次發球」：桌面應只待命，直到影片觸球後才出現物理球。方向 C 採用這項正確限制，但不把人物影片和桌面做成兩個同權、長時間競爭注意力的面板。

方向 C 的構圖原則是：影片在觀察階段為主，桌面持續提供接球空間脈絡；觸球後，影片層快速讓位給原本已存在的桌面。桌面可在觀察階段較安靜（例如較暗或降低細節），但球網、遠端桌緣與接球空間應維持可辨識。

## 6. 最小視覺實驗（prototype 範圍內自由進行）

只用一支已知素材驗證 C，不調整物理核心、不批次擴充題庫，也不接入正式 Trainer。

比較兩個「影片層讓位、桌面不中斷」的節奏即可：

1. 觸球後立即讓影片層退場，桌面立即成為主畫面。
2. 觸球後以極短、明確的視野回拉／收束讓影片層退場，再讓同一張桌面成為主畫面。

每個節奏只搭配少量、內容已允許的來球變體。人工檢視應回答：

- 玩家能否先辨識旋轉？
- 桌面從一開始可見，是否增強接球端的空間直覺？
- 影片層切換是否自然且不造成誤解？
- 玩家是否清楚影片期間沒有第二顆可玩球？
- 球路入口是否可被理解為訓練場的規則？
- 影片線索與生成球是否感到相容，而不是隨機或被欺騙？

## 7. 非目標與停止線

- 不以本實驗修改 `shared-physics-core.js`、正式遊戲頁、`videos.json` 或 `physics-presets.json`；若後續需要動到這些紅線檔案，先提出目的、風險與驗證再討論。
- 不把 C 宣稱為已完成的物理校準、真人影片連續鏡頭，或已通過 Trainer 驗收。
- 第一輪只比較 C1／C2、單支影片與單一固定測試來球；任何額外素材、變體、校準或完整回合都必須另行說明其目的，不能藉由本實驗自動宣稱為已驗證能力。
- 不把為了掩飾真人—物理落差而拉長的 crossfade、透明度變化或「成功」標籤當作解法。
