## 1. 背景與已得到的結論

`prototypes/video-physics-timeline/` 的原始目的，是把影片時鐘、物理 bridge、玩家 Hit、對手動畫與事件流程組裝在一起，觀察它距離成品體驗還有多遠。這個探索目的已完成：目前可以看見完整的單回合節奏，以及影片交給物理、玩家回擊、對手回球的可能性。

但本輪觀察也得到另一個明確結論：原型頁把下列四種責任放在同一個頁面與設定檔中，導致可操作性低、呈現感偏向工程 Demo，而非訓練產品。

- 訓練體驗：選情境、開始、等待來球、回擊與回饋。
- 影片—物理交接校準：trigger、anchor、初始物理狀態與交接誤差。
- 對手／動畫節奏設定：prep、delay、counter 與方向對應。
- 工程驗證：WebM 產製、frame 對齊、事件紀錄、物理 telemetry 與 JSON 匯入匯出。

目前原型的 Node 測試可驗證狀態機、隔離 physics bridge 與 WebM 產製契約；本輪本地執行結果為 18、8、11 項通過。這是「流程與工具契約」的證據，**不是**真人影片與物理球自然銜接、也不是正式 Trainer 體驗已完成的證據。

單支真人影片 `real_backspin_001` 的探索紀錄也已發現：其 handoff 當下，影片 anchor 與物理投影相差約 349–382 px（960×540），現有物理初始狀態直接沿用合成 serve，尚未對該影片校準。這是已知實驗落差，不應被視為已解決的產品能力。詳見 `AI_CONTEXT/CHECKPOINT_REAL_VIDEO_HANDOFF_EXPERIMENT.md`。

## 2. 本計畫的決策提案

不推翻或重寫既有物理核心、時間軸狀態機、WebM 產製器；先把原型當成已完成的可行性實驗。

下一階段改採「功能拆分、逐塊校正、逐塊驗收」的方式，而不是繼續把更多控制項、素材或效果堆進原型首頁。

目標架構是三個有單一產物的工作面：

```text
影片—物理交接校準器
        ↓ 產出已驗收的 handoff calibration
動作／動畫節奏編排器
        ↓ 產出已驗收的 response timeline
Trainer 情境組裝與預覽
        ↓ 只讀取已發布的情境
```

這是產品工作流的拆分提案，不是本輪將要修改的正式檔案架構或資料 schema。

## 3. 目標與非目標

### 目標

1. 讓每個工具只解決一種問題，且有一種明確輸出。
2. 讓校準者能用視覺引導完成一支影片的交接校準，不必手動編輯 JSON 或直接理解 UV／世界座標。
3. 把工程診斷從未來的 Trainer 體驗移出；保留為 Inspector 或開發工具。
4. 讓「可播放」、「已校準」、「已人工驗收」、「可進 Trainer」成為可區分的狀態。
5. 在不修改 `shared-physics-core.js`、正式遊戲頁與既有題庫結構的前提下，取得一個可審查的下一步任務包。

### 非目標／停止線

- 不在本計畫中修改 `shared-physics-core.js`、`game4.html`、`match-trainer.html`、`physics-presets.json` 或 `videos.json`。
- 不把隔離原型直接併入 MVP 主線。
- 不承諾由校準工具自動推導真實世界完整旋轉、球拍接觸力學或普遍正確的物理真值。
- 不建立所有影片的批次校準流程，不擴充成 CV 自動追球系統，也不開始製作完整教練課程。
- 不把 WebM 產製器改成一般使用者必經流程；它仍屬工程素材工具。
- 不承諾以拉長 crossfade 掩蓋交接落差。若影片與物理不對齊，應保留為未校準或明確的場景切換，不得宣稱自然連接。

## 4. 工作面與輸出契約

### A. 影片—物理交接校準器（第一優先）

**唯一問題**：指定影片在指定影格，能否以可量測、可審查的方式交給物理球。

**校準者可見流程**：

1. 從已授權的素材清單選一支影片。
2. 在引導式播放器中定位 trigger；可逐幀前後移動。
3. 在影片上標記 trigger 當下的球位置。
4. 選一個既有「玩法近似」的物理起始 profile；Phase 1 不從相鄰影格推導或調整初始速度。
5. 在同一畫面預覽影片球、物理球與交接差距。
6. 儲存草稿；只有通過既定驗收與人工確認後才可發布。

**校準者不必直接操作**：`video_anchor_uv`、世界座標、`generation_status`、完整 `timeline-config.json`、JSON 複製貼上、事件 ID。

**輸出（概念性，非本輪 schema）**：一份 immutable 的 `handoff calibration` 記錄，至少可追溯來源影片、trigger、單點影像標記、使用的玩法 profile、量測誤差、建立時間、驗收狀態與人工確認者。它不是「真實物理模型」的證明。

**明確排除**：從多個影格位置推導速度、軌跡擬合、數值微分、重力／旋轉分解與自動求解，都不屬於 Phase 1。若日後需要，必須另立子任務重新審查其演算法假設。

### B. 動作／動畫節奏編排器（第二優先）

**唯一問題**：在已知事件序列中，對手提示、預備、回球動畫與回饋何時出現，是否可讀且節奏自然。

**輸入**：已校準 handoff 的事件點，以及一個有限的對手反應 profile。

**輸出**：一份 response timeline，描述 `player hit → opponent prep → counter → complete` 的視覺／節奏近似。它不主張對手動作或回球是逐幀真實影片。

**限制**：先維持目前單回合、一次玩家回擊、一次對手回應的 MVP 範圍；不開始多拍 rally、角色動畫系統或完整比賽流程。

### C. Trainer 情境組裝與預覽（第三優先）

**唯一問題**：把已發布的影片交接與已發布的 response timeline 組成玩家可理解的短訓練回合。

**使用者可見流程**：選情境（或由系統指定）→ 開始 → 看球／回擊 → 得到簡短回饋。

**不可見項目**：原始素材選擇、trigger 秒數、JSON、session ID、physics position／velocity、事件 log、WebM 偵錯資訊。這些可在開發模式 Inspector 查看，但不屬 Trainer 預設 UI。

**方向選擇的待決項**：原型目前在開始前預選 left／center／right，這較像測試輸入。Trainer 的實際輸入時機與介面需在 C 階段另行決定；本計畫不預設它一定是按鈕、感測器或自動判讀。

## 5. 發布狀態的概念模型

現有 `pending_generation / ready` 不足以表達校準與人工驗收。下一份實作任務包可考慮下列概念狀態，但名稱、儲存位置與向下相容策略均留待審查後決定：

| 概念狀態 | 意義 | 是否可出現在 Trainer |
|---|---|---|
| `draft` | 素材已建立，但尚未完成標記 | 否 |
| `anchored` | 已標記 trigger／anchor，尚無可接受交接證據 | 否 |
| `fitted` | 有可重跑的玩法近似與誤差報告 | 否 |
| `reviewed` | 人工檢視過結果，仍可保留為未發布 | 否 |
| `published` | 明確被允許作為訓練情境使用 | 是 |

這些狀態只表示產品資料的可用性與審查完成度；**不表示物理模型已被真實量測證明**。

目前的 `real_backspin_001` 應在後續實作開始前被視為「探索性、未校準」案例；本計畫不在本輪直接變更任何既有狀態值。**現況例外必須明示**：repo 目前仍把它的 `generation_status` 設為 `"ready"`，但原始 checkpoint 已記錄這只是探索實驗的操作性設定，尚未完成 SPEC §8 人工驗收，也不代表 handoff 已校準。Phase 0 必須由使用者決定是否降級該值，或至少將它登錄為「資料值為 ready、不得視為已驗收／可發布」的已知例外；在決定前，後續工具不得只依此欄位判定它可進 Trainer。

## 6. 分階段交付與驗收

### Phase 0：範圍收斂（僅文件與任務包）

- 為 A 階段撰寫最小實作任務包。
- 指定一支真人影片作為唯一測試樣本；不擴大到其他題庫影片。
- 定義什麼量測會被顯示、什麼情況只能存草稿、誰可以人工確認。
- 將原型首頁定位為開發／Inspector，不宣稱它是 Trainer。
- 由使用者確認 `real_backspin_001` 現行 `generation_status: "ready"` 要降級，或以文件／程式門檻明確排除其「ready 即已驗收」的錯誤解讀。

**完成條件**：Claude 審查通過且使用者明確批准 A 階段任務包。  
**停止條件**：若無法說明誤差量測與發布門檻，就不進入 UI 實作。

### Phase 1：一支影片的交接校準器

- 僅服務一支指定真人影片與一種既有玩法 profile。
- 以引導式視覺操作替代手動 JSON 編輯。
- 顯示 raw handoff 差距與交接預覽；量測定義必須寫清楚。
- 可儲存草稿與可重跑的校準資料；不可自動發布。

**完成條件**：使用者能從影片選擇到草稿輸出完成一次校準；草稿可重載；已知 large-delta 案例會被清楚標示而不會被標成可用。  
**停止條件**：不調整物理核心、不追求零誤差、不處理第二支影片、不接入正式 Trainer。

### Phase 2：單回合動畫節奏編排（僅供理解整體形狀，未授權）

本節不構成 Phase 2 的任務包或預先批准的驗收標準。只有在 Phase 1 完成、結果被檢視，且使用者另行確認後，才可重新定義本階段的範圍、輸入方式與完成條件。

- 讀取 Phase 1 的已校準輸出（或明確的測試替代資料）。
- 將有限的對手 reaction profile 做成可視化時間軸調整。
- 保留事件名稱與 telemetry 作為 Inspector，不作為一般操作介面。

**完成條件**：一個已知情境可反覆預覽其 prep／counter 節奏，並有明確資料輸出。  
**停止條件**：不擴增 rally、不製作完整角色動畫、不接入正式遊戲。

### Phase 3：受控 Trainer 預覽（僅供理解整體形狀，未授權）

本節不構成 Phase 3 的任務包或預先批准的驗收標準。Trainer 的實際輸入方式、情境範圍與回饋規則，必須在使用者另行確認後重新審查；不得因本文件存在而自動延伸。

- 僅讀取 `published` 的情境。
- 預設 UI 不呈現工程欄位。
- 使用單回合 MVP 的開始、回擊與回饋流程。

**完成條件**：使用者可在不接觸校準參數的情況下體驗一個短回合；Inspector 與 Trainer 可明確切換。  
**停止條件**：不將它聲稱為正式主線整合，直到另有主線評估與批准。

## 7. 驗收與證據原則

每階段都要同時保留機器檢查與人工檢視，但兩者不可互相取代。

- **機器檢查**：schema、資料範圍、媒體 duration／trigger 合法性、狀態轉移、草稿可重載、輸出引用完整性。
- **視覺檢查**：指定 trigger 前後的球點、交接是否明顯跳動、動畫是否在正確事件點出現、Trainer 是否暴露工程資訊。
- **人工發布**：發布只在人工明確確認後發生；工具不得自行把草稿升級為可用情境。
- **不確定性標示**：誤差、近似 profile、手動判讀與未驗證部分必須顯示或被記錄，不能以「ready」一詞掩蓋。

交接誤差門檻的數值、量測畫布與取樣方式尚未批准；Phase 0 必須先把它們寫進任務包。不得從現有原型的測試通過結果直接推論該門檻已達成。

## 8. 主要風險與待決問題

| 問題 | 目前狀態 | 本計畫的處理方式 |
|---|---|---|
| 真人影片與物理初始狀態不相符 | 已在單支探索案例觀察到大落差 | 先做單支校準與誤差可見化；不靠 crossfade 掩蓋 |
| 真人影片 trigger 後剩餘畫面短 | 單支案例只剩約 0.283 秒 | 將「明確切換到訓練場」列為可接受替代；不假裝是真人影片連續畫面 |
| 校準工具又變成工程表單 | 已有 JSON／UV／多個獨立工具的使用性問題 | 以單點標記與既有 profile 的引導步驟為優先；進階值收進 Inspector |
| 範圍擴張成全題庫或完整比賽 | 高風險 | 每階段只接受一個樣本、一個單回合，完成後再決定下一步 |
| 玩法近似被誤當物理真值 | 高風險 | 所有 calibration／profile 都標為玩法近似與驗收資料，不宣稱量測真相 |
| `ready` 被誤當已驗收／可發布 | 現有探索資料已有這個落差 | Phase 0 先由使用者決定降級或建立明確例外門檻；後續不得單憑 `ready` 放行 |
