# 切球 push/chop 接觸模型摘要

> 本文件僅整理既有資料，不做最終物理判斷。所有結論依來源標示「已套用」、「研究中」、「尚未開始」、「不確定」。

---

## 1. 模型目的

- 修正早期回擊模型「完全不感知來球旋轉」的根本問題：舊的 `makeDirectReturnVelocity()`/`makePushReturnVelocity()` 只對速度做反應，物理上不可能自然產生旋轉差異化成敗。
- 改用與桌面接觸力學同一套通用框架（法向反彈 + 切向庫倫摩擦/純滾動判斷），把球拍視為一個有自己法向量、自己揮拍速度的「移動中的平面」。桌面是此通用力學裡「法向永遠 +y、永遠靜止」的特例。
- 讓旋轉數值有真實物理單位（rad/s），行為有物理推導依據，取代早期「感覺係數」（如 `topspin=-1.15`、`topspinKick=0.30`）。
- 為未來膠皮/海綿/底板參數化預留獨立、可覆寫的參數結構。

### 3.5 力道公式（目前部署版本）

```
magnitude = 0.7 − 0.3 × (來球水平速度 − 2.0)，clamp 0.02~2.0
```

- 狀態：已套用
- 設計依據：使用者提出的負回饋原則（對方打快就放鬆減力，對方打慢就有機會加力），純粹反應上一拍來球速度，跟深度完全無關
- 取代了先前多個版本：
  - `speedScaledZ`（`{low:-0.6, high:-0.2, speedLow:1.2, speedHigh:4.2}`）—已被取代
  - Phase 6 聯合公式（`tiltY`+`z` 雙輸出線性回歸）—已被取代
  - Phase 6 再突破的力道公式（`magnitude ≈ 0.0942 − 0.4069×球速 + 0.4183×距離 + 0.0096×|下旋|`）—已被取代
  - 修正回擊起點偏移後的公式（`magnitude ≈ 1.1469 − 0.4989×球速 + 0.2941×距離 + 0.0055×|下旋|`）—已被取代

## 5. 已知問題

### 5.1 架構性問題

1. **`game4.html` 與 `return-studio.html` 實作可能不完全一致**：`return-studio.html` 含 `PADDLE_BLEND`、substepped push 等研究機制，`game4.html` 的可見段落看起來不完全一致，需要專門 diff。
2. **單一瞬間碰撞模型的結構性盲點**：滾動/滑動 regime 轉換是有門檻的非線性行為，可能在接觸中途切換（單點模型只看最終值）；海綿壓縮/釋放是先後次序的彈簧-阻尼行為，不是一個係數能表達。
3. **球拍速度尺度太小**：模擬值 ~0.2 m/s，真實切球估計 1~3 m/s，文獻實測正手拉球 20 m/s，慢了一個數量級以上。
4. **舊公式在引入 blend 後理論上不再成立**：`tiltX` 回歸公式、`tiltY` clamp 範圍、`friction`、`epsilon` 動態公式全部是在「沒有 blend」的舊物理下校準的。

### 5.2 切球特定問題

5. **切球沒有真的製造反向下旋**：連續對打中旋轉會衰減、甚至反轉成上旋（如 -70.5 → +3.5 → +64.8），不是「抵消再反向」。
6. **`swingDirection.y = +0.3` 可能是錯的方向**：使用者資料顯示 `techniqueVel.y` 全部都是負值，真實切球該「往下刷」。
7. **切球法向反彈係數過高造成「子彈感/碰撞感」**：目前 `epsilon≈0.877`，非常接近完全彈性碰撞，不像真實切球「頓一下、滑出去」的觸感。
8. **「準確率」跟「真實反向下旋」互斥**：四參數聯合搜尋證實，在目前架構下沒有任何一組參數同時做到兩者都好。
9. **反手側旋系列旋轉方向持續錯誤**：根因是 `blend` 用完整 3D 相對速度方向時，x 方向來球速度耦合進了決定上旋/下旋的 y-z 平面計算（研究中已找到修正方法）。

### 5.3 連續對打問題

10. **連續對打節奏「短、深、短、深」規律交替**：不是同一個深度重複，推測是發球端跟接發球端兩個人的接觸幾何本來就不對稱。
11. **連續對打球速超出訓練範圍**：4.4~4.7 m/s，超過原本發球場景最高的 4.2 m/s，是外插，不可靠。
12. **連續對打需要不同技術模型**：切球（push）與削球/擋球是不同技術，目前用單一「切球」硬套所有距離/球速是卡住的根本原因。

### 5.4 其他問題

13. **3~4 顆重下旋 preset 曾過不了網**：已修正拋球高度，但修正後拋球高度明顯比其他球高（如 1.55 vs 0.87~0.95），視覺上拋球動作較誇張。
14. **`physics-studio.html` 與 `game4.html` 曾有兩套物理不一致**：已修正，`physics-studio.html` 已換成真實 v2 引擎。
15. **`wristBrakeRate` 是否為有效正式機制**：不確定，需整理確認。
16. **`gravity: -4.2` 是否為正式設計值**：需進一步確認。
17. **短球力道公式行為可能不合理**：短球（來球速度慢）被力道公式負回饋推到比下限更高的力道值，可能是掛網的部分原因。
18. **過網失敗球的出球垂直速度分量異常偏高**：1.89~2.53 vs 成功球 0.55~1.35，可能跟 blend/tiltX 在特定入射角下對有效法向量的影響有關。

## 6. 系統性失敗案例

### 6.1 側下旋長球

- `sidebackspin` 長球在單點測試下仍出界，不同階段記錄 2/7~6/7
- 4 顆側下旋長球在切球窗口後段出界（力道偏強）
- 側旋分量本身可能也需要影響力道校準（目前只依水平速度）

### 6.2 弱旋轉/不轉球

- 使用者指出：真實球員面對不轉球不會選切球，會直接換技術處理
- `no_spin` 已從 Stage 2 聯合校準指標中排除
- sliding regime 對不轉球幾乎不會產生正確方向旋轉（13 組候選裡 0 組正確）

### 6.3 連續對打外插

- 連續對打第二棒接到的球（離網 2.3m、弱下旋、高速 4.4~4.7 m/s）超出切球校準範圍
- 兩個球速相近的樣本需要互斥的調整方向（一個需要更多力道，另一個同樣參數會出界）
- 套用新公式測試連續對打失敗樣本仍失敗（外插不可靠）

### 6.4 連續對打旋轉衰減

- 連續對打中旋轉逐拍衰減、甚至反轉成上旋
- 舊力道公式（`+0.2941×距離`）在連續對打時正回饋失控：擊球點越遠→力道加越多→球彈更深→下一拍擊球點更遠，約 10 拍飛出界

### 6.5 反手側旋旋轉方向錯誤

- 反手方向來球（x 為較大負值）天生會把 blend 機制算出的旋轉推向錯誤方向
- `xVel` 從 -2 到 +1，`outTopspin` 從 +54.6 單調降到 -38.5

### 6.6 短球過網失敗

- 3 顆短球（`backspin_short_backhand_2`、`backspin_short_forehand`、`backspin_short_backhand`）力道公式算出的 `mag` 明顯比其他球高（0.81~0.95），其餘成功球力道都卡在下限附近（0.40）

### 6.7 過網失敗球垂直分量異常

- 6 顆過網失敗球中 4 顆出球垂直速度分量（y）明顯偏高（1.89~2.53），比成功球（0.55~1.35）高出一截
- 調低 `magFloor` 完全沒用：力道公式主要影響整體出球速度大小，不是專門影響垂直/水平的比例分配

### 6.8 視覺不合理案例

- `scale` 因子放大後輸出速度完全不縮小回去：回擊軌跡飛得又高又遠，直接飛出桌子範圍（「子彈感」）
- `tiltX=1.00`（弧度，約 57°）：拍面幾乎轉成側面朝向，配合 blend 產生「像被吸收」的不自然觸感
- 瞄準求解跟實際碰撞用不同物理：側面視角回擊軌跡幾乎是一條直線，不像正常拋物線

## 7. 不合理參數警訊

| 警訊 | 說明 |
|---|---|
| 將 `blend=0.9` 當作最終物理解 | 接手指令明確要求不可視為最終物理解；高 blend 可能補償了缺少 tangential compliance、contact-point velocity、finite racket mass / passive giving 等機制 |
| 用獨立 `contactOffset` 繞過球面/平面接觸幾何限制 | 球碰拍的接觸點由拍面法向量唯一決定（`r = planeNormal × BALL_RADIUS`），不是可獨立選的自由度；加獨立偏移等於製造物理上不自洽的假自由度 |
| 單獨調摩擦係數或揮拍方向，卻不重做聯合校準 | 摩擦係數不只影響旋轉方向，還同時決定球拍碰撞整條水平速度/彈道行為；`tiltX`/`tiltY`/力道公式/瞄準全部都是在特定摩擦行為下校準的 |
| 搜尋目標只看有沒有上桌，不檢查旋轉方向 | 已實渉：搜尋會找到「球還留在桌上，但已經不是切球動作，比較像某種上旋怪招」的組合 |
| `tiltX` 大幅度旋轉（如 57°）在正常切球情境 | 只在「對方發球很快，來不及選位置，把拍面側得很誇張去碰球」的緊急情境下才合理 |
| `scale` 因子放大後輸出速度完全不縮小回去 | 會產生「子彈感」，回擊速度整整快了 3 倍多，完全不像切球 |
| 瞄準求解跟實際碰撞用不同物理 | 已抓到三次同一類 bug（blend 加入時、tiltX 加入時、outputRescale 加入時），每次新增影響最終速度的機制都要同步檢查瞄準求解 |
| 把「立起來、垂直往下切」誤套用到正常下旋球 | 那只適用於旋轉非常弱的球；正常 backspin/sidebackspin 拍面應該要平 |
| 力道公式只看有沒有上桌 | 舊力道公式（`+0.2941×距離`）在連續對打時正回饋失控 |
| 把 `PADDLE_FRICTION=0.4` 當成不可改的常數 | 文獻估計球拍摩擦係數可能高達 1.0~2.5；但已證實摩擦係數在目前模型裡不是瓶頸（rolling regime 天花板） |

## 9. 給高階模型審查的問題

### 9.1 架構決策

1. 「單一瞬間碰撞 + Coulomb 摩擦」架構是否能同時滿足「準確率」跟「真實反向下旋」，還是必須先做 Stage 1（球拍速度尺度校準）或 Stage 3（dwell 動態加速）才能解開核心矛盾？
2. `blend` 機制（反彈方向跟切向滑動方向耦合）是否為物理上合理的建模方向？還是應該用彈簧-阻尼逐步積分模型取代？
3. `blend` 只看 y-z 平面（忽略 x 方向來球速度）的修正是否物理正確？還是應該用其他方式處理反手側旋的旋轉方向問題？
4. `scale` 因子代表模型漏掉的其他物理效應（Stage 3+4+動力鏈），這個簡化是否合理？還是應該分別建模？
5. `outputRescale`（輸出速度部分縮放，不等於內部 `scale`）的物理意義是否清楚？是否應該用更精細的模型取代？

### 9.2 公式重新推導

6. 舊公式（`tiltX` 回歸、`tiltY` 範圍、`friction`、`epsilon` 動態公式）在引入 `blend` 後是否需要全部重新推導？還是有些可以沿用？
7. `tiltX` 回歸公式（10 顆使用者標記樣本）是否已被瞄準機制完全取代？還是有獨立功能需要保留？
8. `tiltY` 在有 `blend` 之後，該對應到什麼物理量？還是依殘留旋轉查表？還是改依其他變數？
9. 力道公式是否需要跟 `blend`/`tiltY` 耦合的設計（不是三個獨立公式各自調）？

### 9.3 技術模型

10. 連續對打是否需要新增「削球/擋球」技術（`model:'chop'`），跟現有的 `push` 分開設計？
11. 拍面角度的插值除了球速，是否也應把旋轉量（`spinSurfaceSpeed` 或 `topspin`/`sidespin`）當作輸入維度？
12. 「時鐘接觸面」幾何模型是否應正式納入揮拍方向設計？如果是，`swingDirection`/`tiltY` 等參數的設計準則該怎麼定義？
13. 弱旋轉/不轉球是否應改用其他技術模型？如果是，技術選擇本身是否需要另外建模？

### 9.4 搜尋方法

14. 目標函式應用硬性雙門檻（`singleOk>=12 且 correctSpinFrac>=0.7` 才給分）還是 Pareto 多目標搜尋？
15. 搜尋演算法是否應改用模擬退火、同時擾動多個參數、或兩階段法（先各自找高準確率跟高旋轉正確率的候選點，再在中間插值）？
16. Node 優化腳本的桌面反彈物理是簡化版，找到的候選是否必須回真實引擎驗證？如何確保兩者一致性？

### 9.5 部署與驗證

17. 部署前確認清單是否應固定包含「瞄準求解跟實際碰撞用同一套物理」這個重複發生的 bug 檢查？
