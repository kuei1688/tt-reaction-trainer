# 切球乾淨參考案例庫

> 這份文件的目的不是找「最佳解」，是累積像「指尖摩擦案例」那樣乾淨、單變數、可重複查閱的參考點——桌球整體物理很複雜，但單項動作的應對是單純的：拍面角度怎麼變會帶來怎樣的變化、來球速度多快大概要對應多大的力量，這些關係本身不該是過度敏感（差一點點結果就劇烈跳動）的。累積這些乾淨案例，之後任何一次校準卡住、陷入混亂時，都能回來找一個乾淨的基礎重新延伸，而不是每次都要重新從頭摸索。
>
> **格式規則**：只能新增條目，不能刪除或覆寫舊條目——舊條目過時了就在條目內標記「已被 XXX 取代」並保留原文，不刪除（呼應 `AI_CONTEXT/00_READ_ME_FIRST.md` 的「不刪除失敗案例」原則）。每個條目必須說明：測試時固定了什麼（尤其是拍子揮拍速度是否用了真實量級，不能為了「乾淨」而簡化成零，見下方教訓）、掃了什麼單一變數、結果的形狀（曲線/表格，不是單一數字）、有沒有觀察到不該有的劇烈跳動。
>
> 已知教訓（詳見 `docs/ARCHIVE/physics-engine-v2-plan.md` 2026-07-14 補記，以及 memory `project_racket-momentum-must-not-be-omitted-in-isolated-tests`）：任何「乾淨單次接觸測試」都必須用貼近真實任務量級的揮拍速度餵進去，不能設成零或極小值，否則量出來的關係會失真，重現「球只被頂高、飛不回去」的舊失敗模式。

## 現況總覽與已知缺口（2026-07-14，累積 6 條目後的階段性檢討）

**目前記錄的物理量**：全部是球**離開球拍那一刻**量到的東西——出球旋轉（topspin/sidespin）、出球速度、球飛越球網那個瞬間的高度（`netClearance`）、有沒有過網（`clearsNet`）；含瞄準求解器的條目另外記了 `inBounds`（左右落點是否合法）跟診斷用的 `aimedX`。

**沒有記錄反彈後的旋轉量**——這是目前最大的缺口。我們完全沒有測「球落到對方桌面、真正彈起來之後，旋轉變成怎樣」。已經有明確證據顯示這件事不能被忽略：`no_spin_long_forehand` 這個代表案例本身宣告的發球旋轉是 `{0,0}`，但因為觸球前已經彈過兩次桌面，實際觸球旋轉變成 `{topspin:45.47, sidespin:0.60}`——反彈本身會顯著改變旋轉，量級可以差到跟原始值完全不同。也就是說，目前所有條目量到的「出球旋轉」，都不等於「對手真正接到這顆球時的旋轉」，兩者中間還隔著一次反彈，這次反彈的影響完全沒有被本案例庫測過。

**現有 6 條目覆蓋的範圍**：2 顆代表球（`backspin_long_backhand`：重下旋+側旋+22.6°入射角；`no_spin_long_forehand`：幾乎無旋轉+近乎直線）× 2 個掃描變數（`PUSH_TILT_Y`、來球速度）× 含/不含瞄準求解器對照。結論穩固但範圍窄：Stage 4a 接觸模型本身（`bounceOffPlaneSubstepped`/`applyPushContact`）在測過的範圍內平滑、沒有真實物理懸崖；原本觀察到的連續對打存活率退步，強烈指向 `solveRacketVelXForTargetLandingX`（瞄準求解器）「無解時靜默回傳假答案」的設計缺陷，不是接觸模型參數沒調好。

**這個結論還不算完整**，因為：(1) 完全沒有測反彈後旋轉，也就是還沒有真正回答「下旋能不能撐過一整個來回」這個最初的問題；(2) 只測過 2 顆球，側旋球、短球都還沒有專屬的乾淨曲線（`game4-push-sustainable-rally-validation.js` 之前的完整回合驗證顯示 sidebackspin 一直是弱項，但案例庫裡目前沒有側旋球的乾淨資料可以對照）；(3) 只掃過 `PUSH_TILT_Y` 跟來球速度兩個變數，`TANGENT_KP`/`PADDLE_BLEND`/`PADDLE_SPRING_K`/`PADDLE_RESTITUTION_LOW/HIGH` 這些原本要校準的接觸模型參數，一次都還沒用同樣的乾淨方法測過。**之前跑過的完整連續對打數字（`avgRounds`/`backspinFraction` 等）全部是透過有問題的瞄準求解器跑出來的，現在需要視為待重新驗證，不能直接沿用當作結論。**


**2026-07-14 尺度一致性修復**：上述所有條目量到的「出球旋轉」和「接球時旋轉」都是在尺度不一致的舊模型下量到的——桌面接觸用模擬尺度速度 + 真實尺度旋轉，導致下旋保留率高估約 18%。修復後（game4.html + return-studio.html），`simulatePath` 輸出的 `spins` 是模擬尺度（需乘 D=1.528 轉真實 rad/s）。現有 6 條目的絕對數值已不適用，但曲線形狀（平滑度、單調性、regime 交界）仍然有效，因為尺度偏差是系統性的比例縮放，不改變曲線的定性特徵。後續新條目必須在修復後的模型上重新量測。

**目前的實驗矩陣規劃見** `AI_CONTEXT/push_experiment_matrix.md`，完整實驗計畫見 `AI_CONTEXT/EXPERIMENT_PLAN.md`。

**固定變數**：

**平滑度檢查**：

1. **有一個明顯的大跳動，在 tiltY=1.0→1.1 之間**：`outTopspin` 從 -3.43 跳到 -20.15（相鄰段落原本都是平滑遞減 ~2~3 一段），`netClearance` 從 -0.162 跳到 +0.007（本身是過網判定的通過/失敗邊界，剛好卡在這裡）。**已用額外診斷欄位 `aimedX`（手動重算 `solveRacketVelXForTargetLandingX` 瞄準求解的 rx 值）找到根因**：tiltY≤1.0 時 `aimedX` 全程死釘在 1.412（= `-incomingVel.x` 這個 solver 內部的初始 fallback 猜測值，不是真正解出來的值）——因為 tiltY≤1.0 時球連過網都過不了，`solveRacketVelXForTargetLandingX` 內部的粗網格掃描（`rx∈[-8,8]` 步進0.1）找不到任何一個「真的有落在對方桌面（z<0）」的候選解，只好整段回傳初始猜測值；一旦 tiltY≥1.1 讓球能過網，網格掃描終於找到候選解，直接跳到接近搜尋邊界的 rx≈7.9，之後隨 tiltY 增加平滑遞減回 6.0 附近。**這正是 game4.html 原始碼自己在 `solveRacketVelXForTargetLandingX` 上方註解裡已經明確警告過的已知現象**（第1438~1440行：「切球在強旋轉/大幅拋高時，rx→落點x 這個函數在滾動/滑動 regime 交界處會有不連續跳動」）——不是接觸模型本身（`bounceOffPlaneSubstepped`）不平滑，是瞄準求解器在「有解/無解」邊界上的 fallback 機制造成的跳動。
2. 次要現象：`outTopspin` 在 tiltY=1.8→1.9 之間變號（-1.11 → +2.16），但這段落差跟前後相鄰段落量級一致（~1~3），是平滑穿越零點的正常現象，不是異常跳動，不需標記。
3. 除了上述 tiltY=1.0→1.1 這個跳動點之外，兩側（0.3~1.0 跟 1.1~2.0）各自都是平滑、大致單調遞減的曲線。

**這個案例告訴我們什麼**：

1. **目前部署的 `PUSH_TILT_Y=1.0` 對這顆代表案例（backspin_long_backhand）剛好落在「過網成功/失敗」懸崖的失敗側**（netClearance=-0.162，掛網），只要 tiltY 往上調 0.1 就會翻正過網——這代表目前的部署值對這顆球來說是一個脆弱、貼著邊界的校準點，不是有安全餘裕的中庸值。這跟 `game4-push-sustainable-rally-validation.js` 觀察到的 Stage 4a 移植後連續對打存活率下降（avgRounds 3.364→0.909）方向一致，值得列為候選根因之一。
2. **tiltY 越大（拍面越平），`|outTopspin|` 大致越小**（下旋量被削弱），跟 `docs/ARCHIVE/physics-engine-v2-plan.md` Phase A 定案的「tiltY 主要負責弧線高度/能否過網，不是旋轉正確性的主要決定者」的結論方向一致——但這裡看到的是「tiltY 對旋轉大小仍有可觀影響（-21.7 到 +2.2 之間變化），不是完全跟旋轉脫鉤」，跟 Phase A 的簡化描述有一點出入，值得後續校準時留意，不要照單全收「tiltY 跟旋轉完全無關」這個簡化假設。
3. **未來重新校準 tiltY 時，不能只看單一點的曲線平滑度，要分別檢查「過網成功」跟「過網失敗」兩個 regime 各自的形狀**——因為兩者中間夾著一個瞄準求解器造成的懸崖，用線性內插橫跨這個懸崖去估計「差不多」的行為會嚴重失真。
4. **`solveRacketVelXForTargetLandingX` 的粗網格搜尋範圍（`rx∈[-8,8]`）本身可能不夠寬**——tiltY=1.1 時解出的 aimedX=7.9 已經非常貼近搜尋上界 8，代表真正最佳解可能落在搜尋範圍之外，被邊界截斷。這是後續如果要處理這個懸崖問題時，一個具體可以動手的地方（擴大搜尋範圍，或改進「找不到解時」的 fallback 邏輯，不要用 `-incomingVel.x` 這種跟真實物理無關的猜測值）。

> **⚠ 重複出現的判讀錯誤註記，2026-07-14**
>
> 程式碼中的 topspin 是繞 X 軸的角速度。同一個正號 canonical topspin，球往不同方向飛時對接收方的意義相反：
>
> | 球飛行方向 | canonical topspin > 0 | canonical topspin < 0 |
> |---|---|---|
> | 朝向你 (+z) | 上旋（觸桌後加速） | 下旋（觸桌後減速） |
> | 離開你 (-z) | 對方接到下旋 | 對方接到上旋 |
>
> 切球把 canonical topspin 從負號翻正（例如 -41 → +38），這不是「產生上旋」——而是在絕對坐標上反轉了旋轉方向。球飛到對方那邊時，對方接到的就是下旋。這是合格切球的定義行為。
>
> **不要用 outgoingTS <= 0 來判斷出球是否為下旋**——出球方向與來球相反，正號 canonical topspin 恰恰就是對方的下旋。之前的 isBackspin 檢查用 <= 0 是符號方向錯誤，已於 2026-07-14 修正為 > 0。
>
> 這個解釋與指尖發球的旋轉反轉能力一致：發球可以產生下旋，切球同樣在絕對坐標上反轉旋轉，對方接到的就是下旋。穩態連纍對打中每一拍都是合格的切球。

### 群組 1：桌面彈跳旋轉轉換曲線（回答 Q4）

#### 實驗 1A：初旋轉 × 衝擊速度 → 彈跳後旋轉

**固定變數**：vz = 3 m/s（真實尺度），sidespin = 0，μ = 0.13

**掃描變數**：topspin ∈ [-250, +250] rad/s（20 點）× |vy| ∈ [1, 6] m/s（10 點）= 200 格

**結果摘要**：

| |vy| (m/s) | topspin=-250 → out | 保留率 | topspin=-125 → out | 保留率 | topspin=0 → out | regime |
|---|---|---|---|---|---|---|---|
| 1.0 | -203.4 | 81.4% | -77.0 | 61.6% | +49.4 | sliding |
| 3.2 | -195.3 | 78.1% | -58.9 | 47.1% | +38.6 | sliding |
| 6.0 | -169.9 | 68.0% | -42.6 | 34.1% | +14.4 | sliding→rolling |

- **regime 邊界線**：rolling/sliding 交界是一條隨 |vy| 移動的曲線。|vy|=1 時邊界在 topspin≈130；|vy|=6 時邊界移到 topspin≈-25。
- **符號翻轉線**：topspin_out = 0 的等高線隨 |vy| 增大往負方向移動。|vy|=1 時在 topspin≈-26；|vy|=6 時在 topspin≈-105。
- **關鍵發現**：弱下旋（|topspin| < 50）在所有衝擊速度下都被翻轉成上旋。中下旋（50~150）保留部分下旋但顯著衰減。重下旋（>150）保留率較高（68~81%）。
- **平滑度**：無劇烈跳動（flagged = 0），曲線完全平滑。

**這個案例告訴我們什麼**：桌面彈跳的旋轉衰減是平滑的，沒有 regime 斷裂。保留率隨衝擊速度增大而下降——這意味著高速下旋球在觸桌後比低速球損失更多旋轉。符號翻轉線是部署時要注意的邊界：如果來球下旋低於此線，推回去的球反而會帶上旋。

## [2026-07-14] 步驟 2：solveRacketVelXForTargetLandingX fallback bug 分析

**已讀取完整實作**（game4.html:1462-1545），未修改程式碼。

### Bug 機制

`solveRacketVelXForTargetLandingX` 的 push 分支（line 1463-1485）：

1. `bestRx` 初始化為 `-incomingVel.x`（物理上無意義的預設值）
2. 粗搜 rx ∈ [-8, 8] 步進 0.1，找讓球落在對方桌面且 landing.x 最接近 targetLandingX 的 rx
3. 若粗搜找不到任何有效著陸（球過不了網或方向不對），`bestRx` 維持 `-incomingVel.x`
4. 二分搜嘗試在 bestRx ± 0.1 範圍精修，但若 `landingForPush(mid)` 回傳 null 或 z≥0 就立刻 break
5. 最終回傳 `(lo + hi) / 2` = bestRx = `-incomingVel.x`

### 觸發條件

當 tiltY/lift/drive 的組合讓球在任何 rx ∈ [-8, 8] 下都無法過網時觸發。回傳的 `-incomingVel.x` 讓 planeVel.x = -incomingVel.x，球以「完全抵銷來球橫向速度」的方向飛出——聽起來合理，但這個值從未經過接觸模型驗證，可能導致球掛網或飛向意料之外的方向。

### 修復方向（待確認後執行）

有兩種策略：

A) **報錯中止**：讓函式在找不到有效著陸時回傳 null（或拋例外），呼叫端 `makeRacketReturnVelocity` 判斷後改用不帶水平修正的 planeVel = {0, lift, -drive}。優點：明確；缺點：改動呼叫端邏輯。

B) **回傳帶標記的值**：回傳 `{rx: -incomingVel.x, fallback: true}`，呼叫端可以選擇使用或忽略。優點：不破壞現有行為；缺點：需要改回傳值結構。

### 與步驟 1 和 3 的關聯

- 步驟 1 的 PADDLE_BLEND 校準確保了在部署值附近球能過網，直接縮小了 fallback 觸發的參數空間。

- 但補償公式只解決側旋引起的偏移，不解決「需要讓球落在桌面中心」的瞄準問題。完整的修復仍需要一個能正確求解 planeVel.x 的瞄準器，或在 fallback 時有合理的降級策略。

- 速度在 5~6 回合內收斂到 2.74 m/s 並保持穩定（速度趨勢 -0.82，輕微收斂非發散）。
- 側旋從 184 rad/s 逐步衰減到 ±7 rad/s 的微小振盪——瞄準求解器成功補償了側旋。
- 過網裕度在 11.6 cm 和 71.7 cm 之間規律振盪（兩個不同彈跳角度交替），最小值 7 cm 仍有安全裕度。
- 落點 x 在 ±0.06 m 之間振盪，遠在桌寬 ±0.76 m 範圍內。
- 回合 6 後進入完美的週期性穩態——真正的穩定循環。

### 測試 B：繞過瞄準求解器的掃描（對照組）

**設定**：同群組 5 的 (tiltY, lift, drive) 80 格掃描，但 PADDLE_BLEND=0.605

**結果**：仍然 0/80 穩定（3 個 preset 全部失敗）。分類與群組 5 幾乎相同（掛網 2~4、飛出界 76~78）。

這證實了：**PADDLE_BLEND 校準本身不解決連續對打問題——瞄準求解器才是關鍵。** blend 從 0.65 改到 0.605 改善的是單球過網裕度，但沒有水平修正就無法維持連續對打。穩定循環的實現來自三項改動的組合：blend 校準（給予過網安全邊界）+ fallback 修復（讓 solver 在邊界條件下仍能降級運作）+ 瞄準求解器本身（每回合將球拉回桌面中心）。
