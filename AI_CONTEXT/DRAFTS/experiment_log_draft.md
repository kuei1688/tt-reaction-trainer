## EXP-001：Phase 1 獨立接觸力學函式驗證

### Experiment ID
EXP-001

### 日期
不確定（文件標註 2026-07-02，Phase 1 完成日）

### 目的
在 Node.js 腳本裡（不影響正式檔案）先把新的桌面接觸力學函式寫出來並驗證。

### 模型版本
`tools/physics-v2-contact-mechanics.js`（獨立 Node 腳本）

### 固定參數
- 薄殼球轉動慣量：`I = (2/3) m R²`
- `m = 2.7g`
- `R = 0.02m`
- ITTF 球桌反彈係數：`ε = 0.876`（30cm 落下彈回 23cm）

### 搜尋參數
不確定（文件未以「搜尋參數」形式列出，Phase 1 為函式實作與單元測試）

### 測試案例
- 零旋轉球反彈行為（sanity check，跟舊簡化模型結果接近）
- 純下旋球撞桌後水平速度被拉低、方向正確
- 強力上旋球撞桌後水平速度被推快、方向正確
- 純滾動 vs 打滑兩種分支都能被觸發、行為合理

### 結果
- 文件記錄：最初 `7/7 通過`
- 修正 `vz` 飽和問題後：`13 通過 / 0 失敗`
- 新增測試「同樣球心速度下，20rps 下旋撞桌後最高點低於不轉球」通過
- `vz=1.5` 實戰基準回歸測試：不轉 `ε≈0.857`、apex≈`1.568m`；20rps 下旋 `ε≈0.807`、apex≈`1.479m`；40rps 下旋 `ε≈0.757`、apex≈`1.394m`

### 失敗案例
- 第一版把 83° 斜向撞擊換算成 `normalSpeed / tan(83°)`，導致 `vz=1.5` 被誤判成極端擦撞，不轉、20rps、40rps 都直接掉到 `EPSILON_MIN=0.45`，高度完全一樣
- 修正後改為兩層模型（`baselineEpsilon` + `spinPenalty`）解決

### 判斷
文件標示「✅ 已達成」，Phase 1 結束。

### 下一步
進入 Phase 2 摩擦係數校準。

## EXP-002：Phase 2 摩擦係數 μ 校準

### Experiment ID
EXP-002

### 日期
不確定（文件標註 2026-07-02，Phase 2 完成日）

### 目的
因查不到桌球球體 vs 球桌表面的實測摩擦係數 `μ`，用已知「應該有的效果」反推合理值。

### 模型版本
`tools/physics-v2-contact-mechanics.js` + `physics-v2-calibration.html`（視覺化校準頁）

### 固定參數
- `ε = 0.876`（ITTF 正對垂直落下）
- 入射角 83° 斜向撞擊 `ε = 0.57`（文獻實測）
- `EPSILON_MIN = 0.45`（暫定工程常數）
- `SPIN_EPSILON_REFERENCE = 6.0`（暫定工程常數）

### 搜尋參數
- `μ`（摩擦係數）

### 測試案例
- `vz=1.5` 時，`μ=0.12~0.14` 的球感主觀校準
- rps 增加時弧線逐步壓低（不低到貼桌面）
- 位移縮短一些（不突然變超短球）
- 第二跳回彈屬極端趣味情況，非實戰校準目標

### 結果
- **定案：`μ = 0.13`**（`0.12~0.14` 的中點）
- apex 隨旋轉增強嚴格單調遞減：0rps `1.568m` → 20rps `1.479m` → 40rps `1.395m` → 60rps `1.363m`
- `vzOut` 非單調：`0.900 → 0.795 → 0.815 → 0.823`（20rps 最低，之後回升）

### 失敗案例
- 水平位移（`vzOut`）非單調：原因為 `ε` 被旋轉壓低後，法向衝量變小，摩擦力上限跟著變小，旋轉越強、能用來減速的摩擦力反而越少
- 模型架構本身缺少耦合機制（垂直方向只用固定 `ε`，不被旋轉影響）→ 已改用動態 `ε` 解決

### 判斷
- 文件標示「✅ 已達成，Phase 2 結束」
- 使用者原始校準標準明確講過「速度相對減弱不是必要條件」，只有「弧線壓低」才是可靠依據
- 水平位移非單調性不牴觸使用者定義的驗收標準
- `EPSILON_MIN` 與 `SPIN_EPSILON_REFERENCE` 仍為暫定工程常數，非最終校準值

### 下一步
進入 Phase 3 既有 16 顆發球換算成真實單位。

## EXP-006：Phase 5 techniqueVel 重新校準（第一版，已被取代）

### Experiment ID
EXP-006

### 日期
不確定（文件位於 Phase 5）

### 目的
在新桌面反彈物理下重新校準攻球/切球的舊 `techniqueVel`。

### 模型版本
`game4.html` / `return-studio.html`

### 固定參數
- `PADDLE_RESTITUTION = -0.9`（不變）
- 排除 Phase 5 已知壞掉的 3 顆 preset

### 搜尋參數
- `techniqueVel.{x,y,z}`（攻球/切球共用同一組）
- 搜尋方法：粗網格搜尋起點 + coordinate hill-climbing 精修
- 評分函式：最小化「掛網懲罰 + 出界懲罰 + 落點深度懲罰」

### 測試案例
13 顆有效發球

### 結果
- 攻球：`techniqueVel = {x:0, y:0.4, z:-3.52}`（原本 `{x:0,y:0,z:-0.5}`）
- 切球：`techniqueVel = {x:0, y:0.74, z:-2.0}`（原本 `{x:0,y:0,z:-0.35}`）
- 攻球 11/13、切球 12/13 顆有效發球能成功過網並落在界內

### 失敗案例
- 第一版評分函式有漏洞：只要路徑「曾經」穿越 z=0 平面就算「有過網」，優化器鑽漏洞找到大幅往下壓的 `y`（球直接砸在自己這側先落地，亂彈路徑又剛好穿越 z=0）
- 修正為「z=0 穿越事件必須發生在第一次落桌之前才算過網」後，搜尋才收斂到合理解
- **關鍵問題：** 校準只用「盡量讓球過網」當目標，完全沒考慮旋轉，`y:0.4` 讓攻球對 9 顆下旋/側下旋長球中的 8 顆都打得穿，違反 Phase 0 設計原則
- 根本原因：`makeDirectReturnVelocity()`/`makePushReturnVelocity()` 完全沒有讀取來球的 `spin`

### 判斷
⚠️ **已被 Phase 6 取代，僅留存記錄。** 不要把此結果視為目前模型依據。

### 下一步
開啟 Phase 6 球拍接觸力學。

## EXP-007：Phase 6 球拍接觸力學 — 校準與驗證

### Experiment ID
EXP-007

### 日期
不確定（文件位於 Phase 6）

### 目的
修正攻球/切球完全不感知來球旋轉的根本問題。改用跟 Phase 1 同一套接觸力學，把球拍視為有自己法向量、揮拍速度的「移動中的平面」。

### 模型版本
`tools/racket-contact-mechanics.js`（獨立 Node 腳本）→ 整合進 `game4.html` / `return-studio.html`

### 固定參數
- `PADDLE_FRICTION = 0.4`（工程估計，比桌面 `μ=0.13` 高，標記為暫定值）
- `PADDLE_RESTITUTION_LOW = 0.9`（低速 ~2m/s）
- `PADDLE_RESTITUTION_HIGH = 0.75`（高速 ~12m/s）
- `PADDLE_SPEED_LOW = 2.0`、`PADDLE_SPEED_HIGH = 12.0`
- 球拍材質參數化設計：之後可覆寫兩個常數（反彈/摩擦），不用動接觸力學本身

### 搜尋參數
- `techniqueVel.{y,z}`（`x` 由瞄準修正自動解出）
- 拍面法向的 y 傾角 `tiltY`
- 搜尋方法：自動化搜尋，只用 2 顆無旋轉發球當基準

### 測試案例
- 校準基準：`no_spin_long_forehand` / `no_spin_long_backhand`（2 顆無旋轉發球）
- 驗證：全部 16 顆發球（扣掉 Phase 5 已知 3 顆過不了網的）

### 結果
- **最終校準結果（含 `dynamicPaddleEpsilon` 後）：**
  - 攻球：`techniqueVel = {x:0, y:-0.234, z:-1}`，`racketNormal = normalize({x:0, y:0.1, z:-1})`
  - 切球：`techniqueVel = {x:0, y:0.011, z:-1}`，`racketNormal = normalize({x:0, y:0.1, z:-1})`
  - 2 顆無旋轉基準：攻球/切球皆成功（netClearance ≈ 0.08~0.09）
- **16 顆驗證結果：**
  - `no_spin`（2 顆）：攻球/切球 4/4 全部成功
  - `backspin`（8 顆，扣掉 Phase 5 已知壞掉的）：攻球/切球 0/8 全部失敗（掛網或出界）
  - `sidebackspin`（14 顆，含 1 顆短球）：2/14 成功，都是短球 `sidebackspin_short_forehand`
- `game4.html` 跟 `return-studio.html` 兩邊批次驗證結果完全一致
- 桌面退化案例對拍驗證：通用版拍面法向設為 `(0,1,0)`、拍面速度設為 `0`，跟已驗證的桌面公式完全一致

### 失敗案例
- 第一版切平面正交基底 `e1`/`e2` 對應的「有效角速度」符號寫反了，導致連零旋轉的簡單案例都輸出錯誤符號
- 反手發球的側向速度被接觸力學放大（例：`no_spin_long_backhand` 的 `hitVel.x ≈ -0.92`），導致回擊側向亂飛、出界

### 判斷
- 文件標示「✅ 已達成，結果完全符合 Phase 0 的設計原則」
- 重下旋長球確實打不穿，短球下旋打得回去，符合預期模式
- 反手側向速度放大問題已加入「瞄準修正」機制 `solveRacketVelXForTargetOutX()` 解決

### 下一步
- 回擊技術等級（初階/中階/熟練）
- 切球時機容錯窗口擴大
- 切球力道依球速動態調整

## EXP-010：Phase 6 切球力道依球速動態調整

### 日期

### 目的
診斷切球為什麼沒有製造反向下旋，連續對打中球的旋轉會衰減、甚至反轉成上旋。

- `swingDirection.y = +0.3`（往上帶）

- **第一層診斷：** `swingDirection.y = +0.3` 是往上帶，比較像「往上挑」的攻擊動作，不是真正切球「往下刷過球」的動作
- 測試把 y 改成負值：拍面接觸後的旋轉確實會跟著變負（y=-0.5 時可以做到 -22.3 rad/s 的反向下旋）

- 真正卡住的是球拍速度本身：目前切球的球拍速度模擬出來只有 0.198 m/s，真實世界估計至少 1~3 m/s，文獻實測正手拉球高達 20 m/s

- 光換揮拍方向會讓其他公式全垮：單拍準確率從 backspin 7/7 掉到 1/7、連續對打穩定顆數從 13/16 崩到 1/16

### 判斷
- 真正卡住的不是參數不夠精細，是模型本身有一個由球拍速度決定的「滾動天花板」，而且這個天花板現在被壓得太低
- 就算球拍速度校準到位，真實技術還是能超越這個「單一瞬間碰撞」的天花板，靠的是完全沒建模的兩個機制：接觸期間球拍持續加速、海綿彈性儲能/釋放

### 下一步
- Stage 1：球拍速度尺度校準
- Stage 2：揮拍方向正負號修正 + 摩擦係數 + 重新校準
- Stage 3：加入「接觸期間球拍會加速」的效果
