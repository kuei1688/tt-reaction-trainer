## 背景：已經查到的物理依據

來源：Ricard Alert Zenón, "The physics of table tennis"（RFETM 桌球協會教材）+ Phys. Rev. E 107, 055007 (2023) "Oblique impact of a buckling table-tennis ball on a rigid surface" 的 synopsis。

### 球體參數
- 桌球是**薄殼球**，轉動慣量 `I = (2/3) m R²`（不是實心球的 `(2/5)mR²`）
- `m` = 質量（ITTF 規格 2.67~2.77g，取 2.7g）
- `R` = 半徑（ITTF 規格直徑 40mm，`R=0.02m`）

### 反彈係數（垂直方向，已查證、已套用）
- ITTF 球桌反彈測試：30cm 落下彈回 23cm → `ε = √(23/30) ≈ 0.876`
- 已經在 `physics-presets.json` 套用（0.95 → 0.88，commit `138d84b`）

### 接觸力學（水平/旋轉方向，還沒套用）
- 庫倫摩擦定律：`F_f = μN`（`μ` 是摩擦係數，`N` 是法向反作用力）
- 兩種接觸情境（這是這次工程最核心的判斷邏輯）：
  1. **入射角小於臨界值 → 純滾動**：接觸時間內球從打滑變成純滾動，**不需要知道 μ**，純粹用動量守恆 + 角動量守恆就能精確算出反彈後的線速度跟角速度
  2. **入射角大於臨界值 → 全程打滑**：接觸期間球持續打滑，反彈結果直接跟 `μ` 掛勾，摩擦力全程等於 `μN`
- 轉速改變量公式：`ωf ≈ ωi − 3τδN / (2mR²)`（`τ`=接觸時間，`δ`=摩擦力矩偏移量）
- 球心額外橫向動量（觸桌後球心偏移的來源）：`L_CM,f ≈ −τδN`

### 已知的缺口
**沒有查到「桌球球體 vs. 球桌表面」的實測摩擦係數 `μ`。** 這代表 `μ` 最終要靠校準，不是查表查到的。物理框架是對的，但 `μ` 這個常數的「真實性」有極限，要誠實面對這件事。

### 真實轉速參考值（用來把 topspin/sidespin 換算成真實單位）
2009 世界錦標賽日本研究：發球轉速 13.7~62.5 rps，男子平均 ~46 rps，女子平均 ~39 rps。
換算：`ω(rad/s) = 2π × rps`

#### 座標與單位規格（定案）

**座標系統：** 沿用現有引擎的世界座標（`x`=左右、`y`=上下、`z`=前後，正方向朝接發球方）。角速度 `ω` 用同一組右手座標系表示：

- **`ωx`（對應現有的 `topspin` 欄位）**：旋轉軸沿左右（x）方向，控制上旋/下旋。正負號延續現有程式碼慣例——正值＝上旋（觸桌後加速前進），負值＝下旋（觸桌後減速甚至反向），跟 `applyBounceSpin()` 現有的 `forwardSpeed = speed*0.98 + topspin*0.30` 方向定義一致，不用重新定義正負號。
- **`ωz`（對應現有的 `sidespin` 欄位）**：旋轉軸沿前後（z）方向，控制側旋（左右偏移/觸桌側滑）。正負號延續現有程式碼慣例，且依使用者確認：目前所有發球都只模擬反手方向，側旋一律是負值。
- **`ωy`**：暫不使用（保留給之後可能的特殊旋轉類型，目前技術庫沒有用到垂直軸自旋），固定為 0。

**單位：** 每個 preset 的 `spin.topspin`/`spin.sidespin` 直接改成儲存**真實角速度，單位 rad/s**（取代現在的任意數字，如 `-1.15`）。JSON 欄位名稱不變，只有數值意義跟單位改變。

**換算公式：**
```
ω (rad/s) = 2π × rps
```
下旋基準 20 rps → `ω = 2π × 20 ≈ 125.66 rad/s`（Phase 3 實際指定各 preset 數值時的主要錨點）。

**留給 Phase 3 決定的細節（不是 Phase 0 要解決的）：** 側下旋 preset 同時有 `topspin` 跟 `sidespin` 兩個分量時，兩者合成的「總旋轉量」`√(ωx²+ωz²)` 要不要受到某個物理上限（球拍摩擦不可能無限製造旋轉），或是 20rps 這個錨點該套用在單一分量還是合成總量——這個留到 Phase 3 實際指定 16 顆 preset 數值時再決定，不影響 Phase 0 的座標/單位規格本身。

#### 使用者確認的旋轉設計原則（2026-07-02 對話中確認，任何人调整旋轉數值都要照這個方向）

1. **同一位發球者，下旋（topspin/backspin）強度應該固定，不管長球短球**——不是「總動能固定、依長短分配」這種籠統說法。下旋是這個訓練系統的核心考驗對象，長短球的下旋強度理想上要一樣強。
2. **短球的側旋（sidespin）應該比長球多**。目前 `physics-presets.json` 裡的數值是反過來的（`側下短球到正手` sidespin=-1，比大多數長球的 -1.23 還弱）——**這是測試遺留的錯誤數值，Phase 3 重新指定真實轉速時要修正，不要照現有數值比例去換算**。
3. **主群集（10 顆 preset，topspin≈-1.15）代表「強度足以讓平擊攻球掛網/失敗」的下旋基準**，是用真實球感驗證出來的（軌跡看起來像使用者實際打球時看到的樣子）。這是換算真實 rps 時的主要錨點。
4. **3 顆例外（`下旋長球到反手` topspin=-0.8、`下旋短球到反手-2` topspin=-1 且 sidespin=+0.37 方向相反、`側下長球到正手` topspin=-0.35）是早期測試遺留物**，當時只是在驗證系統能不能正確算出第一跳/第二跳落點，不是刻意設計的「弱旋轉版本」。Phase 3 應該把這 3 顆校正到跟主群集一致的下旋強度，**不要當成獨立的強度等級保留**。
5. **側旋方向應該統一是負的**（目前只在模擬反手方向發球）。`下旋短球到反手-2` 的 `sidespin=+0.37` 是遺留的測試值/錯誤，Phase 3 要修正成負值。
6. **下旋轉速的真實單位錨點：已定案 = 20 rps。** 使用者是業餘水準（校隊練兩年左右的程度），推導過程：
   - 職業選手「拉球」轉速平均 140 rps，職業選手「發球」轉速平均 46 rps，兩者比例約 1:3（發球動作受限，轉速天生比拉球低）
   - 業餘/俱樂部球友「拉球」轉速約 60 rps（有分級研究資料：初學者 ~30rps、中階俱樂部 ~60rps、俱樂部高手 ~100rps、職業 ~140rps）
   - 用職業選手的發球:拉球比例（1:3）反推業餘球友的發球轉速：`60 ÷ 3 ≈ 20 rps`
   - 兩條路徑（「業餘水準明顯低於職業發球平均 46rps」+「用拉球:發球比例反推」）都收斂到同一個數字，不是巧合湊出來的
   - **設計用途：** 這個 20 rps 是「主群集」（10 顆 preset，代表足以讓平擊攻球掛網的下旋強度）的真實轉速。**這也是驗證新物理模型夠不夠真實的檢查點**——如果新模型算出來 20rps 還不足以讓平擊攻球失敗，代表模型本身或攻球的 `techniqueVel` 需要重新檢討，不該反過來把轉速數字往上調去湊結果。

**Phase 1 過程中發現的重要物理細節（跟舊模型不一樣，要記住）：**
「上旋是否會在觸桌後加速球」不是只看 `topspin` 正負號，是看**接觸點的滑動方向會不會真的反轉**（`ωR` 有沒有超過 `v`）：
- `ωR < v`：上旋只是讓減速變少，球觸桌後**還是會減速**（只是減得比零旋轉時少）
- `ωR > v`：接觸點滑動方向真的反轉，這時候上旋才會讓球**觸桌後加速**

這是一個門檻效應（threshold effect），舊模型（`forwardSpeed = speed*0.98 + topspin*0.30`，topspin 線性疊加）沒有這個門檻，只要 topspin 是正的就線性加速，跟真實接觸力學不一樣。**Phase 5 重新驗證 16 顆發球時要注意這一點**——同樣的 `topspin` 數值，配上不同的球速，可能會落在門檻的兩側，產生質變的行為（減速 vs 加速），不是線性效果，調參數時不能再假設「topspin 越大、加速越多」這種線性直覺。

#### 收尾：水平位移不單調的問題怎麼處理（2026-07-02）

上一輪發現：`μ=0.13` 時，弧線最高點（apex）隨旋轉增強是嚴格單調遞減（0rps 1.568m → 20rps 1.479m → 40rps 1.395m → 60rps 1.363m，符合「弧線壓低」的校準標準），**但彈起後的前進速度（vzOut）不是單調的**（0.900 → 0.795 → 0.815 → 0.823，20rps 最低，之後又回升）。

原因：`ε` 被旋轉壓低後，連帶讓法向衝量 `J_n=m(1+ε)|vy|` 變小，摩擦力上限 `μN` 也跟著變小，旋轉越強、能用來減速水平方向的摩擦力反而越少——這是 `ε` 與摩擦衝量共用同一個中介變數造成的耦合副作用。

**判斷：不修改架構，直接接受這個結果。** 使用者原始的校準標準明確講過「速度相對減弱**不是必要條件**」——只有「弧線壓低」才是可靠依據，而弧線壓低這件事是嚴格單調、完全符合標準的。水平位移的非單調性不牴觸使用者定義的驗收標準，不需要為了一個沒被要求的性質去增加模型複雜度。

**驗收標準：** `μ` 有明確數值，校準依據寫進文件。**已完成區間判斷，尚未鎖定單一最終值。**

#### 中途發現的模型缺口（下一個接手的人先看這段）

用 Phase 1 的 `tools/physics-v2-contact-mechanics.js` 實際跑數字驗證使用者描述的「弧線往下壓」現象，發現**現在的模型做不出這個效果**：

- 目前模型把「垂直方向（彈起高度）」跟「水平方向（前進速度/旋轉摩擦）」完全解耦：垂直方向只用固定 `ε=0.876`（跟旋轉、摩擦完全無關），水平方向才會被 `μ`/旋轉影響
- 實際測試（20rps 下旋 vs 不轉 vs 40rps 下旋，同樣的落桌前速度）：**彈起後的最高點高度完全一樣（0.8209m），不管旋轉多強都不會變**——因為模型結構上垂直方向本來就不可能被旋轉影響到
- 這代表：不管 `μ` 校準成多少，都做不出使用者要的「弧線壓低」效果，因為問題不在 `μ` 的數值，是在模型的**架構**本身少了一個耦合機制

**已經查到的解法依據（不是憑空提議）：** Phase 0 背景資料裡「The physics of table tennis」那份簡報，有一組**斜向撞擊的實測數據**：入射角 ~83°（不是正對垂直落下）量到的反彈係數是 `ε=0.57`，比 ITTF 正對垂直落下量到的 `ε=0.876` 低很多（原文有提到 "reflection and incidence angles do not match due to collision inelasticity"）。這代表**真實球體斜向、帶旋轉撞桌時，垂直方向本來就會損失更多能量，`ε` 不是固定的 0.876**。

**已確認採用的下一步（2026-07-02）：**
把 `ε` 從固定常數，改成依撞擊角度（或水平/垂直速度比、或摩擦劇烈程度）動態調整的函數，用手上這兩個校準點內插：
- 入射角 90°（正對垂直落下，無水平分量）→ `ε=0.876`
- 入射角 83°（有明顯水平分量的斜向撞擊）→ `ε=0.57`

最初實作在 `tools/physics-v2-contact-mechanics.js`，仍然沒有動 `game4.html`。第一版選擇不是只看球心入射角，而是看**接觸點切向滑動速度**：

```
slipZ = vz - ωxR
slipX = vx + ωzR
slipSpeed = sqrt(slipX² + slipZ²)
```

這樣同樣球心速度下，下旋會因為接觸點滑動更劇烈而得到較低的動態 `ε`，才能做出使用者描述的「弧線壓扁/壓低」。但第一版把 83° 的斜向撞擊換算成 `normalSpeed / tan(83°)`，導致 `vz=1.5` 這種正常前進速度也被誤判成極端擦撞；不轉、20rps、40rps 都會直接掉到 `EPSILON_MIN=0.45`，高度完全一樣。

修正後的模型改成兩層：

```
baselineEpsilon = f(horizontalImpactSpeed / (normalSpeed * tan(83°)))
spinPenalty = f(spinSurfaceSpeed)
epsilon = clamp(baselineEpsilon - spinPenalty, EPSILON_MIN, EPSILON)
```

也就是：正常水平前進速度只影響「入射角基準反彈」，旋轉球面速度才額外壓低 `ε`。這樣 `vz=1.5` 的不轉球不會被誤殺，下旋仍會因 20/40rps 產生較低彈跳。

### Phase 6：球拍接觸力學 ✅
**目標：** 修正「攻球/切球完全不感知來球旋轉」的根本問題。改用跟 Phase 1（球對桌面）同一套接觸力學（法向反彈 + 切向庫倫摩擦/純滾動判斷），把球拍視為一個有自己法向量、自己揮拍速度的「移動中的平面」——桌面是這套通用力學裡「法向永遠 +y、永遠靜止」的特例。

**使用者的長期考量：** 之後球拍會有不同膠皮（正膠/反膠/防弧）跟底板（全木/碳纖維），這些都會影響回擊表現。所以這次不是單純修 bug，而是要把「球拍材質」做成獨立、可覆寫的參數，即使現在只給一組共用預設值，之後要做不同膠皮設定時只需要覆寫兩個常數，不用動接觸力學本身。**球的材質差異使用者決定先忽略不計。**

**文獻依據（WebSearch，非精確數值，工程估計居多）：**
- 拍面法向反彈係數：多篇桌球膠皮衝擊研究（[MDPI 9(1):158](https://www.mdpi.com/2076-3417/9/1/158)、[ScienceDirect Non-Linearity of Ball/Rubber Impact](https://www.sciencedirect.com/science/article/pii/S1877705816307548)）指出橡膠反彈係數約在 **0.8~0.9** 之間。
- **反彈係數會隨撞擊速度變化，不是固定值**（[ISJOS Impact Velocity and CoR](https://www.isjos.org/pdfs/ISJOS_v15_p2.pdf)、ScienceDirect 同上）：CoR 會隨撞擊速度增加**線性下降**，低速（~2 m/s）到高速（~12 m/s）之間差異可達 **0.15**。這點原本查到但沒有整合，後來使用者主動要求補上——已實作成 `dynamicPaddleEpsilon()`，比照 Phase 2 幫桌面做的 `dynamicEpsilon()`：低速端 `PADDLE_RESTITUTION_LOW=0.9`，高速端 `PADDLE_RESTITUTION_HIGH=0.75`，用撞擊速度（拍面相對速度的法向分量）在 `PADDLE_SPEED_LOW=2.0` ~ `PADDLE_SPEED_HIGH=12.0` 之間線性插值。物理意義：殺球等重擊會讓球拍損耗更多能量比例，速度越快、反彈效率越低，這一點原本的固定係數版本沒有反映。
- 拍面切向摩擦係數：ITTF 自 2008 年起規定膠皮最低摩擦力為 25 mN（用專用儀器量測的規範值，不是直接的無因次庫倫 μ，無法直接套用）。查無精確的無因次摩擦係數文獻值。**工程估計** `PADDLE_FRICTION = 0.4`（比桌面的 `μ=0.13` 高，因為橡膠抓球力明顯強於桌面硬質貼合材），標記為之後可能需要再校準的暫定值，之後要做膠皮差異化時應該是主要調整對象。

- **意外發現並修正的問題：反手發球的側向速度會被接觸力學放大。** 校準過程中發現，反手方向發球在擊球點本身就帶有明顯側向速度分量（例：`no_spin_long_backhand` 的 `hitVel.x ≈ -0.92`）。舊的簡單加法模型下這個分量影響很小，但新模型下它會透過拍面法向量的向量分解被放大，導致回擊側向亂飛、出界。**解法：** 加入「瞄準修正」機制 `solveRacketVelXForTargetOutX()`——用割線法數值反推球拍速度的 x 分量，讓打出去的球側向速度趨近 0，模擬真實球員會依來球側向飄移自然調整拍面角度把球打直的行為，而不是死板地用固定揮拍方向。

**校準（自動化搜尋，只用 2 顆無旋轉發球當基準，刻意不用下旋 preset 校準）：**
- 使用者明確指出第一版校準的錯誤：不應該拿全部 13 顆（含 9 顆下旋/側下旋）一起搜「怎樣參數讓最多顆成功」，這等於在教攻球「作弊打穿旋轉」。正確做法是只用 `no_spin_long_forehand`/`no_spin_long_backhand` 這 2 顆無旋轉發球去校準「一般平擊回擊該有的基礎力道跟拍面角度」，再讓下旋 preset 的成功/失敗自然浮現，不去人工調整讓它們成功。
- 搜尋變數：`techniqueVel.{y,z}`（`x` 由瞄準修正自動解出，不參與搜尋）、拍面法向的 y 傾角 `tiltY`。
- **最終校準結果**（已套用，含 `dynamicPaddleEpsilon` 後的重新校準值）：
  - 攻球：`techniqueVel = {x:0, y:-0.234, z:-1}`，`racketNormal = normalize({x:0, y:0.1, z:-1})`
  - 切球：`techniqueVel = {x:0, y:0.011, z:-1}`，`racketNormal = normalize({x:0, y:0.1, z:-1})`
  - 2 顆無旋轉基準：攻球/切球皆成功（netClearance ≈ 0.08~0.09）。

**驗收標準：** 用只靠無旋轉球校準出的參數去測全部 16 顆發球（扣掉 Phase 5 已知的 3 顆過不了網的），旋轉應該要自然造成差異化的成功/失敗結果，不需要額外的人工補償。✅ **已達成，結果完全符合 Phase 0 的設計原則：**
  - `no_spin`（2 顆）：攻球/切球 **4/4 全部成功**。
  - `backspin`（長球/短球共 8 顆，扣掉 Phase 5 已知壞掉的 preset）：攻球/切球 **0/8 全部失敗**（掛網或出界）——重下旋長球確實打不穿。
  - `sidebackspin`（14 顆，含 1 顆短球）：**2/14 成功，都是短球 `sidebackspin_short_forehand`**——短球接發點離網近、旋轉來得及生效的時間短，跟「短球下旋還打得回去，長球下旋打不回去」的預期模式一致；其餘長球/半長球側下旋全部自然失敗。
  - `game4.html` 跟 `return-studio.html` 兩邊批次驗證結果完全一致（16 preset serve 無 NaN；attack/push/loop 三種技術 × 16 preset 皆無執行錯誤）。

**拍面反彈係數速度相依性（使用者主動要求補上的文獻整合）：** 稍早查文獻時發現但沒整合的資訊——球拍膠皮的反彈係數會隨撞擊速度**線性下降**（低速~2m/s 約 0.9，高速~12m/s 約 0.75），比照 Phase 2 幫桌面做的 `dynamicEpsilon()`，新增 `dynamicPaddleEpsilon()`，讓殺球等重擊自然損耗更多能量比例。重新跑過校準，旋轉差異化結果不變（無旋轉 4/4、長球下旋 0/8、短球下旋 2/14）。

**回擊技術等級（初階/中階/熟練）：** 使用者要求把「瞄準修正」（`solveRacketVelXForTargetOutX`，讓打出去的球側向速度趨近 0）做成依技術等級而有不同完成度的機制，而不是永遠完全修正。
- 設計：`RETURN_SKILL_LEVEL` 是獨立於 `DIFFICULTY_MODE`（那個管發球時機提示）的新設定，UI 新增「回擊技術等級」下拉選單。`makeRacketReturnVelocity()` 先解出「完全修正側向」所需的球拍 x 分量 `aimedX`，再乘上一個「補償完成度」係數 `correction`（1=完全修正，0=完全不修正）。
- **踩坑：** 一開始想用「完全不修正」跟「完全修正」兩個端點的完整結果向量（x/y/z）做內插，結果初階玩家的球幾乎必定飛出桌面外好幾公尺。追查發現不是接觸力學耦合造成的異常，而是**正確的物理結果**：側向補償不足時，球在該經過桌邊的瞬間 x 位置已經超出桌寬，程式正確判定「沒有落在桌上」，於是不記錄反彈，球直接以拋物線飛越整張桌子落到地板——距離看起來很誇張，但物理上是對的。後來改成直接內插「球拍 x 分量」本身（`aimedX * correction`），而不是內插最終結果，作法更直接也更好懂。
- 使用者中途簡化需求：「中階直接改成可以修正成功就好，不要這麼複雜，只有初階會失誤而已」——於是中階目前的補償品質跟熟練完全相同（`correctionMean:1.0`，無失手），只有初階有補償不足 + 隨機失手的機制。中階/熟練之後如果要再拉開差異（例如熟練主動把球打到離對手最遠處），要另外設計，這次沒做。
- **參數（`RETURN_SKILL`，已套用到兩個檔案）：**
  - 初階：平均補償 30%（`correctionMean:0.3`，±15% 隨機浮動），35% 機率「失手」（補償降到 -10%~15% 之間，等於幾乎沒修正甚至修正錯方向）。
  - 中階/熟練：補償 100%，無隨機性，等同 Phase 6 已驗證的預設行為。
  - 實測：初階在 2 顆無旋轉發球（唯一有明顯側向飄移、需要補償的案例）上，攻球+切球整體成功率約 50%（因為不同 preset 的原始側向飄移量差異很大，飄移小的即使補償不足也能過關，飄移大的補償不足就直接飛出界，這個 50% 是這兩顆 preset 混合出來的結果，不是每顆都剛好 50%）。

**切球時機容錯窗口擴大（使用者要求）：** 使用者指出切球的設計目標是「彈跳後下降期切，效果最好、離網最近」，但**整個下降期的合理範圍都應該要能進對方球桌**，不是只有極窄的一個瞬間。實測發現原本的切球參數（`{y:0.011,z:-1}`、`racketNormal.y:0.1`）只有約 10 個模擬幀（~0.08 秒）的成功窗口：太早切（球還在上升）會飛出桌尾，太晚切（球墜落太深）會掛網。

- **調整方向：** 拍面角度加大（`racketNormal.y` 從 0.1 加到 0.6，更平/更包）、力道減弱（`techniqueVel.z` 從 -1 減到 -0.2，前推力道大幅降低），讓球主要靠拍面的角度導引而不是額外的推力，這樣的擊法對接觸時機的敏感度低很多。
- **效果：** 用 2 顆無旋轉發球、從頂點到明顯下墜（vy 從 0 到 -1.3，每 2 幀取樣）測試，forehand 方向的成功窗口從原本的一小段擴大到**整個取樣範圍全部成功（19/19）**，backhand 方向覆蓋前半段（8/19，深墜段仍會出界，跟下方的「短球/深墜」問題同源）。兩種情況下，落點都會隨切球時機變晚而穩定地越來越靠近網——符合使用者要的「越晚切效果越好」。
- **意外的認知修正：** 一開始以為新參數讓切球對下旋不那麼敏感（下旋成功率從 0/4 變 2/4）是要取捨的缺點，使用者糾正了這個判斷——**切球本身就是對付下旋的專門技術，執行正確的話對下旋成功率本來就該高**，之前「旋轉應該讓回擊自然失敗」的設計原則是針對「平擊攻球」，不該套用到切球上。所以下旋成功率提高是往正確方向修正，不是妥協。
- **已知殘留問題（本輪已解決，見下）：** 2 顆短球下旋掛網（力道不夠）；部分側下旋長球在窗口後段出界（力道偏強）。

**切球力道依球速動態調整（使用者依實際打球經驗指出）：** 使用者指出「長球速度快，力道要偏被動（輕輕切），不然相對速度太多會變成碰撞為主而不是摩擦為主，準確度會變差；短球速度慢，需要比較主動的力道才能順順地送過網」。這正好對應到上面記錄的殘留問題——固定力道沒辦法同時適應短球（需要更主動）跟長球（需要更被動）。

- **設計：** 新增 `speedScaledZ` 設定（`TECHNIQUES.push`/`DEFAULT_TECHNIQUES.push`），讓 `techniqueVel.z` 依來球速度線性插值：`{low:-0.6, high:-0.2, speedLow:1.2, speedHigh:4.2}`。16 顆發球在切球最佳擊球點的速度剛好呈現明顯的兩群（短球 1.1~1.9 m/s，長球/半長球 3.4~4.2 m/s），符合線性插值的錨點選擇。
- **踩坑：** 第一版用「接觸當下的總速度」（含 `vy`）當插值依據，結果把「短球 vs 長球」跟「切球時機多晚（vy 隨下墜變負）」兩件事混在一起——用單一擊球點校準出的參數（`Z_HIGH≈0.024`，非常弱）套到寬時機窗口測試後，剛擴大的窗口又縮小回 12/38。改用「水平速度」（`Math.hypot(vx,vz)`）當依據解決：水平速度在同一條拋物線上全程不變（只有重力影響垂直方向），才能真正只反映「哪一顆發球」，不受「什麼時候切」影響。同時把 `Z_HIGH` 從單點最佳化找到的 0.024 調整回原本寬窗口驗證過的 -0.2（維持窗口穩健度），只加強 `Z_LOW`（短球端）到 -0.6 修正短球掛網。
- **最終結果（已套用到兩個檔案）：** 短球下旋 2 顆掛網問題解決（原本 2/4 → **4/4 全部成功**），寬時機窗口測試維持原本的 27/38（沒有因為加了速度調整而退步）。16 顆發球的整體切球表現：`no_spin` 2/2、`backspin` 4/4、`sidebackspin` 3/7。
- **殘留：** `sidebackspin`（側下旋）目前只有 3/7，其餘 4 顆側下旋長球在單點測試下仍出界——這批還沒進一步細調，可能需要側旋分量本身也影響力道校準（目前只依水平速度，沒有分開處理側旋強度）。

**機制釐清（使用者追問）：** 使用者問「長球該輕切」這件事跟 `dynamicPaddleEpsilon`（拍面反彈係數隨撞擊速度下降）是不是同一件事。實際算給使用者看：
- **CoR-速度效應確實存在但影響很小**：`techniqueVel.z` 從 -0.2 加到 -1.0，法向撞擊速度從 3.56 升到 4.24 m/s，`epsilon` 只從 0.877 降到 0.866（因為切球場景的實際速度落在 `PADDLE_SPEED_LOW=2~PADDLE_SPEED_HIGH=12` 這個區間的平緩段，離高速端還很遠），不是造成「長球該輕切」現象的主要原因。
- **切向「打滑 vs 純滾動」的相變也沒有發生**：直接檢查 `bounceTangentialAxis` 的 regime，在整個測試的 z 力道範圍內兩個切向軸全程都是 `rolling`（純滾動），從未進入 `sliding`（打滑）。所以使用者原本「力道太多會變成碰撞為主而非摩擦為主」的框架，在目前這套模型裡字面上不成立（沒有相變）。
- **真正的機制：拍面角度傾斜造成的幾何耦合。** `racketNormal.y=0.6`（拍面法向有向上的分量），球拍前推力道透過這個角度耦合，不只增加前進速度，也同比例增加了球的出手高度（`outVel.y` 隨 `|z|` 增加而線性上升）。長球本身已經帶有足夠的前進動能，疊加額外的高度/力道後容易直接飛出桌面尾端——這是「多餘力道被拍面角度轉成額外高度」的幾何效應，不是接觸型態的相變。使用者對「長球該輕切」這個經驗判斷是對的，只是模型裡的物理路徑跟原本設想的（摩擦轉碰撞）不同。

### Phase 6 附加功能：範圍解模式（呈現桌球動作的多樣性）

**使用者的構想：** 想呈現桌球的多樣性，「有點像是一個生態箱，而不是一個約定好的套招」——不是用明知不可行的解法，而是同一個技術動作每次執行會有自然的人手變異，但方向依然正確（人不是機器）。

**範圍先探索了發球解算，後來排除：** 一開始想讓 `solveServeBounceVelocity` 每次從「夠好」的候選解裡隨機挑一個，而不是固定挑分數最高的單一解。實測發現這個空間比預期窄很多——把 `no_spin_long_backhand` preset 的整個 `(time, yNudge)` 網格候選解按分數排序，只有 2 個落在合理範圍（53、86 分），其餘全部跳到 440 分以上，中間沒有平緩過渡帶；連對最佳解做 0.1% 量級的時間微擾都會讓分數暴衝（`firstGuardPenalty` 權重高達 9000，一旦第一跳誤差超過 2.5cm 就會被嚴重懲罰）。**這代表至少部分 preset 的發球解算是「刀鋒式」的，幾乎沒有容錯空間，硬做變異容易產生表面隨機、實際不合理的軌跡。** 使用者確認先只做回擊動作的變異，發球解算暫緩，留給以後視需要評估。

**回擊動作變異（已實作）：**
- 新增 `RANGE_SOLUTION_MODE` 開關（UI 勾選框「範圍解模式」），開啟後 `makeRacketReturnVelocity()` 對 `techniqueVel`/`racketNormal` 套用小幅隨機抖動（`applyExecutionVariance()`），模擬人手每次揮拍的自然變異；關閉時完全不影響現有已驗證的確定性行為（逐位元相同）。
- 抖動幅度用 `TECHNIQUES.*.executionVariance`（`{vel, normal}`），批次測試 13 顆有效發球校準：push 容錯空間大（寬時機窗口驗證過 27/38），用 `vel:0.12`；attack 容錯空間較窄，用 `vel:0.08`；`normal` 抖動固定抓 `vel*0.3`。
- **驗證結果**：關閉時同一輸入每次結果逐位元相同；開啟後同一輸入每次結果明顯不同（速度/角度自然浮動）。批次測試（13 顆發球 × 10 次試驗）：push 從確定性基準 90/130 降到 81/130，attack 維持 30/130 不變——都還在合理範圍，沒有大幅崩潰。
- 已套用到 `game4.html` 跟 `return-studio.html`，兩邊逐字對齊。

### Phase 6 附加功能：連續對打模式（構想 2，探索中，尚未接進遊戲）☐

**使用者的構想：** 發球後雙邊持續用切球/攻球互打，觀察目前的參數範圍在連續交手下會不會發散，技術選擇依「來球有沒有下旋」自動判斷（有下旋用切球、沒有用攻球）。

**架構設計（已驗證可行）：** 因為現有的 `findHitIndex`/`findPushHitIndex`/`makeReturnVelocity` 等函式都是假設「接球方永遠站在 z>0 側」寫的，重複使用來模擬雙邊互打時，每次換邊都要把整條路徑（`points`/`velocities`/`spins`/`bounces`）做鏡像：位置與速度的 `z` 分量變號，`topspin` 變號、`sidespin` 不變（旋轉是偽向量，鏡像反射下沿鏡面法向的分量不變、沿平面內的分量變號，跟位置/速度這種一般向量的變換規則相反；已用桌面版本驗證過的物理公式在鏡像後仍自洽）。這個鏡像機制本身測試沒問題，可以重用。

**卡住的地方：技術模型泛化到連續對打的球況時失效。** 用「攻球回擊發球」後，鏡像模擬「切球接攻球的反彈」，發現接球點/球速大幅超出目前切球校準的範圍（球速 4.4~4.7 m/s，超過原本發球場景最高的 4.2；擊球點離網達 2.3m，遠比正常發球的接發點深）。嘗試把 Phase 6 已有的「依球速插值力道/角度」機制往外推（延伸 `speedScaledZ` 的速度上限、把拍面角度 `racketNormal.y` 也一起納入插值），批次測試 2 個代表性樣本，**發現這兩個球速相近的樣本，需要的調整方向卻互斥**（一個需要更多力道才過網，另一個同樣的參數會出界）——代表只用「球速」一個變數插值已經不夠，且延伸探索了「球拍主動往後收」（`techniqueVel.z` 給正值）也沒能單獨解決問題。

**文獻研究結果——找到真正的根因：**
- 拍面角度應該依**來球旋轉量**調整，不是只依速度：查到的資料指出「拍面角度要依來球衝擊力道與旋轉調整，旋轉量越大，拍面角度要越封閉」；切球本身「來球下旋弱時拍面要更封閉，重下旋則要打開」。這解釋了為什麼那兩個球速相近、但側旋量不同（0.28 vs 23.6）的樣本會需要相反的調整方向——側旋量沒有被目前的插值公式考慮進去。
- **更關鍵的發現：切球（push）與削球/擋球（chop/block）根本是不同技術，適用不同情境，不是同一個技術調參數就能涵蓋的。** 查到的資料明確寫：「切球通常用在靠近球桌、來球已經帶下旋的情況；削球則用在離桌較遠、來球帶上旋的情況」，削球的拍面開合角度是 30-45 度、短促的 15-20cm 揮拍動作，跟切球的動作模式不同。連續對打第二棒接到的球（離網 2.3m、弱下旋、高速）根本不是「切球」該處理的情境，比較接近「削球/擋球」該出手的情境。**目前用單一「切球」技術硬套所有距離/球速的情況，這才是真正卡住的根本原因，不是參數不夠精細的問題。**

**結論與下一步建議：** 連續對打要做得對，至少需要：
1. 依「擊球點到網的距離」區分近台/遠台情境（類似發球已經有的 `tags.length` 長短球分級概念），可能需要新增一個「削球/擋球」技術（`model:'chop'` 或類似），拍面角度/力道模型跟現有的 `push` 分開設計，不要硬拉一個 `push` 去涵蓋所有距離。
2. 拍面角度的插值除了球速，應該也要把旋轉量（`spinSurfaceSpeed` 或直接用 `topspin`/`sidespin`）當作輸入維度一起插值，不能只用球速單一變數。
3. 這是一個需要重新設計的多變數技術模型，不是可以快速調完的參數校準，**建議另開一輪 session 專門處理**，不要在既有的 Phase 6 校準基礎上硬湊。
4. 連續對打本身的「雙邊互打 + 鏡像座標」架構已經驗證可行，可以直接沿用，不用重新設計這部分。

**尚未接進遊戲**：`RANGE_SOLUTION_MODE`、`RETURN_SKILL_LEVEL` 等既有機制都還沒有跟連續對打整合測試（因為連續對打本身的技術模型還沒解決），`game4.html`/`return-studio.html` 都還沒有實際的連續對打 UI/程式碼，這次的成果全部停留在瀏覽器 `preview_eval` 的原型驗證階段，沒有寫進檔案。

### Phase 6 附加研究：固定框架下「哪些該穩定、哪些該調整」的受控實驗

**使用者提出的研究問題：** 桌球練習因為反應時間有限，會用「固定框架」（穩定的動作範圍）當作實用策略。使用者想透過模擬回答：在這個框架裡，哪些東西該保持穩定，哪些該調整、怎麼調整，才能做出穩定的連續對打。這是把「連續對打模式卡住」這件事重新框架成一個更根本的研究問題，而不是繼續硬湊參數。

**實驗設計：** 合成一個可獨立控制三個因子的「來球狀態」（球速、擊球點到網距離、下旋角速度），對每組輸入用網格搜尋 + 爬山法找出「能成功回擊」所需的最小拍面角度（`racketNormal.y`／`tiltY`）與力道（`techniqueVel.z`），分別做三組單因子實驗（每次只變一個因子，其餘固定）：

- **實驗 1（只變球速，距離固定 2m、無旋轉）**：`tiltY` 隨球速增加而下降（0.4→0.2，2.5~5m/s 範圍），`z` 力道隨球速增加而變被動（-0.6→0）。
- **實驗 2（只變距離，球速固定 3.5m/s、無旋轉）**：`tiltY` 隨距離增加而明顯上升（0.02→0.3，0.5~3m 範圍），`z` 力道相對穩定（大多在 -0.6~-0.8 之間，沒有明顯趨勢）。
- **實驗 3（只變下旋量，球速固定 3.5m/s、距離固定 2m）**：`tiltY` 隨下旋量增加而上升（0.2→0.4，0~125 rad/s 範圍），`z` 力道同樣相對穩定（-0.6~-0.8 之間）。

**核心發現：拍面角度是三個因子共同的敏感變數，力道相對穩定。** 三組實驗中 `z` 力道的變化範圍都遠小於 `tiltY`，且大多落在 -0.6~-0.8 這個窄帶內；`tiltY` 則對速度、距離、旋轉三者都有清楚、平滑（大致單調）的反應。這跟稍早查到的文獻（拍面角度依來球衝擊力道與旋轉調整）吻合，且進一步指出**距離**也是拍面角度的主要驅動因子，文獻只提到了速度跟旋轉。

**簡化控制律（27 組多因子網格資料的線性回歸擬合，26 組有效、RMSE=0.066）：**
```
拍面角度(tiltY) ≈ 0.18 − 0.073 × 球速(m/s) + 0.146 × 擊球點到網距離(m) + 0.0015 × |下旋角速度(rad/s)|
```
- 距離的影響係數最大（0.146），球速次之（-0.073，負向），下旋量係數換算成常見單位約為「每多 1 rps 下旋，角度加大約 0.01」。
- 這代表：**距離感（判斷來球深淺/自己該站多遠）是拍面角度調整裡最關鍵的線索**，比精準抓球速或旋轉量更重要——這對訓練設計有直接意義：要練「用最少調整量做出穩定連續對打」，優先練距離判斷，力道可以當作框架裡固定不動的部分。

**這個發現如何餵回連續對打模式（Phase 6 附加功能：連續對打模式）：** 印證了那邊的結論——連續對打卡住不是「參數不夠精細」，而是缺一個真正吃「距離」當輸入的拍面角度模型。這裡的線性公式可以當作下一輪重新設計技術模型時的起始點（不是最終解，只用 3 個變數、27 個網格點擬合，還沒有大範圍驗證，尤其還沒驗證側旋、也還沒排除 `z` 力道被我簡化成「固定」是否在更極端的情況下仍然成立）。

**尚未做的：** 這次只驗證了「單因子」跟「27 點網格」兩種實驗設計，還沒有：
- 用這條擬合公式重新跑一次連續對打的兩個失敗樣本，確認它是否真的能解決 Phase 6 附加功能（連續對打模式）那邊卡住的案例。
- 驗證側旋（`sidespin`）對 `tiltY`/`z` 的影響（這次三組實驗都固定 `sidespin=0`）。
- 把這條公式或類似機制寫成程式碼，接進 `game4.html`/`return-studio.html`。

**追加發現：旋轉對「拍面角度容錯窗口寬窄」的影響，比對「角度數值本身」的影響更關鍵。** 使用者提出一個現場教練教過的觀念反問：如果對方發球下旋很弱，教練會教「拍面立起來一點、由上往下切」，這跟切球的一般做法不太一樣，這是不是一個針對弱旋轉球、介於切球跟攻球之間的應變動作？

- 先驗證了「旋轉→角度」這條曲線本身有沒有轉折：在零旋轉附近加密取樣（0, -2, -5, -10, -15, -20, -30...-125 rad/s），`tiltY` 是平滑遞增（0.2 一路升到 0.4），**沒有任何跳變或轉折**——代表在我們的物理模型裡，「切球」這個動作本身沒有在某個旋轉量遇到相變或失效邊界，跟強下旋用同一個 `bounceOffPlane` 公式就能連續地涵蓋到零旋轉。
- 但改測「角度的容錯窗口寬度」（固定球速 3.5m/s、距離 2m，掃描 `tiltY` 前後範圍找出多少角度誤差內仍然成功），發現關鍵差異：
  - **零旋轉球**：容錯窗口只有 **0.13**（`tiltY` 落在 0.17~0.30 之間才成功）
  - **重下旋球（-125 rad/s）**：容錯窗口有 **0.29**（`tiltY` 落在 0.40~0.69 之間都成功），寬度是零旋轉球的 **2.2 倍**
- **解讀：** 教練教的「換一套更立、由上往下切」的動作，很可能不是因為切球在物理上對零旋轉球做不到，而是因為**用切球去接零旋轉球時，角度容錯窗口窄很多，實戰中很難每次抓準，選一個對零旋轉球更寬容的替代動作，是規避這個窄容錯風險的合理選擇**。這跟本專案一路以來的設計哲學一致（容錯範圍寬才是好動作），只是這裡的應用是「選擇哪個技術」而不是「調整同一技術的參數」。
- **這對「哪些該固定、哪些該調整」這個核心問題補了重要的一塊：** 上面的線性公式只回答了「角度數值該怎麼調」，但**角度容錯窗口本身的寬窄，才是決定「這個技術在這個球況下能不能穩定執行」的真正關鍵**——窄窗口的球況，就算算出了正確的角度數值，人手實際執行時還是容易出錯，這時候該做的不是更精準地瞄準那個窄窗口，而是換一個容錯窗口更寬的技術/動作。
- **尚未做的**：只驗證了零旋轉 vs 重下旋這兩個端點的窗口寬度，還沒有：
  - 掃出「容錯窗口寬度 vs 旋轉量」的完整曲線，確認窗口寬度是不是也對旋轉量單調遞增（還是零旋轉附近有特別窄的區域）。
  - 驗證同樣的「窗口寬窄」邏輯，套用在球速、距離這兩個因子上是否也成立（也就是說，會不會存在某個球速/距離組合，角度數值算得出來但窗口極窄，一樣需要換技術）。
  - 沒有實際測試過「立起來、由上往下切」這個教練描述的動作在我們的模型裡對應什麼樣的 `racketNormal`/`techniqueVel` 參數，也還沒驗證這個動作對零旋轉球的容錯窗口是否真的比切球寬。

**這個研究做完了：拍面角度＋力道聯合控制律已正式取代固定角度，成為切球的正式參數（已套用到兩個檔案）。**

使用者要求把這個方向做完整，而不是半途而廢。過程：

1. **第一次嘗試失敗**：直接把先前的合成實驗公式（27 組合成網格資料）套進真實引擎，backspin 從 4/4 掉到 0/4、sidebackspin 從 3/7 掉到 1/7——比原本固定角度的版本還差。查出原因：合成樣本假設的來球下墜比例（`vy = -0.15×球速`）跟真實引擎在 `findPushHitIndex` 那個時間點的實際彈道形狀不一致，公式的方向對但數值不能直接套用。
2. **第二次嘗試也失敗**：改用真實的 13 顆有效發球在 `findPushHitIndex` 單點擬合（8 個成功樣本），擬合誤差看起來很小（RMSE=0.013），但套進真實引擎時只調整 `tiltY`、`z` 仍沿用舊的 `speedScaledZ` 公式，結果更差（no_spin 甚至從 2/2 掉到 0/2）。抓出真正的原因：**`tiltY` 跟 `z` 是聯合搜尋出來的一組解，兩者耦合，不能只取角度、力道另外用別的公式**，且 8 個樣本對「兩個輸出、三個輸入」的模型來說資料量太少。
3. **第三次成功**：把取樣範圍從「16 顆發球的單一 `findPushHitIndex` 點」擴大到「沿彈跳後整個下降窗口，每 4 幀取一次樣」，得到 196 個真實樣本、其中 88 個能找到成功解，同時對 `tiltY` 跟 `z` 兩個輸出做線性回歸（RMSE 分別是 0.081、0.155，樣本數足夠支撐 2 個輸出 × 4 個係數的模型）。

**最終公式（已套用到 `TECHNIQUES.push`/`DEFAULT_TECHNIQUES.push`，取代舊的 `speedScaledZ`）：**
```
tiltY ≈ 0.198 − 0.1106×球速(m/s) + 0.1235×擊球點到網距離(m) + 0.005×|下旋角速度(rad/s)|（clamp 0.02~0.7）
z     ≈ −0.6737 + 0.2976×球速(m/s) − 0.2233×擊球點到網距離(m) − 0.0035×|下旋角速度(rad/s)|（clamp -1.0~0.3）
```

**驗證結果：**
- 16 顆發球單點驗證（原本固定角度校準過的驗收標準）：完全維持 `backspin 4/4`、`no_spin 2/2`、`sidebackspin 3/7`，沒有退步。
- **時機窗口整體成功率大幅提升**：196 個沿下降窗口取樣的真實樣本，固定角度只成功 12 個（6%），新公式成功 80 個（41%），將近 7 倍。這才是原本 Phase 6 想要的「彈跳後整個下降期都該成功」的效果，比先前手動調出的固定角度更好。
- **連續對打卡住的案例仍未解決**：套用新公式測試之前 Phase 6 附加功能記錄的 2 個連續對打失敗樣本，還是失敗——這兩個樣本的球速（4.4~4.7 m/s）超出訓練資料涵蓋的範圍（最高約 4.2 m/s），是外插，不可靠。連續對打模式要解決，需要專門收集更高球速範圍的真實樣本重新擬合，這次的公式只解決了「發球+單次回擊」這個主要遊戲情境，沒有意外解決連續對打。

**實作細節：** `game4.html`/`return-studio.html` 的 `makeRacketReturnVelocity()` 新增 `contactZ` 參數（呼叫端從 `contactPoint.z`/`hitPoint.z` 傳入），`TECHNIQUES.push`/`DEFAULT_TECHNIQUES.push` 新增 `adaptivePush:true` 旗標觸發 `computeAdaptivePushTiltZ()`；舊的 `speedScaledZ` 機制整個移除（被新公式取代）。`racketNormal`/`techniqueVel` 的固定值只在拿不到 `contactZ` 時當備援。

### Phase 6 再突破：拍面角度與揮拍方向解耦（使用者從球的軌跡看出盲點）

**使用者的觀察：** 看了球的實際飛行軌跡後指出兩個問題：(1) 拍面朝向（`racketNormal`）跟球拍移動方向（揮拍方向）是**兩個獨立的向量**，我們先前的公式把兩者鎖在一起、用同一組耦合變數去湊「有沒有落桌」，順序錯了；(2) 正確的優先順序應該是**先固定一組揮拍方向，讓它產生一條低平、剛好過網的弧線，再單純用力道大小去控制落點深度**，而不是同時調角度跟力道去湊落點。

**驗證：固定方向、只變力道，球的仰角真的幾乎不變。** 直接測了：固定拍面角度跟揮拍方向後，力道從 0.2 掃到 1.5（7.5 倍），出手仰角只從 14.2° 漂移到 15.2°（不到 1°）。這證實了使用者的直覺完全正確——揮拍方向決定弧線的「形狀」，力道大小才是控制距離的維度。（過網高度會隨力道變高，是拋物線的自然特性：同一個角度、球飛更快，過網當下花的時間更短，重力還來不及往下拉，不是模型的問題。）

**用這個新架構重做校準：**
1. 先把拍面角度跟揮拍方向鎖成同一個向量測試，容錯率 32%（比原本聯合擬合公式的 41% 差）——但使用者指出這樣做等於還是沒有真正解耦，只是換了一種方式把兩者綁在一起。
2. 把拍面角度**獨立**出來（不等於揮拍方向）重測，找到 `tiltY=0.4~0.6` 這個平原區間可以達到 **46%**（用精確逐點搜尋力道，比原本的 41% 更好）。
3. 用固定方向（`swingDirection = normalize({0, 0.3, -1})`）+ 固定角度（`racketNormal = normalize({0, 0.5, -1})`），對 196 個真實樣本（16 顆發球沿彈跳後整個下降窗口取樣）做 1 維力道搜尋，找出每個樣本「剛好成功」的最小力道，只對力道這一個輸出做線性回歸（比先前雙輸出模型更簡單、更穩健）。

**最終力道公式（已套用，取代 Phase 6 的角度+力道聯合公式）：**
```
magnitude ≈ 0.0942 − 0.4069×球速(m/s) + 0.4183×擊球點到網距離(m) + 0.0096×|下旋角速度(rad/s)|（clamp 0.02~2.5）
```
拍面角度固定為 `racketNormal = normalize({x:0, y:0.5, z:-1})`，揮拍方向固定為 `swingDirection = normalize({x:0, y:0.3, z:-1})`，兩者都不隨球況調整。

**驗證結果：**
- 16 顆發球單點驗證：完全維持 `backspin 4/4`、`no_spin 2/2`、`sidebackspin 3/7`，沒有退步。
- **時機窗口容錯率**：精確逐點搜尋力道可達 46%（比 Phase 6 聯合公式的 41% 更好，證實新架構的理論上限更高）；目前這條線性擬合的力道公式實測是 **37%**，比 41% 略低——公式本身還沒完全逼近 46% 的上限，可能需要非線性擬合或更多樣本才能榨出來。這是一個誠實的取捨：新架構「對」但目前的力道公式擬合品質還沒追上舊模型的實測分數；使用者決定先套用這個更正確的架構，之後看軌跡再回頭指出問題所在。
- 已套用到 `game4.html`/`return-studio.html` 兩個檔案。

### 修正「回擊起點被憑空抬高 0.12 公尺」的問題，重新校準切球力道公式

**使用者發現的問題：** 觀察球的軌跡時發現回擊線的起點沒有跟發球線的終點接在一起、中間有一段空隙，追問之後發現根因是 `game4.html`/`return-studio.html` 裡 `returnBall()`/`simulateReturnForPreset()` 都有這一行：
```js

```
查了 git 歷史，這行從專案最初的 rally loop 版本就存在，沒有任何說明，看起來只是一個很粗略的「安全邊界」寫法，並不是刻意設計的物理效果。使用者追問「這是不是表示所有回擊點都被抬高了？這會是很嚴重的問題」——實測後證實**是真的**：同一組 incomingVel，起點抬高 0.12m 會讓落點 z 從 0.29 變成 0.88（將近 3 倍），因為抬高的高度會讓球多花時間下落，這段時間球會繼續往前飛。而且這個誤差量還不是固定的，會隨擊球當下的垂直速度 vy 而變（vy 越小，多摔的時間佔比越大），代表這是一個會隨球況跳動、非均勻的系統性偏差，混進了 Phase 6 所有校準數字裡。

```
magnitude ≈ 1.1469 − 0.4989×球速(m/s) + 0.2941×擊球點到網距離(m) + 0.0055×|下旋角速度(rad/s)|（clamp 0.02~2.5）
```

RMSE≈0.104（跟修正前的舊公式 RMSE≈0.106 同一個量級）。已套用到 `game4.html`/`return-studio.html`。

### 拍面平面角度改為相對揮拍方向（使用者觀察到的架構問題）

**使用者的觀察：** 拍面法向量（racketNormal）的垂直角度（開合幅度）是世界座標的絕對概念（因為「上下」有桌面/重力當固定參考），但平面角度（左右偏轉）其實應該是**相對於揮拍方向**的概念——現實中「開/合拍面」講的是相對於你揮拍路徑的偏轉，不是相對於球桌座標系的絕對偏轉。但舊實作把兩者都當成世界座標的絕對分量（`racketNormal:normalize({x, y, z:-1})`，x 是絕對世界左右偏移），這是錯的。

**連帶發現的 bug：** 使用者實測回報「不管怎麼調參數都無法把球打回來球方向」。查出根因是 `makeRacketReturnVelocity()` 裡的 `solveRacketVelXForTargetOutX(..., 0)`——最後一個參數（目標出球 X 速度）寫死是 0，只要「停用側向自動瞄準修正」關閉（正式遊戲的預設行為），系統就會完全無視 `techniqueVel.x`/`swingDirection.x`/`racketNormal.x`，直接反算一個讓出球方向強制打直的球拍 X 速度。實測驗證：修正修正開啟時，不管 `techniqueVel.x` 設 0/0.4/0.8/-0.4，出球 X 速度永遠鎖在 ~0、落點 X 完全不變；關掉之後才會隨參數變化。**這不是這次的新 bug，是原本就存在的行為**，攝影棚的「停用側向自動瞄準修正」開關就是設計來測試方向用的。

**修法（拍面角度重新參數化）：** `racketNormal` 不再是技術設定裡的靜態向量，改成兩個純量 + 動態計算：
- `racketNormalTiltY`：垂直開合角度，世界座標絕對值，跟以前語意相同。
- `racketNormalTiltX`：平面偏轉角度，改成**繞揮拍方向這個軸旋轉**的相對角度（Rodrigues 旋轉公式）。
- 新函式 `computeRacketNormal(tiltY, tiltX, swingDirRef)`：先算出 `normalize({x:0, y:tiltY, z:-1})`（絕對垂直開合），再繞 `normalize(swingDirRef)` 這個軸轉 `tiltX` 弳度。`swingDirRef` 在切球自適應力道公式開啟時用 `tech.swingDirection`，其他情況（攻球、切球手動模式）用實際要打出去的球拍速度方向本身。
- `tiltX = 0` 時直接回傳垂直開合結果、跳過旋轉，確保跟舊行為完全一致（不需要重新校準）——已驗證 16 顆發球單點測試維持 backspin 4/7、no_spin 2/2、sidebackspin 2/7，跟修正前一模一樣。
- `game4.html`/`return-studio.html` 的 `TECHNIQUES`/`DEFAULT_TECHNIQUES` 都已改成 `racketNormalTiltY`/`racketNormalTiltX` 兩個欄位；攝影棚的 UI 讀寫、匯出程式碼片段也都同步改掉。

**驗證：** 用真實 `input`/`change` DOM 事件（不是直接呼叫函式）測試：固定 `swingDirection.x=0.3`、開啟「停用側向自動瞄準修正」，`racketNormalTiltX` 從 -0.2 掃到 0.4，落點 X 從 -0.24 一路變到 -0.10，確認角度確實會影響方向，而且是相對揮拍方向轉動（不同的 `swingDirection` 對同一個 `tiltX` 會產生不同的絕對法向量，用直接呼叫 `computeRacketNormal()` 驗證過）。手機版面確認無元素溢出。

### Codex 交接後的一輪災難排查與收拾

**背景：** 這一輪交接給 Codex（在另一個本機資料夾 `2026-07-01/tt-reaction-trainer-game-4-github`）處理發球 preset 校準，結果使用者回報「發生大災難」。排查後發現是好幾個獨立問題疊在一起，逐一釐清、修復如下。

**1. `physics-studio.html` 的旋轉數值換算比例錯誤，導致每顆球都無法發球成功。**
`physics-studio.html` 是專案早期就有的「發球 preset 製作工具」，內部用的 `applyBounceSpin()`／`BOUNCE_PHYSICS.topspinKick` 是在專案還沒把旋轉改成真實 `rad/s` 儲存**之前**、針對舊版「隨手調」的旋轉數字（範圍約 -1.23~1）校準的——不是 rps，也不是 rad/s（用 git 歷史比對過：`781dda8^` 版本 `backspin_long_backhand` 的 topspin 是 `-0.8`，現在真實版本是 `-125.66`）。Codex 把 rad/s 換算成 rps（除以 2π≈20），量級還是差了約 20 倍，導致 `forwardSpeed` 整個變號、球彈跳後直接往反方向飛。第一輪先改成除以 `LEGACY_BOUNCE_SPIN_SCALE=125.66`（對回舊版校準基準量級）暫時解決，讓球至少能正常發出去；根本解法見下面第 4 點（整個換成真實引擎，不用再猜換算比例）。

**2. `physics-studio.html` 開頁畫面跟任何 preset 都對不上。**
開頁時 HTML 裡寫死的欄位預設值（起點/第一跳/第二跳座標）跟 `presetName` 輸入框的預設文字，是兩次不同編輯留下的殘影，彼此對不上、也不對應任何真實 preset。修法：`fetchRepoPresets()` 讀到 repo 資料後，如果目前沒有選定的 preset，自動載入第一顆。

**3. GitHub Pages 部署卡住。**
Push 之後 GitHub Pages 一直沒更新，`gh run list` 查到 10 個 `pages build and deployment` workflow 卡在 `queued` 好幾天（有的甚至卡了兩天），`gh run cancel` 對這些卡住的 run 回報「已完成」自相矛盾（GitHub Pages 這個自動 workflow 的顯示狀態本身有問題）。用 `gh api -X POST repos/OWNER/REPO/pages/builds` 手動觸發新建置後，第一次還遇到後端回傳 `Deployment failed, try again later.`（GitHub 自己的暫時性問題），重試一次後才成功。**這是這次修復流程一路上會反覆用到的手法**：每次 push 完都手動戳一次 `pages/builds` API、輪詢 `pages/builds/latest` 的 `status` 直到變成 `built`，不要只看 push 完就假設會自動部署好。

**4. 4 顆發球在真實 v2 引擎下永遠過不了網（不是這次任何人造成的，是更早就存在的問題）。**
`findHitIndex()` 找不到合法擊球點時會退回一個備援heuristic（用球的位置/高度猜一個接觸點），這個備援長期把「發球本身根本沒有合法過網」這件事蓋住了，導致這幾輪的回擊驗證看起來正常，但其實從沒真正檢查過發球本身是否完整。實測 `simulateServe()` 找出 4 顆球（`backspin_long_backhand`、`backspin_short_forehand_2`、`sidebackspin_long_forehand`、`sidebackspin_short_forehand`）在自己這側連續碎跳（最多到 9 次），從未真正越過網。

根本原因：這 4 顆球的拋球高度（`start.y`）太低，配上強旋轉，真實接觸力學下每次觸桌垂直反彈的能量遞減太快（每次約剩 75%），球沒有足夠滯空時間飛越剩下的距離。用真實引擎（`simulatePath`＋`solveServeBounceVelocity`）搜尋找出可行解：提高這 4 顆的拋球高度，其中 2 顆再微調第一跳 Z 座標：

| Preset | 拋球高度 | 第一跳 Z | 落點誤差 |
|---|---|---|---|
| backspin_long_backhand | 0.87→1.33 | 不變 | 0.109m |
| backspin_short_forehand_2 | 0.95→1.55 | 不變 | 0.003m |
| sidebackspin_long_forehand | 0.88→0.95 | -1.03→-0.89 | 0.137m |
| sidebackspin_short_forehand | 0.95→1.03 | -0.6→-0.45 | 0.162m |

### 用使用者手動標的切球資料，回歸出新的 tiltX 公式

使用者用 `return-studio.html` 的「球例校正資料庫」逐顆標出 10 顆下旋/側下發球（其中 7 顆是關掉自適應力道公式、完全手動調的）他覺得合理的切球參數，透過「儲存全部校正到待處理檔案」存成 `pending-return-technique-changes.json` 交給我統計。

**回歸發現：**
- **`racketNormalTiltX`（拍面平面角度）**：跟來球的左右速度分量（`incomingVel.x`）有關，側旋幾乎沒有影響（回歸係數只有 0.0004）。`tiltX ≈ -0.144 − 0.538×incomingVel.x`。**這是全新加入的能力**——之前這個值在 `adaptivePush` 模式下一直固定是 0，從沒隨球況調整過。
- **力道公式**：用同一批資料重新擬合出新係數，跟原本公式同一個形狀（`magnitude ≈ a − b×speed + c×z`），實測 16 顆發球驗證結果沒有變化。
- **`racketNormalTiltY`（拍面垂直角度）**：也試著回歸過（`tiltY ≈ 0.453 − 0.217×speed + 0.348×z`），但套用後實測 16 顆發球驗證會讓 backspin/sidebackspin 雙雙倒退（6/7,2/7 → 4/7,0/7）。逐一拆解三個公式（力道／tiltY／tiltX 各自單獨套用測試）確認退步完全來自 tiltY 這個公式，另外兩個單獨套用都完全不影響、跟原本一樣好。研判樣本量不足（只有 10 顆、其中只有 7 顆是真正自由調的）不足以擬合出穩定的 tiltY 關係，所以維持原本固定值 0.5，只部署驗證過沒有退步的 tiltX 公式跟新力道公式。

**已套用：** `game4.html`／`return-studio.html` 的 `makeRacketReturnVelocity()` 都改成 `adaptivePush` 時動態算 tiltX（非 adaptivePush 情況維持讀 `tech.racketNormalTiltX`），tiltY 維持固定不變。驗證：16 顆發球回擊技術驗證維持原本水準（backspin 6/7、no_spin 2/2、sidebackspin 2/7），兩個檔案結果一致。

**已知限制：** 樣本量小（10 顆，其中 7 顆自由調），tiltX 公式只看了 `incomingVel.x` 一個變數，沒有機會驗證側旋單獨的影響（因為這批資料側旋幾乎都是同一個值 -125.66）。之後樣本變多、涵蓋更多樣的側旋強度後，可以重新擬合 tiltY，也可以驗證 tiltX 公式是否需要加入側旋項。

### 回擊瞄準機制升級：從「打直」改成「真正瞄準落地點」

使用者點出調切球時遵守的一個原則：**揮拍方向（球拍移動方向）要對準想打的落點，不是打直而已**。拿「球例校正資料庫」的資料驗證：手動調的 `techniqueVel.x` 跟擊球點 x 座標相關係數 **-0.82**，而且這批資料的實際落點 x 幾乎都收斂在 0 附近——證實使用者的原則其實是「瞄準桌面中央」，而且是有意識地依擊球點位置反向修正揮拍方向去達成，不是隨便亂調。

**根本問題：** 原本的 `solveRacketVelXForTargetOutX()` 只反推「出球 X 速度=0」，這不等於「球會落在固定位置」——因為不同球的飛行時間、拋物線形狀不同，同樣的出球速度會落在不同的 x 座標，這只是一個近似的「打直」而已，不是真正的瞄準。

**修法：** 新增 `solveRacketVelXForTargetLandingX()`，改成用估算的飛行時間（`estimateFlightTimeToTable()`，解拋物線方程式）直接反推「落地點 x 座標＝目標值」，預設 `RETURN_TARGET_X = 0`（瞄準桌面中央）。`makeRacketReturnVelocity()`/`makeReturnVelocity()` 的參數從 `contactZ` 改成完整的 `hitPoint` + `gravity`（飛行時間估算需要）。

**驗證結果（意外地不只是中性，而是真的變好）：** 16 顆發球回擊技術驗證，**sidebackspin 從 2/7 進步到 5/7**（backspin 6/7、no_spin 2/2 維持不變），總分從 10/16 提升到 13/16。落地點 x 座標全部收斂在 0 附近（誤差 <0.02m），確認真的瞄準到中央，不再只是「打直」的近似值。已套用到 `game4.html`／`return-studio.html`，兩邊結果一致。

**這件事的啟示：** 「打直」（出球速度=0）跟「瞄準中央」（落地點=0）聽起來很像，實際上是兩個不同的目標函式，後者才是物理上正確、跟真實揮拍原則一致的做法。這也解釋了為什麼側旋球的容錯率一直偏低——側旋球的飛行時間/軌跡形狀跟一般球差異較大，用「打直」去近似「瞄準中央」的誤差在側旋球上被放大了。

**下一步（可能的擴充方向，尚未開始）：** `RETURN_TARGET_X` 目前是寫死的常數（瞄準中央），之後可以依技術等級/站位擴充成可變目標（例如熟練等級刻意打對手最遠端），這是之前提過但被延後的「熟練等級側旋完全修正要能打到 X≥0.75」需求的技術基礎。

### 連續對打測試 → 發現力道公式正回饋失控 → 改成負回饋控制律（已套用）

用「鏡射座標系」技巧（每次回擊後把整條軌跡沿 z=0 鏡射，重複使用「接球者永遠在正 z 側」的既有函式）建了連續對打模擬器，發現舊力道公式（`+0.2941×擊球點離網距離`）在連續對打時會正回饋失控：擊球點越遠→力道加越多→球彈更深→下一拍擊球點更遠，實測大約 10 拍就飛出界，16 顆發球都逃不掉，因為這條公式是拿「單次發球回擊」資料校準的，從未在「擊球點會持續變化」的情境下驗證過。

使用者提出真實練習原則：對方打快（出力）就放鬆減力，對方打慢（放鬆）就有機會加力攻擊，否則同樣力道帶回去——純粹反應「上一拍來球速度」的負回饋控制律，跟深度完全無關。重新設計：
```
magnitude = 0.7 − 0.3 × (來球水平速度 − 2.0)，clamp 0.02~2.0
```
**驗證：** 16 顆發球有 13 顆可以連續對打 200+ 拍（幾乎無限），單拍準確率同時變好（backspin 7/7、sidebackspin 6/7、no_spin 2/2，比舊公式全面更好）。曾經試過額外疊加「擊球點離網距離」的修正項（模擬「深度感知」），結果反而讓全部 16 顆都變不穩定——兩個負回饋疊加會互相打架、造成過度修正震盪，所以先只用球速這一個變數，深度感知留待之後用不同時間尺度（即時反應球速 vs. 觀察多拍趨勢才修正深度）重新設計。已套用到 `game4.html`/`return-studio.html`。

**新增「連續對打展示模式」**（`return-studio.html`）：按「開始連續對打」用目前選的發球當開球，雙方輪流用切球回擊，最多展示 60 拍，用綠色線畫出完整真實軌跡並連續動畫播放。

**弧線過高問題（已套用）：** 使用者看展示後反映「弧線太高，希望能低低地互切」。固定 `racketNormalTiltY=0.5` 對所有球一視同仁太保守——早於此的「拍面角度容錯窗口寬窄」實驗已證實強下旋需要立起來的拍面（窗口窄）、弱旋轉可以用比較平的拍面（窗口寬）。改成依接觸當下**實際殘留**的旋轉強度（不是發球時的原始旋轉——真實碰撞後旋轉會衰減/反轉，backspin 落桌後在接觸點量測到的 `|topspin|` 大約只剩 45~83 rad/s，no_spin 甚至會反轉成正的 42~45）內插：`tiltY = 0.006×|topspin|`。clamp 下限試過 0.15（單拍準確率不變，但連續對打時旋轉逐拍衰減、tiltY 一路探底變太平，穩定顆數從 13/16 掉到 0/16）；改成 clamp 0.4~0.55 後單拍準確率跟固定 0.5 完全一樣，連續對打穩定顆數維持 13/16，平均弧線高度從桌面上方 ~1.0~1.1m 降到 ~0.81m（約 25% 下降）。

**連續對打的「節奏」觀察（尚未解決）：** 使用者提到真實穩定對練「打起來會很有節奏感，不會忽快忽慢」。實測發現：球速確實收斂到完全固定的 2.0 m/s（沒有忽快忽慢），但**落點深度是「短、深、短、深」規律交替**（例如穩定後在 0.26 / 1.25 兩個深度之間交替），不是同一個深度重複——推測是發球端跟接發球端兩個人的接觸幾何本來就不對稱，造成兩種站位各自收斂到不同深度，而不是同一個。使用者也發現連續對打中球的旋轉會衰減、甚至反轉成上旋（見下方「切球沒有真的製造下旋」）。

### 切球沒有真的製造反向下旋——發現摩擦力/球拍動力學建模的根本落差（規劃中，尚未開始執行）

**觸發問題：** 使用者指出真正的切球對練，每一板除了抵消對方旋轉，還要主動製造反向下旋還給對方（相對於對方而言）；如果實測發現球逐漸變成上旋，代表根本沒有完全抵消對方旋轉，不是穩定對打的正常現象。實測連續對打的旋轉數字證實：拍面接觸完當下，旋轉就已經從 -70.5 rad/s（強下旋）幾乎抹平到 +3.5（接近中性、甚至反向），不是「抵消再反向」。

**第一層診斷：揮拍方向的正負號錯了。** 現在 `swingDirection.y = +0.3`（往上帶），這比較像「往上挑」的攻擊動作，不是真正切球「往下刷過球」的動作。測試把 y 改成負值，拍面接觸後的旋轉確實會跟著變負（往下刷越多，下旋越強：y=-0.5 時可以做到 -22.3 rad/s 的反向下旋）。而且回頭檢查使用者自己在「球例校正資料庫」手動調過的 7 顆球，`techniqueVel.y` **全部都是負值**——這個訊號當時就已經在資料裡了，是這一輪重新分析時才發現我當時只拿那批資料去回歸 `tiltX` 跟力道，沒有去看 `techniqueVel` 的 y/z 比例，沒把「揮拍方向該往下」這個訊號套進 `adaptivePush` 模式固定用的 `swingDirection`。用這 7 顆的 y/z 比例平均出來的真實方向約 `y/z ≈ -0.24`。

**但光換方向會讓其他公式全垮。** 因為 `swingDirection` 是整套切球技術的地基，換了它，原本配合 +0.3 校準出來的力道公式跟拍面角度公式全部一起崩掉——單拍準確率從 backspin 7/7 掉到 1/7、連續對打穩定顆數從 13/16 崩到 1/16。用同一套「196 樣本視窗取樣＋逐點搜尋磁道＋線性回歸」方法論，固定新方向後找出最佳拍面角度（`tiltY=0.6`，容錯率上限 48%，比舊架構的 46% 更好），重新擬合力道公式（`magnitude ≈ 1.1497 − 0.5986×speed + 0.3648×dist + 0.0099×|topspin|`），但實測還是輸給舊版（單拍 10/16、連續對打穩定顆數 4/16，旋轉在連續對打中還是雜亂震盪在 -20~+28 之間，沒有乾淨地維持下旋）——判斷是這次線性回歸擬合的品質不夠好，不是方向錯了，需要更多迭代（可能要非線性擬合、或先固定角度用更精細的方式找最佳值）才能做好，這輪先沒有繼續往下調。

**第二層診斷（更根本）：摩擦力係數不是問題，我們的碰撞模型本身有一個由球拍速度決定的「滾動天花板」，而且這個天花板現在被壓得太低。**
1. 實測：把 `PADDLE_FRICTION` 從 0.4 加到 2.5，輸出旋轉完全不變（都是 +14.0）。原因：接觸在摩擦係數 0.4 時就已經達到「滾動」（`regime='rolling'`，零滑動）這個數學平衡點——這個平衡點只由球的質量/轉動慣量/半徑，加上**球拍當下速度**決定，跟摩擦係數完全無關（只要摩擦力夠讓球達到這個點，多少都一樣）。所以摩擦係數不是問題，是還沒摸到摩擦力真正的限制、先被別的東西卡住了。
2. **真正卡住的是球拍速度本身：目前切球的球拍速度模擬出來只有 0.198 m/s**，真實世界一個輕的切球估計至少 1~3 m/s，文獻實測正手拉球的球拍接觸瞬間速度高達 **20 m/s**——我們慢了一個數量級以上，「滾動天花板」自然壓得很低。
3. **就算球拍速度校準到位，真實技術還是能超越這個「單一瞬間碰撞」的天花板**，靠的是我們完全沒建模的兩個機制：
   - **真實接觸有 5~8 毫秒（不是瞬間），手臂在這段時間裡還在加速**——球拍放開球的那一刻比剛接觸時更快，我們的模型假設整個碰撞期間球拍速度固定不變。
   - **海綿本身的彈性儲能/釋放（catapult 效應）**，會額外注入能量，不是純剛體碰撞。
4. **使用者補充的關鍵原則：真實揮拍的動能不是只靠手臂，是靠全身（腿、腰、軀幹轉動）把動能一路帶上手臂，只要接觸瞬間「對上那個點」（時機抓對），能傳遞的總動能非常可觀**——這是真實桌球技術裡「動力鏈」（kinetic chain / summation of speed）的原理，也是文獻裡拉球球拍速度可以達到 20 m/s、旋轉可以達到 140 rps 這麼誇張數字的原因：不是手臂本身多快，是全身動能在對的時機集中釋放到球拍上。我們現在的模型只有「球拍速度」這一個純量／方向，完全沒有代表「動作時機品質」的維度。

**文獻依據（已查證，來源品質見下方）：**
- 真實球拍-膠皮接觸時間約 5 毫秒（高速攝影實測，可信度高）
- 一篇建模研究（Open Dynamics Engine）估出球桌碰撞摩擦係數僅 0.05，但球拍碰撞摩擦係數高達 1.0（可信度中高，比我們現在的 `PADDLE_FRICTION=0.4` 高出許多，但如上述已證實摩擦係數在我們的模型裡不是瓶頸，這個數字比較適合當作「膠皮真的很滑/很磨」的參考量級，不代表調高它就能解決問題）
- PMC 同儕審查論文實測正手拉球：球拍接觸瞬間速度 20 m/s，球飛出速度可達 45 m/s，旋轉可達 140 rps（≈880 rad/s）——可信度高，是拉球最終應該逼近的量級基準
- 商業/愛好者網站（topspin11.com、pingpongstar.com）提供的膠皮軟硬 → dwell time（2~8ms）、摩擦係數（0.8~2.5，依膠皮新舊/類型）等數字，量級可信但精確度未經同儕審查，需要之後交叉驗證

### 摩擦係數系統掃描：孤立測試有希望，但套進完整流程會整條崩掉（重要教訓，併入 Stage 2）

**動機：** 先前發現「摩擦力已經用不完（rolling regime）」不是真的用不完，而是 `PADDLE_FRICTION=0.4` 剛好落在讓碰撞鎖死在 rolling 天花板的區間。要驗證「調低摩擦係數能不能解放更大旋轉量」這個假設是不是普遍規律，寫了一支獨立 Node 腳本（逐字搬 `bounceOffPlane`/`bounceTangentialAxis` 等接觸力學核心函式），對「摩擦係數 μ（0.05~2.5）× 揮拍方向 y/z 比例 × 球拍速度 × 拍面 tiltY」做 6615 組合的系統掃描，固定球的來球狀態（強下旋 -70.5、中下旋 -45、不轉三種情境）。

**孤立測試的發現（只看單次碰撞的輸出旋轉，不接完整流程）：**
- 決定 rolling/sliding 的主要是摩擦係數，不是入射角——同一個 μ 在測試的角度範圍（0~32°）內幾乎都給同一個 regime。
- 以強下旋案例、固定其他參數為例，μ=0.13→0.15 是 rolling/sliding 的分界點（意外跟球桌碰撞用的 `CONTACT_FRICTION_MU=0.13` 接近）；μ 越低於分界點，sliding regime 給出的旋轉量越大，最多可以到 rolling 天花板的 4 倍（-49.1 對 -12.3 rad/s）。
- 但「不轉」的來球，sliding regime 幾乎從不產生正確方向的旋轉（13 組候選裡 0 組正確）——這個機制的本質是「延續/放大既有旋轉」，不是「憑空刷出旋轉」。

**套進完整流程（16 顆發球單拍 + 連續對打）驗證，結果是負面的：** 把 `PADDLE_FRICTION` 從 0.4 直接調到候選值（0.05/0.1/0.13/0.15），單拍準確率從 15/16 崩到 0~5/16，連續對打穩定顆數從 13/16 崩到 0~5/16。追查 `backspin_long_backhand` 這顆的崩潰原因：過網高度（netClearance）從正常的 0.08 暴增到 0.95~0.98，球飛太高、根本沒有落點。**原因：摩擦係數不只影響旋轉方向，還同時決定了球拍碰撞整條水平速度/彈道行為——目前的 `tiltX`/`tiltY`/力道公式/瞄準用的 `solveRacketVelXForTargetLandingX`，全部都是在 μ=0.4 這個特定摩擦行為下校準出來的，單獨換掉地基會讓上層所有公式一起跑掉。** 這跟先前 `swingDirection` 正負號修正單獨換會整套崩潰是同一個教訓：**碰到接觸力學核心參數的改動，不能單獨換，必須連同力道/拍面角度/瞄準一起重新校準。**

**决定：** 把摩擦係數重新校準併入 Stage 2（原本只規劃 `swingDirection` 正負號），跟揮拍方向、力道、拍面角度、瞄準公式一起做「196 樣本視窗取樣＋逐點搜尋＋回歸」，不要再單獨測試/單獨部署任何一個參數。

**使用者補充的真實世界對應觀察（很有價值，記錄下來當作之後設計的錨點）：** 使用者指出，實務練習中「如果對方回球下旋不夠，要合適地切回去就需要更高階的技術，不然容易處理得很失敗」——這正好呼應了「sliding regime 對不轉的球幾乎不會產生正確方向旋轉」這個發現的兩面：**這個摩擦力機制的物理本質決定了它只能延續/放大既有旋轉（借力），沒辦法無中生有創造旋轉。** 這代表：處理「來球旋轉不足」這種情境，很可能真的需要一套不同的技術模型或額外的自由度（例如更依賴球拍速度本身而非借力），而不是同一套切球公式硬調參數就能涵蓋——之後設計「熟練度/技術等級如何處理弱旋轉來球」時，應該以這個物理限制為基礎，而不是假設任何等級的切球公式都能線性地處理任意旋轉強度的來球。

### Stage 2 第一輪聯合搜尋：分數指標的漏洞，以及「不轉球不該用切球處理」的關鍵重新分類

**第一輪嘗試：** 用既有的 16 顆發球驗證題當目標函式，對 `swingDirection` 的 y/z 比例、`PADDLE_FRICTION`、固定 `tiltY`、力道公式的 `neutralMagnitude`/`k` 四五個參數做黑箱網格搜尋（直接跑真實驗證題算分，不是先對使用者標的樣本做回歸），比先前「196 樣本視窗＋線性回歸」的方法論更直接。搜尋分數從最早只換方向+摩擦係數的 10 分（滿分 32），逐步加入拍面角度、力道公式後進步到 20~23 分，但一直沒能追上目前部署版本的 28 分（15 單拍 + 13 連續對打）。

**發現分數指標本身有漏洞：** 追查得分最高的候選組合（`yRatio≈-0.15`、`μ=0.2`、`tiltY≈0.65`、力道公式重新調參）在連續對打時的實際旋轉變化，發現旋轉一路往上旋飄（-70.5 → +2.3 → +64.8 → +25.7 → +50.2 → 掛網），**完全沒有維持反向下旋**——因為評分函式只看「球有沒有出界/掛網」，沒有檢查「打出去的旋轉方向對不對」。搜尋因此找到的是「球還留在桌上，但已經不是切球動作，比較像某種上旋怪招」的組合，不是我們真正要的東西。

**討論後定位出更根本的問題：搜尋空間本身被不該存在的目標拉歪了。** 使用者指出：**弱旋轉/不轉的來球，真實球員通常直接換技術處理（上旋拉球去抵消、或撥球擋回去），不會硬選切球**——除非是刻意的訓練情境。回頭看搜尋過程：`yRatio` 一直被拉往 -0.05~-0.15（幾乎打平、沒有往下刷）、`tiltY` 被拉往較平的角度，正是因為搜尋目標裡包含了 `no_spin_long_forehand`/`no_spin_long_backhand` 這兩顆——用平拍面硬湊不轉的球，導致真正該有的「立起來、垂直往下刷」的切球動作被平均掉了。

**决定（雙方一致同意）：重新劃分驗證範圍。**
- `no_spin_*`（2 顆）**不再算進切球技術的聯合校準指標**——真實球員面對不轉球不會選切球，這兩顆屬於「應該換技術處理」的情境，不是切球公式的失敗案例（之後如果要模擬技術選擇本身，才需要另外處理，跟這裡的 Stage 2 是兩件事）。
- 剩下 14 顆 `backspin_*`/`sidebackspin_*` 才是切球技術真正該優化的對象，搜尋時 `swingDirection`/`tiltY` 應該偏向使用者描述的真實動作（拍面立起來、揮拍方向接近垂直往下），不該被不轉球拉往打平的方向。
- 「來球旋轉不夠強但仍選擇切球」這個情境（呼應使用者更早提到的「回下旋不夠需要更高階技術才切得回去」），屬於切球技術內部「熟練度如何處理偏弱旋轉來球」的難度分級問題，跟「不轉球該換技術」是兩個不同層次的問題，不要混在同一個參數搜尋裡解決。
- 目標函式也要修正：除了單拍準確率/連續對打穩定顆數，還要加入「打出去的旋轉方向必須維持/加強反向下旋（不能長期飄向正旋轉）」這個檢查，避免重蹈這輪的覆轍。

**重要修正（使用者當場指出，避免下次又搞反）：「拍面立起來、拍子垂直往下切」不是正常下旋球的動作，是旋轉非常弱的球才需要的特殊處理方式。** 正常的 backspin/sidebackspin（就是這輪要校準的 14 顆）**拍面應該要平**。上面這段討論把「弱旋轉球的特殊技術」誤套用成了整個 Stage 2 搜尋的目標方向，導致搜尋一路把 `tiltY` 往 0.6~0.8（偏立起來）拉，這跟 14 顆正常下旋球該用的平拍面方向是反的。既然不轉球已經排除在 Stage 2 範圍外，「立起來、垂直往下切」這個特殊技術這輪不需要出現在搜尋空間裡——14 顆的搜尋應該偏向平拍面（比較接近目前已部署的 `tiltY` 0.4~0.55 這個範圍），不該往立起來的方向搜。

**尚未執行：** 用修正後的範圍（14 顆 backspin/sidebackspin）、修正後的搜尋方向（拍面偏平，不是立起來）、跟修正後的目標函式（含旋轉方向正確性）重新做一次聯合搜尋。

### 使用者提出的「時鐘接觸面」幾何模型——已用模擬數據驗證成立（指導 Stage 2 以後的揮拍方向設計）

**模型內容（使用者提出）：** 從側面看球由左往右飛的拋物線，把球面本身當一個時鐘（12點=正上方、3點=飛行前進方向、6點=正下方、9點=飛行後方）。不同技術有各自的「目標接觸面」：
- 切球（push/chop）：目標接觸面是球的 **4-6 點鐘**方向。
- 拉球（loop/topspin）：目標接觸面是球的 **12-2 點鐘**方向。

使用者的推論：球本身的**速度方向**，也可以用同一個時鐘來描述，而這個方向決定了哪個接觸面「順勢可得」——因為球是球面，拍子要接觸某個面，選擇是有限的（不是拍子角度跟接觸點角度可以互相獨立選）。當球處於上升期，速度方向偏向 1-3 點鐘，這時貼近 12-2 點鐘的拉球接觸面最順手；球處於下降期，速度方向偏向 3-6 點鐘，這時貼近 4-6 點鐘的切球接觸面最順手。若要在上升期做切球，就要讓拍子在球速指向 1-3 點鐘的同時、持續咬住 4-6 點鐘的面而不先撞到別的面，難度自然變高。這也解釋了為什麼初學者被教導在下降期練切球——這時目標接觸面跟球速方向最一致，窗口最大最穩定。

**驗證方法：** 用 `backspin_long_backhand` 這顆發球的真實模擬軌跡，取接球方那一側、從最高點前一小段到下降中段共 5 個取樣點，用 `angleFrom12 = atan2(水平前進速度, vy) × 180/π` 換算成時鐘位置（0°=12點、90°=3點、180°=6點）。

**結果（完全對上模型）：**

| 階段 | vy | 時鐘位置 |
|---|---|---|
| 上升期 | +0.892 | 2.54 點鐘 |
| 接近最高點 | +0.472 | 2.76 點鐘 |
| 最高點附近 | +0.052 | 2.97 點鐘（幾乎正好 3 點，水平飛行） |
| 下降初期 | -0.403 | 3.21 點鐘 |
| 下降後期 | -0.858 | 3.44 點鐘 |

球速的時鐘位置隨著 vy 從正變負，平滑地從 2.5 點鐘掃過 3 點鐘整（水平飛行的轉換點），一路往下降期的 3.4 點鐘、繼續朝 4-6 點鐘方向前進——跟使用者的模型完全吻合，不是跳躍或矛盾的。

**對後續設計的指導原則（尚未套用，留給 Stage 2 執行時參照）：**
每個技術的「目標接觸面」本身，配合球速時鐘位置隨時間平滑掃過的軌跡，就決定了這個技術「順勢而為」的最佳時機窗口——不是任意指定一個揮拍方向再湊角度，而是：
1. 先確定這個技術的目標接觸面對應時鐘的哪個區間（切球 4-6點、拉球 12-2點）。
2. 再回頭找「球速方向掃過那個區間」對應的時間窗口（球的相位：上升期／最高點／下降期）。
3. `swingDirection`、`tiltY` 等揮拍參數的設計，應該以「在目標時機窗口內，用最小/最自然的動作咬住目標接觸面」為準則去反推，而不是先固定一個方向再回歸擬合。

這條原則跟 Stage 2（`swingDirection.y` 正負號修正）直接相關：目前部署的切球揮拍取樣/擬合都是在下降期擊球點附近做的，方向上是合理的（下降期本來就是切球該用的相位）；但下一輪重新校準力道/拍面公式時，應該同時記錄「擊球當下球速的時鐘位置」當作一個特徵，而不是只看擊球點座標，可能有機會用它取代目前效果不佳的變數，或是解釋連續對打時偶爾出現的不穩定樣本（時鐘位置偏離目標接觸面太多的那幾拍）。

**新發現：切球「子彈感/碰撞感」問題，根源可能是法向反彈係數（epsilon），不是切向摩擦。** 使用者觀察：真實切球幾乎純摩擦、沒有撞擊加速感，接觸點該有個「頓點」、均勻沿切線滑出去，但現在的手感很像碰撞彈開。實測 `backspin_long_backhand` 這顆切球目前套用的 `epsilon`（由 `dynamicPaddleEpsilon` 算出）是 **0.877**，非常接近完全彈性碰撞——這很可能才是「子彈感」的真正來源，不是這輪一直在調的 dwell/grip/friction（那些調的是切向，這個問題出在法向）。`PADDLE_RESTITUTION_LOW=0.9`/`PADDLE_RESTITUTION_HIGH=0.75` 目前是攻球/切球共用一組常數。**下一步（尚未執行）：** 幫切球用遠低於目前的法向反彈係數（量級待測，可能 0.3~0.5）測試觸感跟驗證分數；注意 epsilon 會連動影響 `normalImpulse`，進而影響摩擦衝量上限，不能只孤立測 epsilon，要跟其他參數一起重新驗證。

**膠皮/海綿參數化構想（討論過，尚未實作）：** 使用者指出真實球拍的膠皮選擇同時決定「反彈力強不強」跟「咬球/製造摩擦的能力」，這兩者不是誰好誰壞的取捨，而是可以獨立調整的兩個維度——使用者自己用的新手平衡膠皮就是活生生的例子：**反手膠皮彈性高、可牽制對手，但切球仍有一定水準（摩擦力沒有被犧牲）；正手膠皮彈性沒那麼誇張，但吃球更深，適合拉球品質。** 這推翻了原本想像的「進攻膠皮=高反彈/低摩擦、防守膠皮=低反彈/高摩擦」兩極對立模型——真實膠皮是分佈在 `(epsilon, friction)` 這個二維空間裡的不同點，不是一條線的兩端。

**建議的整合方式（尚未執行）：**
1. 不要新增一套獨立的膠皮物理路徑——讓「膠皮類型」直接覆寫既有的 `epsilon`（`dynamicPaddleEpsilon`/`PADDLE_RESTITUTION_LOW/HIGH`）和 `friction`（`PADDLE_FRICTION`）這兩個既有參數，例如 `RUBBER_TYPES = { ... }`，每種膠皮各自一組 `(epsilonLow, epsilonHigh, friction)`，不是二選一的 preset，是二維空間裡的自由點。
2. 正/反手可以用不同膠皮參數，之後甚至能變成玩家可選的遊戲功能，不只是內部調參工具。
3. 換膠皮參數一樣要照 Stage 2 的教訓，跟 `swingDirection`/`tiltY`/力道公式一起重新聯合校準，不能只驗一半就上。

**驗證結果（用量重置後已實測）：epsilon 果然不能單獨調，模式跟摩擦係數/揮拍方向一模一樣。** 固定 `dynamicPaddleEpsilon` 回傳值，掃 0.877/0.7/0.6/0.5/0.4/0.3，16 顆驗證分數隨 epsilon 降低單調崩潰（epsilon=0.877 時 singleOk=11/rallyStable=10〔60拍上限〕，降到 0.3 時 singleOk=4/rallyStable=0）。追查 `backspin_long_backhand` 這顆的崩潰原因：**出球速度隨 epsilon 降低單調下降**（2.94→2.65→2.32→2.01 m/s），球飛不到對方半場（落點 z 從 -0.58 一路推正到 +1.03，等於連自己這側都沒飛過），netClearance 也跟著出問題（掛網或直接飛不夠遠）。**原因：法向反彈本身貢獻了不少出球動能，目前的力道公式是配合 epsilon≈0.877 這個「有反彈加成」的前提校準的，直接降低 epsilon 等於抽掉這部分動能，力道公式沒有補償，球自然飛不夠遠。**

**結論：epsilon 正式併入 Stage 2 聯合校準範圍（現在是揮拍方向 + 摩擦係數 + 拍面角度/力道公式 + epsilon，四個參數一起搜）。** 邏輯上，如果要做到「真實切球的頓一下、軟死手感」（低 epsilon）同時維持過網距離，力道公式/揮拍速度需要整體上調，補償法向反彈損失的動能——這也呼應「頓一下、靠切向摩擦帶走動能而不是法向反彈」的真實手感描述：出球的能量來源应该更依賴揮拍本身的速度（切向 drive），而不是依賴法向反彈；目前的架構剛好相反（大部分能量來自法向反彈，切向摩擦只負責旋轉）。**尚未執行：** 四參數（`swingDirection`、`PADDLE_FRICTION`、`epsilon`、力道公式）聯合網格搜尋，目標函式一樣要包含旋轉方向正確性檢查，範圍限定 14 顆 backspin/sidebackspin。

**四參數聯合搜尋已執行（用量重置後），結論：這個架構下「準確率」跟「真實反向下旋」互斥，不是搜索範圍不夠。** 目標函式改成同時記錄 `singleOk`（14顆單拍成功數）跟 `correctSpinFrac`（連續對打每拍 outgoing topspin < 0 的比例，取代之前只看最後有沒有出界的漏洞指標）。兩輪搜尋（第一輪：`swingDirection.y比`×`PADDLE_FRICTION`×`epsilon`×力道 neutralMag；第二輪：把力道範圍推更高到 2.4、`tiltY` 也一起搜）得到一致的結果：
- 只要 `swingDirection.y` 用物理上正確的負值（往下刷）、`epsilon` 調低（軟死手感），`correctSpinFrac` 可以到 0.5~0.67（旋轉方向多數正確），但 `singleOk` 死死卡在 2~4/14（力道再怎麼加大都補不回來，最高只到力道=2.4時 singleOk=8 但代價是 correctSpinFrac 崩到 0.17）。
- 反過來，維持接近目前部署的參數（`swingDirection.y`正值、`epsilon`≈0.877），`singleOk`可以到 11/14、連續對打可以撐 100+拍，但 `correctSpinFrac` 只有 0.04~0.05（幾乎都是錯誤方向的正旋轉）。
- 沒有任何一組參數同時做到兩者都好——這不是網格不夠密，是這個「單一瞬間碰撞 + Coulomb 摩擦」架構本身，兩個目標在數學上互相拮抗：要嘛靠法向反彈+順向揮拍拿到落點準確率（但旋轉方向錯），要嘛靠低反彈+逆向刷球拿到正確旋轉方向（但動能不夠、球飛不夠準/不夠遠）。

**結論與下一步：Stage 2（純參數重新校準）本身可能不足以同時解決這兩個目標，需要先做 Stage 1（球拍速度尺度校準）或 Stage 3（接觸期球拍持續加速/dwell），把物理空間撐大，Stage 2 的聯合搜尋才有機會找到真正兩全的解。** 建議下一輪優先做 Stage 1：把切球揮拍速度從目前的模擬尺度（~0.2 m/s）校準到真實量級（1~3 m/s），重新跑這輪已經寫好的四參數＋旋轉正確性聯合搜尋，看看有沒有機會在更大的速度空間裡找到兩全的解——如果 Stage 1 做完還是找不到，才需要認真考慮 Stage 3（dwell 動態加速）或接受「這個技術層級的架構就是有取捨，需要另外的自由度如熟練度分級」。

**背景研究：拍身（底板）材質與 dwell 時間的關係，補充 Stage 3 的真實對應（查過文獻，尚未實作）。** 一支球拍是「底板+海綿+膠皮」三層組合，之前只討論了膠皮/海綿（`epsilon`/`friction`），拍身這一層還沒對應到模型裡。查到的重點：
- 拍身的「軟硬」直接決定撞球時彎曲多少，進而直接影響 dwell 時間——**硬底板（碳纖維複合材質、或外層碳纖構造）彎曲少 → dwell 短 → 反彈直接穩定，適合防守/快攻，但製造旋轉能力較弱；軟底板（全木、尤其內層碳纖構造）彎曲多 → dwell 明顯拉長（文獻估計內層碳纖構造可讓 dwell 拉長 15~20%）→ 咬球更久，更容易做旋轉跟精準控制。**
- 攻擊型底板的「速度分級」約 8.5~9.5（滿分10），厚度 5.7~6.2mm，重量 85~95g——可以當作「揮拍動能轉換成出球速度效率」的量級參考，對應 Stage 1（球拍速度尺度）跟 Stage 5（時機/傳遞效率）。

**對模型結構的啟發：dwell 時間不該是單一常數，應該拆成「底板彎曲貢獻的時間」+「海綿壓縮貢獻的時間」兩部分相加，才有跟真實器材對應的物理意義。** 這也可能是目前卡住的「epsilon 低（軟、正確觸感）但動能不夠打準」這個矛盾的一個潛在出路：如果 Stage 3 把「底板彎曲的彈性儲能/釋放」也算進去（不是只有海綿在儲能），出球動能的來源就不再只依賴法向反彈係數，可能提供我們目前缺的那個「補償機制」，讓「低反彈+正確旋轉」跟「動能足夠打準」不再互斥。這條線等 Stage 1 之後、真正要做 Stage 3 時再詳細設計。

### Stage 1 卡住之後的深入討論：揮拍角度實驗、動力鏈/時機文獻、以及「單點碰撞模型」本身的方法論極限

**背景：Stage 1（力道公式 clamp 下限拉高到真實尺度）單獨測試，跟摩擦係數/epsilon/揮拍方向一樣，會單獨崩潰（乾淨環境下實測：floor=0.02 時 15/13，floor=0.65 時崩到 5/2）。這確認了這 5 個參數（揮拍方向、摩擦係數、epsilon、力道公式、拍面角度）高度耦合，沒有一個能單獨調——已經不是「先做 Stage 1 再做 Stage 2」的順序問題，而是要一起解一個聯立方程式。手動網格搜尋在這個維度下已經碰到瓶頸（要嘛組合數太大 timeout，要嘛切太粗漏解），建議之後改用真正的優化演算法（Node 腳本、隨機搜尋+爬山法），不要再繼續手動網格搜。**

**過程中發現一次真實的測試方法論失誤，已修正：** 連續多次 monkey-patch 測試，因為每次都是「抓當下的全域函式當作 original、測完還原成那個 original」，一旦前面某次沒還原乾淨，後面每次都會把污染狀態當成基準繼續傳下去。這輪真的發生過（`swingDirection`/`computeAdaptivePushMagnitude`/`computeAdaptivePushTiltY` 都被污染），靠重新整理頁面解決。**教訓：以後每輪大範圍搜尋前後，都要重新整理頁面拿回乾淨狀態，或者把「真正原始版本」的函式定義存到獨立、不會被覆寫的地方，不要每次都抓當下的全域函式當 original。**

**使用者提出一個關鍵疑惑，帶出很有價值的一連串討論：「拍面傾斜的狀態下，不管把力量調多大，都切不出很高的球，這很奇怪。」** 追查過程：
1. 實測發現：固定揮拍速度=0（完全不加力），光靠拍面角度（`tiltY`）跟法向反彈係數（`epsilon`）的幾何反彈，就能讓不轉球有一定的上升高度；把 `tiltY` 從切球預設的 0.5（傾斜約26.6°）一路開到 1.7（傾斜約59.5°），上升高度從 0.22m 衝到 0.745m（存在一個最佳角度，過了之後角度太斜，垂直分量反而下降）。**但使用者指出：這個角度（~60°）對正常切球技術來說明顯太大，不該是 push 常見的技術範圍。**
2. 使用者提出另一個可能的因素：**「合力」——全身協調動作透過球拍在恰當時機點合在一起，不用很費力，能打出效率超過好幾倍的球。** 查文獻確認「動力鏈/加總速度原則」（summation of speed principle）：下肢→軀幹→手臂依序加速，每節都要抓到「上一節剛好達到最大轉速」的瞬間接手，才能讓最末端（球拍）達到最高速度——這不是動作變猛，是時機精準對齊的加乘效果。
3. 直接測試「改變揮拍方向讓它更對準來球方向」這個簡化假設，**結果是負面的**：固定拍面角度跟力量大小，換方向反而讓出球速度變小（因為目前預設方向已經跟拍面法向量對得不錯）。這排除了「只是方向沒對準」這麼簡單的解釋。
4. 使用者接著描述真實揮拍的加速度曲線：「球拍在揮擊過程中不斷加速，好的揮擊動作剛好在擊球點加速度達到最快，擊球後自然減速。」**使用者質疑：我們模擬的是碰撞瞬間發生的事，怎麼會跟球拍加不加速有關？**——這是一個很尖銳的方法論問題，逼出了核心澄清：**不是碰撞前後的加速度影響碰撞本身，是「碰撞本身其實不是瞬間，是一段有 2~8 毫秒真實長度、其間加速度持續變化的過程」，我們把這段過程硬塞進一個零時間的瞬間模型時，選錯代表速度值，就會系統性低估真實動量傳遞。**
5. 使用者進一步歸納：接觸期間至少有三件事同時隨時間變化——**(a) 球拍線速度在變（Stage 3的加速度曲線）、(b) 球拍法向量角度也因為揮拍是弧線運動而微幅變化（新發現，之前沒特別談過）、(c) 海綿/拍身壓縮儲能再釋放（Stage 4的catapult效應）**——並質疑：如果這是一個先後發生、彼此影響的過程，單一個點的模型有可能解得出等價結果嗎？
6. **誠實的答案：一般情況下不行，除非運氣好湊到一組「有效參數」能巧合等於真實積分結果——但要找到那組參數，通常得先做過完整的逐步模擬或有真實實驗數據，這是邏輯上有點循環的問題。** 單點模型至少有兩個結構性盲點：滾動/滑動 regime 轉換是有門檻的非線性行為，可能在接觸中途切換（單點模型只看最終值，看不到中途換檔）；海綿壓縮/釋放是先後次序的彈簧-阻尼行為（先吃能量、後還能量），不是一個係數能表達的。

**文獻查證：真實學術研究確實是用「質量-彈簧-阻尼」跟有限元素模型做球拍-膠皮碰撞，不是單點瞬間碰撞（增強了我們這條思路的可信度）：**
- 有論文（"Non Linearity of the Ball/Rubber Impact in Table Tennis: Experiments and Modeling"）用 3D 有限元素模型，把海綿/膠皮的「時間相依黏彈性行為」（隨時間變化，不是固定係數）納入計算。
- 摩擦力在真實過程裡是**非線性、速率相依**的能量耗散機制（跟我們發現摩擦係數在簡化模型裡表現得不直觀，是同一件事的不同面向）。
- 拍柄/底板的彈性可以讓 dwell time 拉長到**三倍**——量級比之前查到的「15~20%」更大，值得之後校準時參考。

**使用者補充一個重要的校準錨點：便宜球拍（摩擦力、彈性都不好）基本動作還是做得出來，只是沒有俐落的能量轉換/陡峭曲線。** 這代表：**「揮拍時機/動力鏈」（Stage 3）是必要條件，決定有多少動能進入接觸期；「膠皮/海綿/拍身品質」（Stage 4）是效率放大器，決定這些動能轉換得多俐落、多陡峭，兩者是分開的槓桿，不是同一組參數——便宜球拍是「Stage 3 正常、Stage 4 效率低」，不是兩者都壞掉。**

**規劃好的下一步（研究版，尚未實作，只放 `return-studio.html`，不動 `game4.html`）：把單一瞬間衝量模型，換成在真實 dwell 期間內逐步積分的模型，拆成兩個獨立槓桿：**

- **Stage 3a｜球拍運動學（時機/動力鏈槓桿，決定「有多少動能進場」）**：定義一個簡化的加速度曲線（揮拍速度先升後降），新增一個「時機品質」參數（0~1，代表擊球瞬間落在這條曲線峰值的哪個位置），先只決定「接觸期間各個小片刻的球拍速度分別是什麼」，不動摩擦力/彈性部分。
- **Stage 4a｜接觸力學（器材品質槓桿，決定「動能轉換得多俐落」）**：法向力改用彈簧-阻尼模型（法向力=彈性係數×壓縮深度+阻尼係數×壓縮速度，壓縮深度逐步積分，先壓縮後釋放），摩擦力改用「當下那個小片刻的法向力」去限制（不是像現在整個碰撞只用一個總衝量），讓滾動/滑動 regime 有機會在接觸中途真的切換。彈簧係數/阻尼係數/摩擦係數三個，就是「膠皮/海綿/拍身品質」的量化代表（便宜球拍＝三個數值都偏低但不是零）。
- **整合方式**：把 dwell 時間切成 N 個小步（20~50步），每步：算球拍瞬時速度（來自3a）→ 更新壓縮深度 → 算當下法向力+摩擦力 → 把這一小步的衝量積分進球的線速度/角速度 → 進下一步。
- **驗證項目（對應這輪三個現象，缺一不可）**：(1) 低彈簧/阻尼/摩擦（便宜球拍）時，技術仍可執行、只是曲線不陡峭；(2) 時機品質高時，就算表面揮拍幅度不誇張，也能有明顯更高效率（解決「要5倍力量才有感覺」的疑惑）；(3) 法向力用彈簧-阻尼取代固定反彈係數，檢查「子彈感/頓點感」是否更貼近真實描述；(4) 一樣要跑 14 顆單拍+連續對打+旋轉方向正確性驗證題，跟目前部署版本比較，不能只看理論合不合理。

**Stage 3a 原型驗證後的重要簡化修正（已用數值實驗確認，Stage 3a 不需要逐步積分）：** 用 sigmoid 揮拍速度曲線（反曲點=加速度最高點）做數值實驗，比較「dwell期間內平均速度」vs「反曲點瞬時取樣值」，發現在合理的真實時間尺度下（dwell 2~5毫秒 vs 整體揮拍 60~300毫秒），**兩者幾乎沒有差異（提升~0%）**——因為反曲點左右對稱，dwell 窗口相對於整條揮拍曲線實在太短，曲率在這麼小的窗口裡看不出效果。但把取樣點從「加速度最高點」往後移到「更接近速度真正峰值」的位置，數值明顯高很多（+53%量級）。**這代表真正決定效率的不是「dwell期間有沒有持續加速」，而是單純「擊球那一刻，球拍在它自己整條揮拍速度曲線上走到哪個位置（接近峰值 vs 提早/太晚）」——這只是一個簡單的時機品質純量乘數，不需要真的把 dwell 拆成小步去積分速度曲線。結論：Stage 3a 改成簡單模型 `有效球拍速度 = 揮拍峰值速度 × 時機品質(0~1)` 即可，不需要逐步積分；Stage 4a（彈簧-阻尼/壓縮釋放）不受此簡化影響，因為壓縮/釋放本身就是在 dwell 這個時間尺度上發生的過程，仍然需要逐步積分。**

**Stage 4a 原型驗證：彈簧-阻尼法向碰撞的數值實驗，重要結論是「catapult 效應不是免費能量，是損失得比較少」（已用數值實驗確認）。** 用真實 dwell 時間反推彈簧係數（半週期目標 5~8ms → k≈416~1066），測試不同阻尼比（相對臨界阻尼的比例）：

| 阻尼比 | dwell | 等效反彈係數 |
|---|---|---|
| 0.05 | 5.02ms | 0.854 |
| 0.15 | 5.06ms | 0.620 |
| 0.3 | 5.24ms | 0.371 |
| 0.5 | 5.78ms | 0.160 |
| 0.7 | 7.02ms | 0.044 |

（閉合速度固定3.0、球質量0.0027kg，dwell隨阻尼比略為拉長，符合真實現象）

**關鍵物理限制：純被動彈簧-阻尼系統，出球速度永遠不會超過進場速度（等效反彈係數必然 ≤ 1）——沒有外部驅動的彈簧不可能無中生有創造能量。** 這代表所謂「catapult 效應」，物理上不是「把能量放大好幾倍」，而是「能量損失得比較少」（阻尼比低=損失少=接近現行部署值0.877，阻尼比高=損失多=接近死球0.04）。**這修正了稍早「合力/timing能無中生有創造效率」的假設方向：真正的解釋是「阻尼比（膠皮/海綿品質）決定能量返還比例，這個比例本身涵蓋將近20倍的範圍（0.85 vs 0.04），配合球拍本身前送速度（`planeVel`，模型裡本來就有疊加），已經足夠解釋『便宜球拍效率差好幾倍』這個真實觀察，不需要額外假設任何免費能量來源。**

**目前部署的固定 `epsilon≈0.877`，對應到這個彈簧-阻尼原型裡，大約是阻尼比 0.05 左右（非常低阻尼、接近理想彈性）的等級——這代表目前的切球觸感設定，其實對應到「膠皮品質很好、幾乎不損耗能量」的那個極端，而不是切球該有的「軟、死、頓一下」的觸感。這跟稍早發現的「子彈感」問題（epsilon=0.877太接近完全彈性）是同一件事，數值上完全吻合：如果切球膠皮該對應中高阻尼比（例如0.3~0.5，等效反彈0.16~0.37），這個範圍剛好跟稍早測試「epsilon調到0.3~0.5會讓落點準確率崩潰」的結果放在一起看，代表「觸感正確的阻尼比」目前確實會讓動能不夠打準，需要力道公式/球拍速度一起補償——這條線索完全串起來了，尚未實作，等 Stage 1（球拍速度尺度）確定後，可以直接把這個彈簧-阻尼原型接進 `return-studio.html` 研究版，取代現在的固定 epsilon。

**重要提醒（使用者當場抓到，接線時務必注意，避免單位錯位）：我們的引擎重力是 -4.2（不是真實的-9.8），時間拉長係數 `SIM_TIME_DILATION≈1.528`（見前面「重力/時間尺度發現」段落）。** 這個 dwell/彈簧-阻尼原型本身的物理（質量、彈簧係數、阻尼、dwell時間）**不受重力影響**——5~8毫秒的接觸過程裡，重力造成的速度變化（無論4.2或9.8，Δv≈0.02~0.05 m/s）跟接觸瞬間速度量級（1~3 m/s）比起來完全可忽略，dwell/彈簧係數本身可以放心用真實物理量去校準。**但單位換算是個真正的陷阱：這個原型測試時用的 `closingSpeed=3.0` 是假設「真實 m/s」，之後要把原型接進 engine 時，engine 裡的 `hitVel`/`techVel` 都是「模擬尺度」（比真實 m/s 小，差了 1.528 倍）——如果直接把模擬尺度的速度值餵進這個假設真實m/s校準出來的彈簧-阻尼模型，會產生一次沒被注意到的單位錯位，算出來的 dwell、等效反彈全部都會是錯的。接線時務必先把 `hitVel`/`techVel` 乘上 1.528 換算成真實 m/s 再餵進模型（模型內部維持用真實物理量算，算完結果再除回模擬尺度），或者乾脆重新用模擬尺度的速度去反推校準 k/c，兩種都可以，但不能兩邊尺度混用。**

**Stage 4a 完整原型已在瀏覽器內驗證成功（切向摩擦+旋轉逐步積分都補齊了）。** 把切向部分改成：每個小時間步，直接重複呼叫既有、已驗證過的 `bounceTangentialAxis()`（沿用它的滾動/滑動判斷邏輯，只是餵進去的是「這一小步的法向衝量」而不是整次碰撞的總衝量），對 e1/e2 兩個切向軸分別處理，跟現有 `bounceOffPlane` 的結構完全對齊。

**過程中抓到兩個真正的隱藏 bug，修正後結果才對齊：**
1. **只把線速度轉換成真實 m/s，忘記把旋轉（omega/spin）也用同一個 `SIM_TIME_DILATION` 係數換算。** 旋轉也是「每秒轉多少」，跟線速度一樣是時間相關的量，必須用同一套換算，結尾也要記得除回模擬尺度。
2. **重建最終旋轉向量時，漏掉了「旋轉沿拍面法向量方向的分量」（`normalSpinComp`）。** 這個分量在整個模型裡完全不受摩擦影響，應該原封不動保留到最後，直接相加回去，不能只用切向分解出來的 e1/e2 分量重建。

**修正後驗證結果：** 用同一個真實接觸情境（`backspin_long_backhand` 這顆），逐步積分版本（阻尼比0.05，等效反彈0.854，接近現行部署的0.877）算出的 `topspin=18.39`、`sidespin=67.47`、`vel.x=-1.044`，跟現行單點模型的結果**逐位精確吻合**——證明逐步積分版本正確泛化了現有的滾動/滑動摩擦物理，不是另一套邏輯，只是換了一種算法路徑。而且測試不同阻尼比（0.05~0.5）時，旋轉維持完全不變（18.39/67.47），只有速度隨阻尼比下降——重現了稍早發現的「一旦進入滾動regime，旋轉不再受摩擦力影響」現象，代表新模型行為自洽。

**尚未執行：** 把這個驗證過的逐步積分接觸函式，正式接進 `return-studio.html` 研究版（取代 `bounceOffPlane` 給 push 技術用，其他技術不動），跑一次完整的 14 顆單拍+連續對打+旋轉方向正確性驗證題，找出合理的彈簧係數k/阻尼比範圍，看能不能同時做到「觸感正確（低反彈、頓一下）」跟「動能足夠打準」——這是稍早卡住的核心矛盾，這個模型是目前最有機會真正解開它的候選方案。

**已實作並接線（研究版，`return-studio.html`）：** 新增 `bounceOffPlaneSubstepped()`（逐字對應上面驗證過的原型）、`PUSH_CONTACT_MODEL`（預設`'instant'`，不影響任何既有行為）、`PADDLE_SPRING_K`/`PADDLE_DAMPING_RATIO` 常數，並在 `makeRacketReturnVelocity()` 裡加了一個 `applyPushContact()` 分派函式——`PUSH_CONTACT_MODEL==='substepped'` 且技術是 push 時才會走新模型，其他情況逐字維持原本的 `bounceOffPlane`。瞄準用的 `solveRacketVelXForTargetLandingX` 內部仍用舊模型求解方向（近似，只有最終套用的碰撞結果換新模型）。**部署前確認：切回預設 `'instant'` 後，16顆驗證分數跟原本完全一致（15/13），證明重構沒有動到既有行為。**

**驗證結果（重要，誠實的負面結論）：把新模型接進去，跟「揮拍方向 y比」一起做聯合搜尋（14顆背景範圍，含旋轉方向正確性檢查），得到的結果模式，跟稍早用固定epsilon/摩擦係數測試時完全一樣：**
- 維持現行揮拍方向（y=+0.3）：`singleOk=12/14`、連續對打可撐 222 拍，但 `correctSpinFrac=0.02`（旋轉方向幾乎都錯）。
- 換成物理上正確的負值揮拍方向（y=-0.15~-0.5）：`correctSpinFrac` 可以到 0.6~0.71（旋轉方向多數正確），但 `singleOk` 崩到 0~1/14，連續對打撐不到1拍。
- **沒有任何阻尼比（0.05~0.7）能讓兩者同時成立——換更真實的接觸模型，並沒有解開這個矛盾。**

**結論：Stage 4a（彈簧-阻尼）本身不是這個問題的答案，它只能讓觸感更真實，沒辦法注入額外能量。真正卡住的仍然是 Stage 1（球拍速度尺度太小）——不管接觸模型多真實，餵進去的球拍速度數值本身還是太小，撐不起「正確方向+準確」同時成立。這證實 Stage 1 不是可有可無的替代方案，是必要的前提，Stage 4a 要跟 Stage 1（球拍速度校準）一起搭配，才有機會真正解開這個核心矛盾。下一步：先做 Stage 1，再回頭用這個已經驗證/接線好的 Stage 4a 模型重新測一次聯合搜尋。**

### 使用者提出的關鍵一致性檢驗：「發球也用差不多的動能，為什麼發球發得起來？」——證實問題是引擎共通的落差，不是切球獨有

**使用者的疑惑：** 如果切球真的需要這麼高的球拍速度才能同時做到「準確+正確旋轉」，那為什麼發球可以正常運作？正常的發球、切球、或任何技術，人打起來動能其實差不多。

**先確認一個結構性事實：發球的旋轉，根本沒有經過球拍碰撞物理。** 查證 `physics-presets.json`，每顆發球的 `spin`（topspin/sidespin）都是**直接寫死的固定數值**，`solveBaseVelocity`/`solveServeBounceVelocity` 只是在解「球要用什麼初速度才能從拋球點飛到指定落點」的純拋物線反推，完全沒有「揮拍速度→球拍碰撞→旋轉」這一段計算。發球從來沒有被拿去驗證過切球撞到的那道牆存不存在。

**直接測試：如果發球也套用同一套球拍碰撞模型（球幾乎靜止剛拋起、球拍用某個速度+角度去碰它），要多大的揮拍力量才能做出典型發球該有的旋轉量級（~70~125 rad/s）？**

| 揮拍力量(magnitude) | 出球旋轉 | 出球速度 |
|---|---|---|
| 0.3（接近切球現行"正常"力道） | 24.8 | 0.59 |
| 0.7 | 7.6 | 0.18 |
| 1.5 | 26.9 | 0.96 |
| **3.0** | **91.5**（進入目標區間） | 3.27 |
| 6.0 | 156.1 | 7.51 |

**結果：要打出典型發球旋轉量級，同一套碰撞模型也需要 magnitude≈3.0，大約是切球「正常力道」（0.5~0.7）的4~6倍——跟切球卡住的倍數是同一個量級。** 這證實：**發球「感覺上」用小動能就做到了，是因為它的旋轉數值根本沒有經過這套物理、繞開了這道牆，不是它真的用小動能就跨過了。問題不是切球的物理需求特別不合理，是我們的簡化碰撞模型普遍性地需要比直覺更快的揮拍速度才能生成真實量級的旋轉——這是整個引擎共通的落差（呼應這個session一開始就發現的「球拍速度比文獻慢一個數量級」），不是切球獨有的異常，只是發球剛好用「直接寫死數字」的方式意外沒被檢驗到。**

**技術附註（之後做發球揮拍模型時要注意，不能直接沿用切球參數）：** 測試中意外發現，同一組揮拍方向，在「發球情境」（球幾乎靜止）下輸出的旋轉方向是正值，跟「回擊情境」（來球已經在快速移動）給出負值不一樣——代表旋轉方向的正負，不只看揮拍方向本身，也跟來球當下的速度狀態有關（相對滑動方向會因為來球速度而改變）。之後如果要讓發球也走真實揮拍模型，需要另外校準揮拍方向/拍面角度，不能直接套用切球那組已驗證的參數。

**這個發現對 Stage 1 的必要性是很有力的佐證：** 不是我們校準切球校準壞了，是整個引擎的「球拍速度尺度」從一開始就偏小，只是大部分技術（發球）目前繞開了這個問題（用寫死數值），只有切球回擊這條路徑真的在用揮拍模型算旋轉，所以率先撞到了這道牆。之後做 Stage 1（球拍速度尺度校準）時，這個發現代表校準完成後，也應該用同樣的方法回頭檢查「發球如果要用真實揮拍模型生成，需要的速度跟校準後的目標範圍對不對得上」，作為 Stage 1 是否真正校準到位的一個交叉驗證。

**使用者身體經驗的交叉驗證：** 使用者指出「發下旋球跟切球的身體感是完全不同的」——這跟模型算出來的「同一組揮拍方向參數，在發球情境（球幾乎靜止）跟回擊情境（來球已快速移動）下給出相反的旋轉方向」互相吻合，是身體經驗跟模擬結果各自獨立得出、但一致的交叉驗證。

**今天整輪推導最精簡的核心洞見（使用者總結）：**
- **單純把球「轉向」（例如彈很高），是接近被動的幾何/彈性問題**——只要拍面角度夠大，光靠彈性反彈的幾何redirect就能做到，幾乎不需要精準的力道/時機/旋轉控制（已驗證：揮拍速度=0，只調`tiltY`從26.6°到~60°，上升高度就從0.22m衝到0.75m）。
- **但要完成真正的切球技術——控制軌跡、落點、旋轉方向都要對——就不是單一因素能決定的，是好幾個變因（揮拍方向、拍面角度、摩擦係數、法向反彈/彈簧-阻尼、球拍速度、時機）互相耦合的問題。** 今天測過的每一次「只改一個參數」（摩擦係數、epsilon、揮拍方向、力道公式），單獨改動都會讓整個系統崩潰——只有這幾個變因同時對到某個組合，技術才會成立，目前還沒找到那個組合（卡在需要先做 Stage 1）。

**Stage 1 初步嘗試（拉寬力道公式上限，搭配 Stage 4a 模型）：把力道公式的 clamp 上限從現行 2.0 拉寬到 4.0（呼應發球一致性測試裡「serve等級旋轉需要magnitude≈3」這個數字），跟揮拍方向/阻尼比/拍面角度一起做聯合搜尋，結果是「有改善但還沒解開」：** 最佳組合（`yRatio=-0.24`、阻尼比0.05、`tiltY=0.5`、`neutralMag=1.5`）達到 `singleOk=9/14`、`correctSpinFrac=0.13`——比起先前「只能二選一」的兩極結果（要嘛11+/0.02，要嘛1/0.6+）更往中間靠了一些，但還遠不到「兩者都好」。**這代表拉高力道上限方向正確、確實有幫助，但目前測試的幅度/組合還不夠，尚未找到真正兩全的解——下一步可以嘗試更大的力道範圍、更精細的搜尋，或是認真考慮 Stage 3（加速度曲線/時機品質）用簡化的乘數模型疊加進來，看能不能補上剩下的落差。
- **背後的道理：「轉向」只是要達成一個寬鬆的目標（球要往上飛，多高都算數），解空間很大；「技術」要同時命中好幾個窄的目標（精確落點+正確旋轉方向+能穩定連續對打），每多一個限制條件，可行的參數組合就被壓縮一次，自然需要更多因素一起被「湊對」才行。**

**Stage 1 微調搜尋（局部精修）：** 診斷發現「加大力道，旋轉反而越來越往錯誤方向飄」（`neutralMag`從1.5→3→4.5，`backspin_long_backhand`這顆的outgoing topspin從+2.05→+25.7→+49.4）——不是準確率單獨變差，是這個揮拍幾何本身，球拍越快滾動平衡點越偏向錯誤方向。但也發現 `neutralMag=1.5` 時旋轉已經很接近零，在這附近微調拍面角度/揮拍方向，找到目前最佳點：`yRatio=-0.2`、`tiltY=0.4`：`singleOk=9/14`、`correctSpinFrac=0.21`（比最初的0.02進步十倍，但仍未解開核心矛盾）。**手動網格搜尋已經開始碰到報酬遞減+30秒逾時的天花板，決定改用獨立 Node 優化腳本，不再繼續在瀏覽器裡切格子碰運氣。**

**使用者提出的正確性驗證：鏡射連續對打時，旋轉的正負號判斷是用「canonical（功能性）座標」還是「真實/絕對座標」？——已用實際追蹤數字驗證，確認做法正確，沒有踩到潛在陷阱。** 追蹤 `flip=true`（真實世界球在-z側）那一拍的實際數字：

```
真實座標來球旋轉:   -30.44
canonical來球旋轉:  +30.44（鏡射後，符合negation規則）
canonical回擊旋轉:  +63.94（correctSpinFrac用這個判斷 → 判定錯誤，因為是正值）
真實座標回擊旋轉:   -63.94（鏡射轉回真實座標，這是球真的帶著飛的旋轉）
```

**這組數字證實了一個真實存在的陷阱：如果拿「真實座標回擊旋轉」（-63.94）去判斷對錯，會誤判為「負值=下旋=正確」，但這其實是錯的方向——這一拍在canonical（功能性）座標下是+63.94，對這顆球實際的飛行方向而言是上旋，不是下旋。** `topspin` 是固定世界座標系的角速度（跟球當下往哪個方向飛無關），但「這對這顆球而言算不算下旋」是相對於球自己的飛行方向定義的功能性概念，兩者不是同一件事——這正是當初設計鏡射技巧時，旋轉要跟著取負號（`mirrorSpin`）的原因。**確認：`correctSpinFrac` 實際使用的是 `ret.spin.topspin`（canonical座標，鏡射轉回真實座標「之前」的值），檢查方式正確，沒有踩到這個陷阱——今天算出來的所有 `correctSpinFrac` 數字都是可信的。**

### 重大突破：反彈方向跟切向滑動方向的耦合，可能才是今天卡了一整天的真正缺失環節（不是球拍速度尺度）

**觸發：** 使用者分享一個真實練習（單手拋球、持拍手用切球角度接，球會同時彈高+快速旋轉，再用拋球手的三根手指接住，俗稱「指間轉球」，用來練摩擦球感）。追查過程：

1. 先假設球「純垂直下墜」測試，結果旋轉方向對但沒彈起來（或反過來），一直測不出兩者兼顧的組合——不管球飛行方向設哪個軸、拍子揮動方向設哪個軸，純水平/零垂直分量的揮動，**在瞬間彈性碰撞模型跟 Stage 4a 逐步積分模型裡，都找不到能同時滿足「彈起來」跟「正確下旋」的窗口**，這個結果很穩定，不是還沒試到對的方向。
2. **釐清一個用詞誤解（重要）：先前描述的「26.6°」/「~60°」是拍面「法向量」偏離水平的角度，不是拍面本身的角度——兩者互為90°互補。`tiltY=0.5`（法向量26.6°）其實對應「拍面本身偏立起來、離水平63.4°」；`tiltY=1.7~2.3`（法向量59~66°）才對應「拍面本身很平、離水平只有24~31°」。使用者對「切球拍面要平」的認知是對的，跟這次發現的有效角度範圍一致，是先前報角度的方式讓人誤會方向。舊註解「強下旋需要立起來的拍面（tiltY較高）」這個文字敘述，跟`tiltY`實際的幾何意義相反，需要修正用詞（數值方向本身可能沒錯，只是「立起來/平」這兩個中文詞用反了）。**
3. **使用者提出的關鍵假設：拍子完全沒有主動揮動（沒有往上的向量），彈高+旋轉的來源，其實是球陷進海綿滑動摩擦、壓縮拍面材質之後，形變回彈的力量——這個回彈方向沒有完全咬合拍身（剛體）的幾何法向量，而是被球實際的滑動方向帶偏。**
4. **驗證：把 `bounceOffPlane` 的反彈方向，從「純粹剛體法向量」改成「剛體法向量與球實際接近方向反方向的混合（blend=0.6）」，在拍子完全靜止（水平/垂直揮動速度皆為0）的情況下，同時得到彈起0.43m跟下旋-50.3——完全吻合描述的練習效果，且不需要任何主動揮拍動作。**

**再拿這個修正過的模型跑 14 顆單拍+連續對打驗證題（跟現有力道公式/瞄準機制/揮拍方向組合），結果：`correctSpinFrac` 從原本卡住的 0.02~0.3，直接衝到 0.93~1.0（`blend=0.6`）——這是質的差異，不是量的差異，是今天所有嘗試裡唯一一次讓「正確旋轉」從稀有事件變成幾乎必然發生。** 但同時 `singleOk` 崩到 0~1——這跟今天每次引入新核心參數的模式一樣，是預期中的事：`blend` 徹底改變了整個碰撞行為，原本配合舊模型（`blend=0`）校準的力道公式/瞄準機制/揮拍方向全部要跟著重新聯合校準，不能只加新機制、不動舊的。

**結論：這個「壓縮反彈方向跟切向滑動方向耦合」的機制，很可能才是今天卡了一整天的真正缺失環節，不是球拍速度尺度（Stage 1）不夠。之前不管怎麼加大力道、換揮拍方向、換阻尼比，正確旋轉永遠是稀有事件，現在稍微一調就能到0.9+，代表問題根源終於找對了。**

**下一步（尚未執行）：** 把 `blend` 正式納入聯合校準的參數清單（揮拍方向、摩擦係數、epsilon/彈簧阻尼、力道公式、拍面角度、`blend`），重新做一次完整的聯合搜尋——這次用獨立 Node 優化腳本（不再用瀏覽器手動網格搜尋），因為參數維度又增加了一個，手動切格子更加不現實。這是目前為止最有希望真正解開核心矛盾的候選方向。

### 策略調整：先解「指尖摩擦案例」（小動作、大旋轉），再回頭做14顆發球的聯合校準

**使用者提出更務實的優先順序：與其直接在14顆發球的複雜情境裡湊合校準，應該先把「指尖摩擦」這個乾淨的參考案例解出來——透過很小的動作（很小的拋球、很小的切球動作），就能產生足以在三指間快速轉動的大旋轉。找到這個案例對的物理參數，比硬湊14顆發球的妥協值更可靠。**

**釐清「小動作」的真正意思（使用者澄清）：不是拍子完全不動，是有正常的切下去動作，但球會在某個瞬間被「卡住」（陷入海綿），旋轉在卡住的點產生，然後從那個狀態往上彈開。**

**驗證結果（用 blend=0.5、tiltY=2.0、正常低角度拋球 z=-2.5，掃描切下速度）：**

| 切下速度(downSpeed) | 上升高度 | 旋轉(rps) |
|---|---|---|
| 0.1（極小動作） | 0.299m | 9.93 |
| 0.3 | 0.217m | 10.15 |
| 0.5 | 0.152m | 10.43 |
| 0.8 | 0.088m | 10.04 |
| 1.2 | 0.042m | 8.85 |
| 2.0 | 0.005m | 6.83 |
| 3.0 | 0（彈不起來） | 4.66 |

**結論（重要，違反直覺但驗證過）：切下的動作越小，彈起來越高，旋轉量幾乎不變（甚至略高）——動作越用力，反而越把球往下壓，把彈起來的高度吃掉。原因：球拍的下切速度會直接疊加進最終出球速度（往下疊加），跟彈起來的方向互相抵銷；而旋轉的來源，主要不是靠拍子往下切的速度，是靠球本身水平拋過去的速度，在拍面上滑動摩擦產生的——切的動作只需要「夠、有方向」讓摩擦跟blend機制發生作用，不需要用力，用力反而扣分。這完整、乾淨地解釋了「小動作、大旋轉」的真實現象，不是「力氣藏在小動作裡」，是「小動作剛好不會抵銷彈起來的效果，旋轉的主要能量來源本來就是球自己的速度，不是揮拍力氣」。**

**下一步：** 拿這組已驗證的乾淨參數（`blend≈0.5`、`tiltY≈2.0`、切下速度宜小不宜大）當作 Node 優化腳本聯合校準 14 顆發球時的起點/錨點，而不是從隨機初始值開始盲目搜尋——這樣找到的解更可能貼近真實物理，而不是為了應付14種不同情境硬湊出來的妥協值。

### Node 優化腳本實測過程：三層互相牽制的張力，以及最終發現的根本方法論問題

**用「指尖案例」錨點（純垂直揮拍、tiltY=2.0）直接跑14顆聯合校準，結果崩潰**：`retVel.z` 幾乎趨近零，14顆幾乎全部失敗——**追查發現不是 `yRatio`（揮拍方向）的問題，是 `tiltY` 太平（2.0）本身就會把球大部分的前進動能轉成垂直方向，這是設計上的必然結果（指尖案例本來就不需要球飛遠），不能直接套用到「切球要過網」這個任務。**

**修正方向：使用者用時鐘模型算出真實揮拍方向向量（11點指向3點）≈ `yRatio≈-0.577`（往下+往前，不是往後），改用溫和拍面角度（`tiltY≈1.0`）重新錨定。**

**評分函式修正：加了「準確率門檻」（`accuracyGate = (singleOk/14)^1.5`），避免爬山法靠犧牲準確率換取虛高的旋轉分數討便宜（今天實測發現爬山法容易卡在這種偏頗解）。**

**追問「blend 到底反映在哪」，意外發現一個重要簡化：固定高 `blendRelease`（如0.9）後，`tiltY` 對「旋轉方向對不對」的影響大幅降低（旋轉穩定維持正確，不太受 tiltY 影響），但對「往前速度」的影響還在——這代表 blend 主導旋轉正確性，tiltY 主導其他行為。**

**用這個理解重新聚焦（固定高blend，只搜tiltY/yRatio/力道），結果 `singleOk` 整個崩到0——追查發現：固定高blend（0.85）本身跟tiltY=2.0一樣，也會把太多前進動能轉成垂直方向，導致球飛不到對面（`fb.z`是正值，落在自己這側）。放寬 `neutralMag` 範圍（讓力道可以加大去補償）後，找到 `singleOk=7/14` 但 `correctSpinFrac=0` 的組合——追查發現第三層張力：`neutralMag` 加大到3.66去補償前進動能，本身又會把旋轉推回錯誤方向（這是很早以前就發現的「力道越大旋轉越飄」現象）。**

**三層互相牽制的張力（今天最終釐清的問題全貌）：**
1. 需要足夠力道（`neutralMag`）讓球飛過網（準確率）。
2. 需要高 `blend` 才能讓旋轉方向正確。
3. 但力道加大本身，又會獨立地把旋轉推回錯誤方向，抵銷 `blend` 帶來的好處；同時高 `blend`／平拍面角度，也會獨立地吃掉前進動能，跟力道的目的互相打架。

**最根本的方法論問題（使用者提出，今天最重要的收穫）：`blend` 徹底改變了球拍反彈的路徑（不再是剛體反射），但 `tiltX` 公式（10顆使用者標記樣本回歸出來的）、`tiltY` clamp 範圍（0.4~0.55）、`friction`、`epsilon` 動態公式，全部都是在「沒有 blend（剛體假設）」的年代校準出來的——引入 blend 之後，這些舊公式描述的「怎麼調整才能瞄準/怎麼樣才穩定」，理論上都不再成立，但今天的搜尋一直把它們當成「已知好用的預設值」，只搜尋 blend/tiltY/yRatio/力道這些新東西，沒有一併重新推導這些舊公式。這很可能是今天不管怎麼搜都感覺卡住的根本原因——搜索空間裡混雜著「該重新推導」跟「還在用舊物理假設」的參數，兩者互相打架。**

**另外發現一個實作缺口：今天的 Node 優化腳本完全沒有實作 `tiltX`／瞄準機制（左右修正），只有 y、z 兩個分量，這是一個簡化，之後要嘛把 `tiltX` 也納入 Node 腳本重新推導，要嘛只能在真實引擎裡驗證左右瞄準這件事。**

**下次接續的正確順序（不是繼續在舊公式上打補丁）：**
1. 先接受 `blend`/兩階段動態blend 是新的物理地基，把 `tiltX` 公式、`tiltY` 範圍、`friction`、`epsilon` 全部當成「未知、需要在新物理下重新推導」的變數，不要沿用舊數值當預設。
2. 把 Node 優化腳本補上 `tiltX`／瞄準機制，讓左右落點準確率也真正被搜尋、被驗證，不是只驗證前後方向。
3. 用更大範圍的聯合搜尋（可能需要比爬山法更聰明的演算法，例如允許同時擾動多個參數、或模擬退火）去找一組全新的、跟 blend 物理自洽的完整解，而不是零散地修補舊公式。
4. 這輪已經把「三層互相牽制的張力」跟「舊公式失效」這兩個問題講清楚了——下次只要照這個更完整的框架重新開始，而不是繼續在舊地基上打補丁，應該能比今天更有效率地收斂到答案。

## Stage 2 重建計畫：以 `blend`（反彈方向-切向滑動耦合）為新地基，重新推導整套切球技術模型

**背景與目的：** 今天發現 `blend` 徹底改變了球拍反彈的物理（不再是剛體反射），但 `tiltX` 回歸公式、`tiltY` clamp 範圍、`friction`、`epsilon` 動態公式，全部是在「沒有 blend」的舊物理下校準出來的。繼續在這些舊公式上打補丁、只搜尋 `blend`/`tiltY`/`yRatio`/力道，會一直卡在互相打架的局部最優（今天的三次搜尋都證實了這件事）。這份計畫的目的，是把整套切球技術**當成一個全新的東西，從角色分工開始重新設計**，而不是修修補補。

### Phase A：先確立每個參數在新物理下該負責什麼（角色重新分工）☐

今天的實測已經初步揭露了新的角色分工，這一步要把它明確定案、寫成規格：

- **`blend`（`blendCompress`/`blendRelease`）：主要負責「旋轉方向對不對」。** 已驗證：`blendRelease` 夠高（0.7~0.9）時，旋轉穩定維持正確方向，且不太受 `tiltY` 影響。
- **`tiltY`（拍面角度）：主要負責「打不打得到桌子、弧線高低」，不再是旋轉正確性的主要決定因素。** 已驗證：`tiltY` 太平（如指尖案例的2.0）會讓前進動能被拍面角度吃掉，太陡則影響彈起高度；需要在這兩者之間找中庸值。
- **`tiltX`／瞄準機制：負責左右落點準確率，已用今天補齊的 `solveTechVelXForTargetLandingX` 實作，跟最終碰撞共用同一套 blend 感知物理（今天修正過的一致性缺口）。**
- **力道公式（`neutralMag`/`k`）：要同時滿足「前進動能足夠過網」，但不能大到「本身又把旋轉推回錯誤方向」（今天發現的第三層張力）——這是最難兼顧的一個，可能需要非線性、或跟 `blend`/`tiltY` 耦合的公式，不是單純的線性負回饋。**
- **這一步的產出：** 一份文字規格，清楚寫下這4個參數各自的「主要職責」跟「已知的副作用/耦合」，作為後面重新推導的依據，不要邊做邊想。

#### Phase A 產出（規格定案）✅

**1. `blend`（`blendCompress` / `blendRelease`）**
- **主要職責：** 決定「出球旋轉方向對不對」。
- **機制：** 有效反彈方向 = `(1-blend)×剛體法向量 + blend×(球實際接近方向的反方向)`；壓縮期（`vN>0`，球還在陷入）用 `blendCompress`，釋放期（`vN<0`，球開始被推出）用 `blendRelease`，呼應「先吃住、後摩擦刷到6點鐘」的真實描述。
- **已驗證行為：** `blendRelease` 越高（0.7~0.99），旋轉越穩定朝正確方向（`topspin<0`），且此時 `tiltY` 對旋轉方向的影響大幅降低（旋轉正確性主要由 blend 決定，不再是 tiltY）。
- **已知副作用：** blend 越高，會把越多球的原始前進動能（z方向）轉成垂直方向（y），可能導致前進動能不足、過不了網——這跟 tiltY 的副作用性質類似，需要跟力道公式一起補償。
- **待補測試：** `blendCompress` 的獨立角色今天沒有充分測試（今天主要固定在0.1，只掃 `blendRelease`），理論上壓縮期該接近0（維持剛體，符合「先吃住」還沒開始摩擦的物理圖像），但沒有實測驗證這個假設，留給 Phase B。

**2. `tiltY`（拍面角度）**
- **主要職責：** 決定「弧線高度」跟「打不打得到桌子」（過網動能的分配），**不再是旋轉方向的主要決定者**（有 blend 之後）。
- **已驗證行為：** `tiltY` 太平（如指尖案例驗證用的2.0）在特定情境下能讓球彈很高，但會嚴重吃掉前進動能（往前速度趨近0）；`tiltY` 較陡（接近舊部署的0.5）前進動能保留較多但弧線較低。
- **待決定：** 最佳範圍還沒找到，需要跟力道公式一起搜尋，找出兼顧「弧線高度」跟「過網動能」的平衡點，不是繼續沿用舊的「依殘留旋轉查表」邏輯（那個邏輯的前提——旋轉正確性由tiltY決定——已經不成立了）。

**3. `tiltX` / 瞄準機制**
- **主要職責：** 左右落點準確率（讓球落在目標x座標，`RETURN_TARGET_X=0`，桌面中央）。
- **實作方式：** 不是回歸公式，是即時反推（`solveTechVelXForTargetLandingX`，割線法迭代求解），已在 `push-optimizer.js` 跟真實引擎裡實作，用同一套 blend 感知碰撞模型（今天修正的一致性缺口：瞄準求解跟最終碰撞必須用同一套物理，不能各用各的）。
- **已驗證效果：** 加入這個機制後，`singleOk` 從卡住的0~10，跳到13/14——證實這是準確率的主要貢獻來源，遠比調 `tiltY`/`yRatio` 更有效。
- **結論：** 舊的 `tiltX` 回歸公式（10顆使用者標記樣本）**不再需要**——瞄準機制直接、即時地反推出正確的左右修正，取代了「靠一個固定回歸公式去猜」的角色。舊回歸公式可以正式淘汰，不用重新推導。

**4. 力道公式（`neutralMag`, `k`）**
- **主要職責：** 決定球拍揮拍速度大小，需要同時滿足 (a) 提供足夠前進動能讓球過網，(b) 不能大到獨立地把旋轉方向推回錯誤方向。
- **已驗證的張力（今天發現的第三層張力，四個參數裡最難兼顧的一個）：** `neutralMag` 越大，旋轉越往正方向（錯誤方向）飄，不管 blend 開多高都一樣會被抵銷；但 `neutralMag` 太小，前進動能不足，球飛不到對面（跟 blend/tiltY 的副作用疊加在一起，三者共同瓜分同一份「動能預算」）。
- **待決定：** 這是唯一還沒有清楚解法的角色。可能需要：(a) 力道公式本身依賴 `blend`/`tiltY` 的耦合設計（不是三個獨立公式各自調），或 (b) 認真考慮 Stage 1（球拍速度尺度校準，把整體能量預算撐大），讓「過網動能」「正確旋轉」不用在同一份小額度裡互搶，這可能是治本的方案，不是繼續在現有小尺度裡湊。

**核心洞見（貫穿四個角色的共同限制）：** `blend`（旋轉正確）、`tiltY`（弧線）、`neutralMag`（過網動能）三者，本質上都在爭奪同一份有限的「揮拍動能預算」——這也是為什麼「三層互相牽制的張力」會出現。如果 Stage 1（球拍速度尺度，目前模擬值比文獻小一個數量級）沒有先做，這個預算本來就先天不足，三個目標互搶的情況可能難以徹底解決，只能在目前的小額度裡找一個「還可以接受」的妥協點。**這代表 Phase A~F 的重建工程，可能最終還是會指向「必須先做 Stage 1」這個結論——如果 Phase B~D 怎麼搜都搜不出兩全的解，這就是最可能的原因，不要忘記把 Stage 1 當作最後的備案。**

### Phase B：對每個參數做獨立的敏感度測試，畫出乾淨的關係圖 ☐

在跳進14顆聯合搜尋之前，先針對Phase A定案的每個參數，用1~2顆有代表性的發球（例如 `backspin_long_backhand` 這種中等長度、中等旋轉的），單獨掃描每個參數對三個指標（`singleOk`能不能過網、`correctSpinFrac`旋轉方不方向、彈跳弧線形狀）的獨立影響，畫成類似今天做過的那種表格。

- **目的：** 建立一個「這個參數往哪個方向調，會讓哪個指標變好/變差」的敏感度地圖，讓後面的聯合搜尋有方向感，不是瞎子摸象。

#### Phase B 執行結果（重要，指向 Stage 1 才是真正的答案）✅

用 `backspin_long_backhand`（長球、中強旋轉）跟 `backspin_short_forehand`（短球）這兩顆代表性發球，逐一測試：

1. **`blendCompress` 獨立敏感度**（固定其他參數）：`blendCompress` 從0加到0.5，旋轉會單調地更負（更正確），但兩顆球在這組基準參數下本來就過不了網（`ok`全部false）——`blendCompress` 本身方向正確、影響幅度不大，不是主要瓶頸。
2. **尋找「至少打得到桌子」的基準點**：固定 `blendRelease=0.85`（先前驗證過的高值），掃 `tiltY`（0.3~1.3）× `neutralMag`（1.0~2.5）共16組，**沒有任何一組成功過網**。
3. **放寬 `blendRelease` 到0.5（較低值）**，才找到幾組能過網的组合——但**全部都是正旋轉（錯誤方向）**，topspin範圍 +61~+109，再次驗證「力道加大會把旋轉推回錯誤方向」這個張力。
4. **最終確認測試（排除「力道不夠大」這個可能性）**：把 `neutralMag` 一路加到公式clamp上限（8，超過8的9、12會被夾住、结果重複），交叉搭配 `blendRelease∈{0.7,0.85,0.95}`、`tiltY∈{0.7,1.0,1.3}`，逐一列出結果：
   - `blendRelease=0.7`：過網高度 -0.06~-0.09（掛網），旋轉錯誤方向（+52~+116）。
   - `blendRelease=0.85`：過網高度更差（-0.5~-0.6，嚴重掛網），旋轉仍是錯誤方向。
   - `blendRelease=0.95`：**不管力道加到多大，球連過網判定都到不了**（`netClearance=null`），但這時候旋轉終於接近/轉為正確方向。

**結論（Phase B 最重要的產出）：在目前這組公式（固定的blendRelease常數、固定揮拍方向、線性力道公式）之下，就算把力道加到公式允許的上限，也找不到任何一個組合能同時滿足「過網」跟「旋轉正確」——連對單獨一顆代表性發球都做不到，不用說14顆。這不是搜索範圍不夠廣或搜尋演算法不夠聰明的問題（Phase C/D 想解決的問題），是這個小尺度的「揮拍能量預算」本身可能就先天不足——這直接證實了 Phase A 結尾標記的備案：`blend`（旋轉正確）、`tiltY`（弧線）、`neutralMag`（過網動能）三者在目前的尺度下，本質上互斥，沒有交集。**

**建議：暫緩 Phase C/D（改良目標函式/搜尋演算法），因為就算演算法再聰明，搜尋的空間裡本來就沒有「兩全」的解存在，聰明的演算法也搜不出不存在的東西。優先做 Stage 1（球拍速度尺度校準，把整體能量預算撐大），把「過網動能」的門檻用更大的球拍速度尺度去滿足，才有可能空出餘裕讓 `blend`/`tiltY` 去專心處理旋轉正確性，而不是三者在小額度裡互搶。**

## Stage 1 執行：球拍速度尺度校準（重大進展）

**關鍵假設修正：不是只放大球拍自己的力道（`neutralMag`），是要連「來球速度」也一起等比例放大。** 原因：許多接觸力學關係是線性的（速度等比例縮放，比例關係不變），只放大球拍這一邊，不會改變「來球動能 vs 球拍動能」的相對比例，自然解不開三層張力（今天稍早已經測過「只放大`neutralMag`到clamp上限」完全沒用，這正是原因）。但摩擦力的滾動/滑動門檻是**非線性**的（有一個判斷式：需要的衝量 vs 可用的衝量），把來球速度**跟**球拍速度**一起**等比例放大，才可能真正改變是否跨過這個非線性門檻，不是原地打轉。

**驗證結果（用 `backspin_long_backhand` 這顆代表性發球，來球速度=原始值×scaleFactor）：**

| scale | blendRelease | 結果 |
|---|---|---|
| 1（原始尺度） | 0.7/0.85/0.95 | 全部失敗（掛網或旋轉錯） |
| 2 | 0.85 | topspin=-4.6（接近正確）但仍掛網(-0.078) |
| 2.6 | 0.85 | **topspin=-6.5，過網成功！** |
| 3.0 | 0.9 | **topspin=-34.2，過網成功！（更強的正確方向旋轉）** |

**這是今天第一次，同一組參數真正同時滿足「過網」跟「旋轉正確」——證實了「連同來球速度一起等比例放大」這個修正假設是對的方向。**

**泛化測試（用 `scale=3, blendRelease=0.9, tiltY=0.7, neutralMag=2.4` 固定力道，測全部14顆）：`10/14 旋轉方向正確`（比先前卡住的0.02~0.5大幅進步），但只有 `2/14` 完全過網成功——**失敗模式從「掛網」變成「飛太遠/衝過頭」**（`netClearance`是正的，過網沒問題，但落點超出桌子範圍）。這代表力道公式現在需要真正的自適應（依每顆球各自的來球速度調整力道大小），不能用同一個固定倍率套用到來球速度差異很大的14顆發球上——固定倍率對某些球剛好、對某些球太大，導致衝過頭。**

**已嘗試補上自適應力道公式（負回饋，`neutralMag - k×(來球速度-基準速度)`，基準速度/範圍都跟著scale=3等比例調整），快速搜尋 `neutralMagBase`/`kAdapt` 沒有明顯超越固定倍率的結果（最佳僅 `okCount=1, correctCount=10`）——代表這個自適應公式的具體係數還需要更仔細地重新校準，不是公式方向錯了，是這輪搜尋範圍/起點還不夠精準。**

**結論：Stage 1 的核心假設（連同來球速度一起等比例放大，不是只放大球拍力道）已經驗證成立，這是今天最重要的單一突破——首次讓「過網」跟「旋轉正確」在同一組參數下同時成立。剩下的工作是把自適應力道公式在新尺度下重新校準準確，讓「過網成功率」也跟上「旋轉正確率」的進步幅度。**

**下一步（尚未完成）：**
1. 更仔細地重新校準自適應力道公式在 `scale≈2.6~3` 這個新尺度下的 `neutralMag`/`k`/基準速度，目標是14顆的過網成功率也跟旋轉正確率一樣大幅提升。
2. 確認「來球速度也要等比例放大」這件事，實務上代表什麼——是要重新校準14顆發球本身的速度（`physics-presets.json`），還是只在切球回擊這一步做一個「內部尺度轉換」（回擊時先把來球速度看成放大過的、算完再縮小回去）？這兩種做法對遊戲其他部分（發球視覺效果、時間軸）的影響不同，需要跟使用者確認。
3. 確認後，正式把這組新公式（`blend`兩階段、`tiltY`新範圍、自適應力道新係數、`scale`因子）寫進 `return-studio.html` 研究版，跑一次完整的16顆驗證（含連續對打），對照 Phase G 的部署前確認清單。

#### 設計決策定案：擬真檢查 → 選「內部尺度轉換」，不動發球本身的可見速度 ✅

**擬真檢查（換算成真實 m/s）：** 目前14顆發球的來球速度（模擬尺度約1.4~3.4）換算成真實 m/s（乘以 `SIM_TIME_DILATION≈1.528`）大概是 **2.1~5.2 m/s**——這已經很接近真實比賽發球的合理速度。**如果把 `scale≈3` 直接套用在球本身的可見速度上，換算後會變成 15~16 m/s，遠超真實發球速度，不擬真。**

**定案：`scale` 因子不代表「球真的飛比較快」，是碰撞計算內部的修正，代表模型目前漏掉的其他物理效應——球拍在接觸瞬間其實還在加速（Stage 3）、海綿有彈性儲能（Stage 4）、還有全身動力鏈在那個瞬間注入的額外動能。這些效應加總起來，讓「接觸瞬間的有效相對速度」比「外部看到的來球平均速度」高很多，但球本身外觀飛的速度不變。**

**做法：** 球本身的可見速度（`physics-presets.json`、發球視覺效果、時間軸）完全不動；只在切球回擊的**接觸計算內部**，把餵進碰撞函式的來球速度／球拍速度都先乘上這個 `scale` 因子再計算，計算出來的**結果（出球速度/旋轉）不用縮小回去**——因為這正是「真實技術裡小動作能打出大威力」的物理意義：球看起來還是同樣速度飛過來，但因為動力鏈+海綿彈性+持續加速，回擊出去的球確實比單純看揮拍外觀速度算出來的更快、更有旋轉，這是真實現象，不是需要修正掉的誤差。

**好處：** 不用重新調16顆發球的視覺速度（牽動很多其他已校準好的東西）；且這個 `scale` 之後如果真的做 Stage 3（dwell加速）/Stage 4（海綿彈性）的完整建模，理論上可以拆解成更精細的分開建模，現在先用一個簡化的整體倍率代表。

#### 自適應力道公式在新尺度下的校準過程（進行中，尚未達到理想解）

用 `neutralMag = clamp(neutralMagBase - kAdapt×(scale後的來球速度 - neutralSpeedBase), magFloor, 8.0)` 這個形狀，對 `scale`/`tiltY`/`blendRelease`/`neutralMagBase`/`kAdapt`/`neutralSpeedBase`/`magFloor` 做多輪網格搜尋（14顆全部一起評分，`okCount`=過網成功數，`correctCount`=旋轉正確數，兩者滿分都是14）：

| 輪次 | 調整方向 | 最佳結果 |
|---|---|---|
| 第1輪 | 初始寬範圍搜尋 | okCount=1, correctCount=10（比固定倍率的2+10還差） |
| 第2輪 | 縮小範圍精修 | okCount=5, correctCount=11 |
| 第3輪 | 拉高力道下限、放緩kAdapt | okCount=4, correctCount=11（略退步） |
| 第4輪 | 更大範圍力道下限+tiltY細調 | okCount=4, correctCount=12 |

**診斷發現的具體問題：** `kAdapt` 對這個新尺度來說容易太陡——很多「長球」（來球速度較快）的力道會被壓到 clamp 下限，導致力道砍過頭、飛不到對面；同時側旋反手系列的發球（`sidebackspin_half_long_backhand`、`sidebackspin_long_backhand`）旋轉方向持續是錯的（+50~+61），可能跟 blend 使用的3D相對速度方向在反手幾何下的行為不同有關，這點還沒深入診斷。

**目前最佳穩定在 `okCount≈4~5, correctCount≈11~12`（滿分各14），比 Stage 1 一開始的 2+10 進步，但還沒達到兩者都接近滿分的理想解。這是持續的漸進式校準工作，不是卡住——建議在 Phase F（完整聯合驗證）時，用今天已經補齊的 Node 優化腳本（含瞄準機制），搭配這裡整理出的參數範圍當起點，跑更完整的自動化搜尋（而非手動網格），應該能比手動繼續調更有效率地收斂。**

**尚未診斷：** 反手側旋系列旋轉方向持續錯誤的原因（可能跟 blend 機制在不同x方向來球的行為有關，需要獨立測試）。

#### 反手側旋旋轉方向錯誤的根因診斷（已找到，且已驗證可修正）✅

**根因：** 固定 y/z 分量、只改變來球 x 方向速度，發現出球旋轉隨 xVel 呈乾淨的單調關係——`xVel` 從 -2（強反手）到 +1（正手），`outTopspin` 從 +54.6 單調降到 -38.5，大約在 `xVel≈-0.1~0` 附近跨過零。**反手方向的來球（x為較大負值）天生會把 blend 機制算出的旋轉推向錯誤方向（正值），這不是bug，是 blend 用「完整3D相對速度方向」去混合法向量時，天生跟x方向的來球速度耦合在一起——正手來球x速度通常較小/正值，恰好落在blend讓旋轉維持正確的範圍；反手來球x速度較大/負值，超出了這個範圍。**

**驗證修正方法有效：加入 `tiltX`（拍面繞揮拍軸旋轉），依 xVel 反向補償，能把旋轉方向修正回來：**

| xVel | tiltXGain=0（現況） | 修正後 |
|---|---|---|
| -2（強反手） | +54.6（錯） | tiltXGain=0.5 → **-10.3（對了）** |
| -1（反手） | +25.5（錯） | tiltXGain=0.8 → **-1.0（接近正確）** |

**但注意：不同 xVel 量級需要不同的 `tiltXGain` 才能剛好修正到零附近（xVel=-2需要gain≈0.5，xVel=-1需要gain≈0.8）——不是簡單的線性正比關係，需要更仔細的公式設計（可能類似原本非線性形狀，或是分段/查表）。**

**這修正了 Phase A 的一個結論：`tiltX` 不是「舊角色被瞄準機制取代、完全不需要了」，是有兩個獨立功能：**
1. **舊角色（已被瞄準機制取代）：** 校正左右落點位置。
2. **新角色（今天發現，瞄準機制沒有處理到）：** 抵銷 blend 機制隨x方向來球速度產生的旋轉方向偏移，需要重新設計一個新的 `tiltX` 公式（不是舊的10顆樣本回歸公式，是全新的、針對這個新耦合關係設計的公式）。

**這很可能是把 Stage 1 卡在 okCount≈4~5/correctCount≈11~12 的部分原因之一**——反手系列的球一直旋轉方向錯誤，拖累了整體的 `correctCount`。**下一步：把這個 `tiltX(xVel)` 補償公式加進 Node 優化腳本，重新跑一次14顆聯合校準，預期能同時把 `correctCount` 拉高（修正反手系列）。**

#### 加入 tiltX 補償後的完整聯合搜尋結果（今天最終、最佳的候選）✅

把新發現的 `tiltX` 補償（`clamp(tiltXGain×(-incomingVel.x), -tiltXMax, tiltXMax)`）正式加進 Node 優化腳本，跟 `scale`／`tiltY`／`blendRelease`／自適應力道公式一起做網格搜尋（過程中修正了一個真正的程式bug：第一輪搜尋忘記把 `yRatio` 放進參數列表，導致方向計算跑飛，結果全部無效，修正後重跑）。經過三輪收斂：

| 輪次 | 結果 |
|---|---|
| 第1輪（修正yRatio bug後） | okCount=6, correctCount=13, total=19 |
| 第2輪精修 | okCount=8, correctCount=13, total=21 |
| 第3輪精修 | okCount=8, correctCount=13, total=21（持平，已收斂） |

**最終候選參數：** `yRatio=-0.55, scale=3.3, tiltY=0.6, blendRelease=0.84, neutralMagBase=0.6, kAdapt=0.06, neutralSpeedBase=2, magFloor=0.5, tiltXGain=0.5, tiltXMax=1.0`

**逐顆結果（14顆中，只剩1顆旋轉方向錯誤、6顆過網失敗）：**
- **13/14 旋轉方向正確**，只有 `backspin_long_forehand` 是 +8.9（接近零、方向仍稍微錯）。
- **8/14 完全過網成功**，其餘6顆多半是「旋轉方向對但衝過頭/沒接住落點」（例如 `sidebackspin_short_forehand` topspin=-292.3方向很對，但ok=false）。

**這是今天從頭到尾最好的結果**——旋轉正確率已經非常接近滿分（13/14），過網成功率也有明顯進步（8/14，比 Stage 1 剛開始的2/14好4倍）。剩下的落差主要集中在「力道公式的自適應係數還需要更精細校準」，不是架構性的問題——今天已經把 `blend`（旋轉）、`tiltY`（弧線）、`tiltX`（反手側旋修正）、`scale`（能量預算）這幾個角色都定位清楚且驗證有效，剩下的是持續的數值精修，屬於 Phase F 的收尾工作。

#### 診斷過網失敗的6顆：不是力道大小的問題（重要修正）

**原本以為過網失敗是「力道公式的下限（`magFloor`）卡住、力道砍不夠低」，但實測發現不是——把 `magFloor` 從0.5降到0.15（力道砍到幾乎最低），同樣的6顆球依然衝過頭（過網高度仍然是0.29~0.50，偏高，`fb=null`）。這代表這6顆球的過網失敗，不是靠調球拍力道能解決的——它們的出球速度過強，主要來源可能是球本身的動能（透過`scale=3.3`放大後）搭配這幾顆球的blend/tiltX組合，產生了某種偏強的耦合效應，不是球拍那一份力道貢獻的。**

**這6顆是：** `sidebackspin_half_long_backhand`、`sidebackspin_long_backhand`、`sidebackspin_long_forehand_2`、`backspin_long_forehand`、`backspin_long_forehand_2`、`sidebackspin_short_forehand`——目前看不出跟正手/反手、長球/短球有明顯的規律關聯（例如同樣是`backspin_long_backhand`卻成功了），需要更細緻地逐顆比較 blend/tiltX 在這幾顆球各自幾何下的行為，才能找到根因。這是 Phase F 收尾時，比自適應力道公式更優先要診斷的下一個問題。

**再往下一層診斷，找到更精確的原因：這6顆失敗球（扣除`sidebackspin_short_forehand`是另一種「掛網」失敗），出球的「垂直速度分量(y)」明顯偏高（1.89~2.53），比成功過網的球（0.55~1.35）高出一截。垂直速度越高，球在空中停留時間越長，飛行距離自然拉更遠——這才是真正「衝過頭」的根本原因，不是水平速度(z)或力道大小的問題（這也解釋了為什麼調低`magFloor`完全沒用：力道公式主要影響的是整體出球速度大小，不是專門影響垂直/水平的比例分配）。**

**下一步（更精確的方向）：需要找出這幾顆球「為什麼垂直分量特別偏高」——可能跟這些球各自的來球角度、或 blend/tiltX 在特定入射角度下對「有效法向量」的影響有關（`tiltX`會旋轉整個法向量，可能連帶把一部分本該是水平的分量轉成垂直分量，在某些入射角下特別明顯）。這是比「重新校準力道公式」更精確的下一個診斷方向。

## 使用者視覺檢查發現的重大架構問題：`scale` 因子的輸出不能完全保留，必須部分縮放回去

**觸發：使用者要求把這組最佳候選參數實際顯示在 `return-studio.html` 的3D畫面裡（`backspin_long_forehand`這顆，垂直分量異常偏高的那顆），用眼睛直接看軌跡形狀。** 螢幕截圖顯示：**回擊軌跡飛得又高又遠，直接飛出桌子範圍，落到背景牆附近——用眼睛一看就知道不合理，比對數字診斷（垂直分量1.89~2.53異常偏高）完全吻合，但視覺呈現更直接、更有說服力。**

**使用者提出兩個關鍵質疑，都被驗證是對的：**
1. **「這顆球沒有側旋轉，往哪裡打就會往哪裡飛，x方向一定瞄的錯的很離譜」**——查證發現：這輪注入瀏覽器測試時，只覆寫了「最終碰撞」（`applyPushContact`），忘記同步覆寫「瞄準求解」（`solveRacketVelXForTargetLandingX`）——瞄準機制還在用舊物理（沒有blend、沒有scale）去算修正方向，跟實際碰撞的新物理不一致，這正是先前發現過、但這輪手滑又重複犯的同一個bug。已修正（瞄準求解也換成同一套 `bounceTwoPhaseBlend`+scale）。
2. **「拍面換算是幾度？看起來沒有成功啟動blend，不然回彈速度怎麼可能這麼快」**——查證：`tiltY=0.55`（法向量角度28.8°，換算拍面本身角度61.2°），blend **確實有生效**（有無blend時y分量有明顯差異：1.91 vs 4.2），**但問題出在別的地方**——不管有沒有開blend，出球總速度都在8+的誇張量級，因為我們稍早的設計決策是「`scale=3.3`放大過的計算結果完全不縮小回去，直接當作最終出球速度」。這個決策造成了「子彈感」——回擊速度整整快了3倍多，完全不像切球，比較像扣殺。

**修正：稍早的設計決策（結果完全保留，不縮小）是錯的。正確做法是：內部用放大過的速度算物理（解決摩擦力非線性門檻問題），但輸出速度要縮小回合理尺度；旋轉維持不縮小（這是「小動作、大旋轉」該有的效果，指尖案例已經驗證過）。**

**技術細節（避免下次重踩）：不能簡單地把輸出速度整個除以 `scale`（3.3~3.5）——這樣會把 y 分量也壓得太扁（球飛不夠高、過不了網），因為 y 分量在放大前後的比例關係跟 z 分量不同。正確做法是引入一個獨立的 `outputRescale` 參數（不等於內部的 `scale`），只用它縮小最終輸出速度，透過重新聯合搜尋校準出這個新參數的合理值。**

**重新校準結果（用修正後的架構）：** 經過三輪搜尋收斂，最佳候選：`yRatio=-0.45, scale=3.5, tiltY=0.7, blendRelease=0.86, neutralMagBase=1.2, kAdapt=0.08, magFloor=0.4, tiltXGain=0.6, tiltXMax=1.0, outputRescale=1.2`，達到 **`okCount=8, correctCount=13`（總分21）**——追平先前（有bug時）的最佳分數，但這次是在視覺合理、瞄準機制一致的正確架構下做到的，不是靠一個視覺上明顯錯誤的「子彈式」輸出湊出來的分數。**在瀏覽器裡用這組新參數重新顯示 `backspin_long_forehand` 的軌跡，確認回擊路徑已經是合理的短弧線、落在桌面範圍內，不再飛出背景。**

**這次的教訓再次證實：數字驗證不能取代視覺檢查——`okCount`/`correctCount` 這兩個指標本身沒有能力偵測「輸出速度離譜地快」這種問題，因為只要球最終落在桌子範圍內、旋轉方向對，分數就會顯示為成功，但視覺上一眼就能看出不合理。之後每次找到「數字上很好」的候選，都應該補一次視覺檢查再定案。**

## 架構性突破：移除 `tiltX` hack，`blend` 改成只看 y-z 平面——不需要拍面誇張旋轉也能修正反手側旋

**觸發：使用者看了「側下長球到反手」的3D視覺化，直接指出問題——「這個動畫看起來就像是球拍完全沒動，巧妙用拍面彈性吸收了所有衝擊（有點像足球停球的技巧），基本上不合理，動能怎麼可能就被吸收掉，一定有哪個環節算錯了」。**

**診斷：先查energy/動量有沒有真的憑空消失——算出等效反彈比例是0.543，數字本身合理，不是憑空吸收。真正的問題找到了：這顆球算出來的 `tiltX=1.00`（弧度，換算成角度約57°！）——代表我稍早加的「tiltX補償」，為了修正反手側旋的旋轉方向錯誤，把整個拍面轉了快60度，幾乎轉成側面朝向，這是不合理的拍面角度，跟正常切球差很多，配合blend機制產生了這種「像被吸收」的不自然觸感。**

**使用者釐清 `tiltX` 補償的正確使用情境：`tiltX` 大幅度旋轉，只有在「對方發球很快，來不及選位置，把拍面側得很誇張去『碰』到球」這種真實存在的緊急情境下才合理；正常情況下，補償側旋轉的機制應該是「透過瞄準方向（憑經驗過量瞄準）去抵銷側旋轉的偏移」，不是靠轉拍面角度。這代表我用 `tiltX` 去修正反手側旋的topspin方向錯誤，從一開始就用錯了機制。**

**重新檢視根本原因：反手側旋的topspin方向錯誤，會不會根本不是「側旋轉需要瞄準補償」的問題，而是 `blend` 機制本身的一個耦合缺陷——用「完整3D相對速度方向」去混合有效法向量時，不小心讓x方向的來球速度摻進了「決定上旋/下旋」的y-z平面計算裡？**

**驗證：把 blend 算「有效法向量」時用的 approachDir，限制只取 y-z 分量（完全忽略x方向），`tiltX=0`（拍面完全不轉，正常角度），測試 `sidebackspin_long_backhand` 這顆：旋轉從 `blend=0`的+133.3，乾淨地轉到 `blend=0.95`的-103.3——完全不需要任何拍面旋轉，只靠這個修正就解決了旋轉方向問題！**

**這證實了使用者的洞察：x方向的來球速度不該摻進決定上旋/下旋的垂直平面計算裡，這是 blend 機制設計上的耦合缺陷，不是側旋轉補償的問題。`tiltX` 的「拍面轉一個大角度去湊旋轉方向」這整個做法都可以移除，左右瞄準完全交給既有的瞄準機制處理（`solveRacketVelXForTargetLandingX`，本來就在正常運作）。**

**用這個修正後的乾淨架構（完全不用tiltX），重新做完整聯合搜尋，結果比原本用tiltX hack湊出來的還要好：**

| 候選 | okCount | correctCount | 總分 |
|---|---|---|---|
| 有tiltX hack（今天稍早最佳） | 8 | 13 | 21 |
| 無tiltX，y-z限制blend（第一輪） | 8 | **14（全對）** | 22 |
| 無tiltX，y-z限制blend（收斂後） | 10~11 | 12~13 | **23** |

**最終候選（其中一組）：`yRatio=-0.45, scale=2.0, tiltY=1.4, blendRelease=0.7, neutralMagBase=1.2, kAdapt=0.13, magFloor=0.5, outputRescale=1.4`：`okCount=11, correctCount=12`（總分23）。**

**這是今天最重要的架構修正之一——不只是分數變好，是找到並移除了一個真正不合理、不符合真實技術的機制（`tiltX`拍面誇張旋轉），換成一個更貼近真實物理的設計（`blend`只作用在決定上旋/下旋的y-z平面，左右瞄準完全交給獨立的瞄準機制）。這再次證實了視覺檢查的價值——如果沒有使用者堅持看3D動畫、並且準確描述「像足球停球」這個不合理的感覺，這個架構缺陷可能會一直被數字掩蓋下去。**

### 第三次抓到「瞄準求解跟實際輸出用不同物理」的同類bug（這次在 `outputRescale` 這一步）

**觸發：使用者看了「下旋長球到反手」（純下旋、沒有側旋）的3D軌跡，指出「側面視角的回擊軌跡幾乎是一條直線飛出去，不像正常拋物線；這球沒有側旋，瞄哪裡就該打哪裡，X軸瞄準的參數一定解錯了」，並在查證瀏覽器結果正確（`x=-0.01`，置中）後，進一步指出「Node腳本一定有問題」。**

**找到根因：Node 腳本的瞄準求解（`landingXFor`）內部，是用「還沒套用 `outputRescale` 縮小」的速度去估計飛行時間、算目標落點；但實際模擬球的飛行軌跡時，用的是「`outputRescale` 縮小過」的速度。這兩個速度不同，飛行時間也跟著不同，瞄準求解算出來的答案，套用到「實際會發生」的縮小後軌跡上，落點自然偏移——手動追蹤這顆球算出來的偏移量約-0.4，跟診斷時看到的異常值完全吻合。**

**這是今天第三次抓到同一類bug（瞄準跟實際輸出用不同物理），只是這次出現在新加的 `outputRescale` 步驟裡，前兩次分別出現在 blend 加入時、tiltX 加入時——每次新增一個影響最終速度的機制，都要記得同步檢查瞄準求解內部有沒有跟著更新，這已經是一個明確的模式，值得列進 Phase G 部署前確認清單當作固定檢查項。**

**修正：讓 `landingXFor` 內部也對 `bounced.vel` 套用 `1/outputRescale`，再拿縮小後的速度去估計飛行時間跟目標落點；初始猜測值 `x1` 也要對應調整。修正後重新驗證目前最佳候選（`yRatio=-0.45, scale=2.0, tiltY=1.4, blendRelease=0.7, neutralMagBase=1.2, kAdapt=0.13, magFloor=0.5, outputRescale=1.4`）：`okCount=10, correctCount=12`（總分22，跟修正前的23很接近，代表這個bug對整體分數影響不大——多數情況下錯誤的x偏移量剛好還在桌寬範圍內，只是瞄準本身不準——但現在14顆的 `fbx` 全部正確置中在-0.01~0.17（除了一顆側旋出界的-0.61），是真正修好、可信任的結果，不是誤打誤撞湊出來的分數。**

**這也代表 Phase F 收尾的驗證清單要新增一項：`return-studio.html` 正式實作時，任何時候新增/修改一個會影響最終出球速度的機制，都要同步確認瞄準求解（`solveRacketVelXForTargetLandingX`）內部的計算路徑，是否跟最終真正套用的碰撞結果完全一致（同一套物理、同一套縮放），不能分開處理。**

#### 修正瞄準bug後的重新校準結果：`correctCount` 首次達到14/14全對

用修正後（瞄準求解內部也套用`outputRescale`）的評分函式重新聯合搜尋，收斂到：**`yRatio=-0.45, scale=2.0, tiltY=1.4, blendRelease=0.7, neutralMagBase=0.8~1.0, kAdapt=0.1~0.12, magFloor=0.3~0.4, outputRescale=1.4`：`okCount=9, correctCount=14`（總分23，旋轉方向100%正確）。**

**逐顆診斷剩下5顆過網失敗，發現新規律：失敗的4顆（`backspin_short_backhand_2`、`backspin_short_forehand`、`backspin_short_backhand`，加上`sidebackspin_half_long_backhand`）裡，有3顆規律地都是「短球」，而且力道公式算出來的 `mag` 都明顯比其他球高（0.81~0.95），其餘全部過網成功的球（不管長短）力道都卡在下限附近（0.40）。這代表短球（來球速度慢）會被力道公式的負回饋機制推到比下限更高的力道值，這可能是掛網的部分原因——下一步該檢查力道公式對「慢速來球」這個區間的行為是否合理，而不是繼續調 `magFloor`。**

**這是今天第四次驗證的完整資料點，也是目前為止最乾淨、最可信任的階段性成果：架構上移除了`tiltX`hack、瞄準求解跟輸出物理完全一致、旋轉方向100%正確、過網成功率9/14。剩下的落差有明確、可追蹤的線索（短球力道公式行為），適合作為下次接續的具體起點。**

### Phase C：重新設計目標函式，避免爬山法掉進偏頗的局部最優 ☐

今天的教訓：`objective = singleOk + correctSpinFrac×14×accuracyGate` 這種單一純量目標，容易讓爬山法找到「一個指標很好、另一個很差」的解。改進方向：

- **選項1：硬性雙門檻，而非乘法軟門檻**——例如只有 `singleOk>=12` **且** `correctSpinFrac>=0.7` 的解才給分，兩個條件都沒達到一律視為0分，逼搜尋演算法必須同時滿足兩邊，不能用一邊去換另一邊。
- **選項2：真正的多目標/Pareto搜尋**——不要壓縮成單一分數，同時追蹤多個「沒有其他解在兩個指標上都更好」的候選解集合（Pareto前緣），最後人工從前緣裡挑一個平衡點，而不是讓單一目標函式的權重去決定妥協點在哪。
- **這一步要先決定用哪一種，再往下走，不要延續今天證實有問題的簡單加權法。**

#### 剩餘失敗案例的規律分析（尚未解決，記錄供下次接續）

逐顆診斷目前最佳候選（`okCount=8, correctCount=13`），失敗的6顆裡有4顆呈現一個規律：**`sidebackspin_half_long_backhand`（x=-1.90）、`sidebackspin_long_backhand`（x=-1.60）這兩顆x方向來球速度很大（負值），跟 `backspin_long_forehand`（x=0.18）、`sidebackspin_long_forehand_2`（x=0.21）這兩顆x速度很小（接近零）的，都會失敗；反觀成功的球，x速度都落在中等範圍（0.57~1.4）。**

**嘗試修正：把 `tiltXMax`（tiltX補償的夾限）從1.0放寬到1.5~3.5，理論上應該能讓大x速度的球不再被夾住——但實測分數反而略降（`okCount=7`，比原本的8還差），代表問題比「夾限太緊」更複雜，可能是 `tiltXGain`（補償的比例係數）本身在極端x值下不是線性關係（今天稍早單獨測試`tiltX`修正時就已經發現，不同x速度需要的補償比例不是固定線性關係），需要更仔細的非線性/查表式設計，不是simple放寬夾限就能解決。這是下次接續時，比繼續調 `magFloor`/`neutralMagBase` 更值得優先深入的方向。

### Phase D：換更聰明的搜尋演算法，或至少改良爬山法本身 ☐

今天的簡單「一次只調一個參數」爬山法，容易卡在 `singleOk`（階梯型分數）不敏感的地方。改進方向（任選其一或組合）：

- **模擬退火**：允許偶爾接受變差的解，有機會跳出局部最優。
- **同時擾動多個參數**：不要一次只改一個，允許小機率同時擾動2~3個參數，跳過「單一參數微調感覺不到進步」的死角。
- **兩階段法**：先各自找到「`singleOk`最高」跟「`correctSpinFrac`最高」的候選點（今天其實已經找到：`singleOk=13`的那組、跟`correctSpinFrac=0.9+`的那組），再以這兩點為錨點，在它們之間或附近做局部搜索/插值，而不是從隨機點開始瞎找。

### Phase E：重新推導 `tiltX` 回歸跟 `tiltY` 範圍（不能沿用舊物理校準的數字）☐

- **`tiltX` 公式**：舊的10顆使用者標記樣本，是在沒有blend的舊物理下標的，理論上已經不適用。有兩個選項：(a) 請使用者在新物理（`return-studio.html` 接上 blend 之後）重新用同樣的方法標記幾顆球，重新回歸；(b) 先用理論推導一個合理的起點（例如維持「揮拍方向要對準想打的落點」這個原則不變，只是要在新物理下重新解一次瞄準方程式，這部分已經在 `solveTechVelXForTargetLandingX` 裡做了，可能不需要獨立的`tiltX`回歸公式了，直接讓瞄準機制取代它）。
- **`tiltY` 範圍**：舊的 0.4~0.55 是配合「殘留旋轉強度決定容錯窗口」這個邏輯校準的，現在旋轉正確性主要由blend決定，`tiltY`該對應到什麼（可能是「弧線高度」或「打不打得到桌子」），需要重新定義它該依什麼變數調整，不是繼續依殘留旋轉去查表。

### Phase F：全部就緒後，重新跑14顆單拍+連續對打+旋轉方向正確性的完整聯合驗證 ☐

用 Phase A~E 的成果（新的角色分工、新的目標函式、更好的搜尋演算法、重新推導的tiltX/tiltY），跑一次完整的聯合驗證，目標維持不變：`singleOk` 接近14/14、`correctSpinFrac` 接近1.0。
