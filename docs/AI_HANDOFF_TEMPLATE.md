# AI 交接模板

> **用途：** 將小任務交給不同 AI model 執行時，用本模板明確界定可改／不可改範圍、驗收標準與待審問題。
>
> **基本原則：**
> - 低成本模型負責整理、表格、摘要、初稿。
> - 高階模型負責審查、架構判斷、物理推論。
> - 人類使用者負責最終決策。
> - 整理模型不確定的地方標示「不確定」，不自行補不存在的資料，不把假設寫成事實，不宣稱某模型已被證明，保留原始參數名稱，區分已驗證／待驗證／主觀判斷／推測，不任意刪除失敗案例。
> - Coding agent 未經明確批准不大範圍修改程式，先整理文件再提建議，每次修改前說明目標、檔案、風險與驗收方式，所有實驗結果寫入 `EXPERIMENT_LOG`。

## 10. 大包任務範例

> 下列範例示範如何一次把一組互相關聯的文件交給同一個 model 處理。這種任務適合 GLM cloud：先產出大包文字，再由 Codex 收斂與落版。

**任務名稱：** `整理 後段研究文件一致性大包`

**背景：** `docs/physics-engine-v2-plan.md` 後段同時混有 `blend`、`tiltX` / `tiltY`、`scale` / `outputRescale` 與部署前確認清單，需要一次把相關正式文件的交叉引用串起來，避免碎片化處理。

**允許修改的檔案：**

| 檔案路徑 | 預期動作 | 具體範圍／段落 |
|---|---|---|
| `docs/MODEL_DECISIONS.md` | 整理 | 對齊 TODO-006 / TODO-007 / TODO-008 與 RES-004 |
| `docs/EXPERIMENT_LOG.md` | 整理 | 補 EXP-019 / EXP-035 / EXP-037 / EXP-038 的交叉引用 |
| `docs/PROJECT_OVERVIEW.md` | 補充 | 強調文件系統與 AI 分工，不做調參 |
| `docs/CORE_FILE_SYNC_STATUS.md` | 補充 | 同步 `tiltX` hack、`blend`、`scale/outputRescale` 狀態 |
| `docs/DEVELOPMENT_MATRIX.md` | 補充 | 說明 GLM / Codex / 人類分工 |

**輸出要求：**
- 一次輸出多個文件的 patch-ready Markdown。
- 不要碎片化成單文件小修。
- 不要改核心程式。
- 不要把研究結果升格成最終結論。

**驗收標準：**
- [ ] 交叉引用可互相找到。
- [ ] `TODO-006` / `TODO-007` / `TODO-008` 都有明確落點。
- [ ] `VAL-009` / `VAL-010` 與決策台帳一致。
- [ ] 語氣保守，保留研究 / 待驗證狀態。

---

> **使用提醒：**
> - 本模板設計給小範圍、單一檔案、可在一輪對話內完成的任務。
> - 若任務需要跨多個檔案或涉及物理參數調整，請拆成多個小任務分別指派。
> - 執行 AI 不做最終物理判斷，不做架構決策，只做整理與標示。
