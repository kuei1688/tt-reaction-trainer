# 計畫起草前置檢查清單

> 本文件是「起草前自我檢查」清單，給 ChatGPT / Codex 在提出會觸及核心檔案或建立跨檔案共用機制的計畫前使用，用來在交給 Claude 邊界審查前先過濾掉常見的邊界問題。它不是物理模型規格，也不取代 Claude 的審查。
>
> 建立日期：2026-07-09
> 觸發時機：任何提案涉及 `game4.html`、`return-studio.html`、`physics-studio.html`、`physics-v2-calibration.html` 之間的比對、抽取、共用或架構調整時，先過一次本清單。

## 為什麼需要這份清單

構想一（抽取共用物理核心模組）與構想二（自動化一致性守門員）在送交 Claude 邊界審查時，反覆卡在同一種模式：具體技術主張（常數名、函式名、「這兩邊是重複的」）沒有先對照 `CORE_FILE_SYNC_STATUS.md` 等既有核對文件，就被寫進計畫。這份清單把常見漏洞前置到起草階段，減少審查來回輪數。

## 檢查項目

### 1. 主張溯源

- [ ] 計畫中提到的每個常數名／函式名，是否都能指出來源（`CORE_FILE_SYNC_STATUS.md` 段落、或本輪實際 grep 結果）？
- [ ] 是否有任何「這應該是重複的」之類的推測，未經查證就當成前提？

### 2. 既有規格檢索

- [ ] 是否已檢查 `docs/` 和 `AI_CONTEXT/` 是否已有涵蓋同一構想的文件（例如 `READ_ONLY_PHYSICS_EXTRACTOR_SPEC.md`）？
- [ ] 若有重疊，計畫是否明講「沿用／簡化／取代」，而非平行發明新格式？

### 3. 核心檔案 vs 純工具檔案

- [ ] 計畫每個步驟是否標註「是否修改 `game4.html`／`return-studio.html`／`physics-studio.html`／`physics-v2-calibration.html`」？
- [ ] 會修改核心檔案的步驟，是否被獨立列出、明確要求人類另外批准（而不是跟純新增 `tools/` 檔案的步驟混在一起）？

### 4. 分類／白名單 schema 的邊界推演

- [ ] 若計畫包含分類規則（例如白名單、status 標記），是否至少用一個「應該被擋下」的邊界案例，逐欄推演過一次邏輯，證明規則真的會觸發？
- [ ] 是否已排除「研究工具頁專屬機制」被誤標成「可共用／可比對一致」的可能？

### 5. 測試／驗證流程的隔離路徑

- [ ] 若計畫包含「先改壞、測試、再改回來」的步驟，是否明講具體隔離路徑（暫存目錄、副本檔名），全程不觸碰真正的核心檔案？
- [ ] 是否避免使用「複製一份」與「還原原檔」同時出現卻邏輯矛盾的寫法？

### 6. 物理決策 vs 程式重構

- [ ] 若涉及「合併」「統一」某個公式／機制，是否已確認這不是在做物理模型決策（而只是單純去重複）？
- [ ] 若兩邊程式碼目前公式不同（非單純重複），計畫是否承認這是需要另開研究任務的決策，而非直接歸入重構範圍？

## 使用方式

1. 起草計畫的 AI（ChatGPT／Codex）在交出計畫前，自己先過一次本清單，逐項打勾或說明「不適用」。
2. 若清單中任何一項打不了勾，計畫先不要交給 Claude 審查，先回頭補足。
3. Claude 審查時仍會用既有的邊界審查五問（見 `AI_CONTEXT/00_READ_ME_FIRST.md`），本清單是前置降低來回輪數，不取代審查。

## 反例（本輪實際案例）

- 構想一原始草稿把 `PADDLE_BLEND` 列為「可抽取進共用模組」的常數，但 `CORE_FILE_SYNC_STATUS.md` 已記載它是 `return-studio.html` 專屬研究機制，`game4.html` 未核對到同等機制。若先過第 1、6 項，這個錯誤在起草階段就能被排除，不需要等 Claude 審查才發現。
- 構想二原始草稿的白名單對 `PADDLE_BLEND` 設 `expected_match: false`，但 `files` 欄位只列 `return-studio.html`，沒有欄位表達「不該出現在 game4.html」。若先過第 4 項（用這個案例逐欄推演一次），就能在起草階段發現 schema 缺欄位。
- 構想二原始草稿的破壞性測試步驟同時寫「複製一份 `game4.html`」與「測試完成後將 `game4.html` 還原」，兩者邏輯矛盾。若先過第 5 項，這個矛盾在起草階段就能被排除。
